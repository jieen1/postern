# 红队自检 Verify · 详细布局与交互设计

> 控制台「观测」组第三页。本文在[设计基座](00-设计系统与信息架构.md)之上展开，复用其令牌、组件库与统一交互模式，不重定义、不偏离。**纯设计，不含实现代码。**

---

## 1. 页面定位

**一句话**：让运维**显式触发**网关对自己发起一批"本该被拒"的数据面探针，逐条看到九项 PASS/FAIL（FAIL 标红 + `gap_note` 指出缺口防线），把"我以为它会拒"变成"我看到它逐条拒了且留痕"。

- **对应场景**：[场景 07 · 审计、拒绝分析与红队自检](../../docs/examples/07-审计与红队自检.md) §4.1-C「红队自检」、§4.2-E6（任一项未被拒报缺口）、§4.2-E8（重启后复跑反映当前策略）。
- **主要 API**：`POST /v1/verify`（控制面，经 `control.sock`，Agent 物理不可达）。
- **辅助跳转**：`GET /v1/audit`（查 verify 探针逐条留痕，深链到[审计页](02-audit.md)）。

**心智落点**：本页不是规则编辑器，而是**应急自检闸**——它**不改任何策略**（无写、无 `version`、无 409），只对**当前快照**自发只读探针并报告强度。触发是**显式动作，非自动轮询**（基座原则二：UI 只呈现事实，不替系统编建议；自检由人发起、对人负责）。

## 2. 布局

遵循基座的「详情页 / 动作页」骨架变体：本页**非列表 CRUD**，而是**单一动作 + 结果报告**。顶部是触发区与整体判定横幅，下方是九项探针结果列表（`VerifyItemRow` 堆叠，非分页 DataTable——九项是固定有界集，一屏可览，无需后端分页）。

```
┌──────────────────────────────────────────────────────────────────────────┐
│ AppShell 顶栏：品牌 · [GlobalEmergencyBar: Freeze切换 · 模式徽章] · 健康灯 · 主题 │
├────────────┬─────────────────────────────────────────────────────────────┤
│ 左侧导航    │  红队自检 Verify                                              │
│            │  ────────────────────────────────────────────────            │
│ 总览        │  POST /v1/verify · 对当前策略快照逐条发起应被拒探针            │
│ ─观测       │                                                              │
│  审计       │  ┌─ 触发区 ────────────────────────────────────────────────┐ │
│  拒绝分析    │  │ 上次运行：2026-06-13 14:22:07 UTC · policy_rev 1187      │ │
│ ▸红队自检   │  │ 整体判定：[ ✓ ALL PASS ]  (9/9)        [ 运行自检 ▶ ]    │ │

│  授权矩阵    │  │       每条走完整管线 [0]→[10] 并落审计。不改任何策略。     │ │
│  ...        │  └──────────────────────────────────────────────────────────┘ │
│ ─接入       │                                                              │
│  ...        │  ┌─ 整体判定横幅（结果态）─────────────────────────────────┐ │
│ ─系统       │  │  ✗ VERIFY FAILED — 1/9 项防线有缺口                     │ │
│  ...        │  │  下方 FAIL 项已标红;gap_note 指出本应卡在哪条防线。      │ │
│  健康       │  └──────────────────────────────────────────────────────────┘ │
│            │                                                              │

│            │  │ [✗FAIL] ⑤ default_deny_unknown_resource                │  │
│            │  │    防线落点：rbac（不存在资源缺格 + 不泄露存在性）       │  │
│            │  │    ┌ gap_note（原样转述，红底）────────────────────┐    │  │

│            │  │    │ 的存在性(your_grants 含被探测代号)            │    │  │
│            │  │    └──────────────────────────────────────────────┘    │  │
│            │  │ [✓PASS] ⑥ credential_zero_touch  deny响应无机密回显    │  │
```

**展开行（FAIL 或主动点开 PASS 行）**：`VerifyItemRow` 折叠区展示该项的**静态描述**（探针在做什么、预期防线落点、PASS 判据——客户端静态目录，按 `name` 索引）+ **后端事实**（`pass` / `gap_note`）。PASS 行默认折叠，FAIL 行默认展开。

## 3. 数据与状态

### 3.1 展示字段（来自 `POST /v1/verify` 响应 `VerifyReport`）

后端 wire 形态**极简**（`crates/postern-daemon/src/control/verify.rs`）：

| 字段 | 形态 | 渲染 |
|---|---|---|
| `all_pass` | `bool` | 整体判定横幅：`true`→ 绿 `ALL PASS`；`false`→ 红 `VERIFY FAILED` |
| `items[]` | 数组（顺序确定，即 `probe_set` 顺序） | 九项 `VerifyItemRow` 堆叠 |
| `items[].name` | `string`（防线代号，如 `scope_out_mutate`） | 等宽行标识 + 序号；按 `name` 查客户端静态目录取人话描述 |
| `items[].pass` | `bool` | `true`→ 绿 `PASS`；`false`→ 红 `FAIL`（`VerifyItemRow` 固定语义色 + 图标，不仅靠色） |
| `items[].gap_note` | `string \| null` | PASS 时为 `null`（不渲染）；FAIL 时**原样转述**红底缺口说明（基座原则二：原样、不加工、不替系统补话术） |

**客户端静态目录（按 `name` 索引，纯描述性标签，不是后端数据）**：wire 不含每项的"探针在做什么 / 预期防线落点 / PASS 判据"。这些是**已知探针名的固定文案**，由前端静态映射提供，仅用于把 `name` 翻成人话与画防线落点徽章（`StageChip`）。**权威判定永远只取后端的 `pass` / `gap_note`**——静态目录绝不参与判定、绝不覆盖后端结论。九项名固定：`scope_out_mutate` `disguised_write` `session_tamper` `multi_statement` `default_deny_unknown_resource` `credential_zero_touch` `origin_not_trusted` `untrusted_origin_auth_stage` `redaction_probe`。

> **诚实说明（防线落点徽章）**：⑥`credential_zero_touch` / ⑨`redaction_probe` 的 PASS 判据是"响应无机密回显"而非某一 `stage` 拒绝，故其 `StageChip` 标"上游匿名化 / 凭据零接触 + 脱敏出口"而非单一管线阶；⑤ 标 `rbac`（缺格）且附"不泄露存在性"；其余标对应 `Stage`（rbac/classify/auth）。这些徽章来自静态目录，是**对探针意图的标注**，非后端逐项回传的字段。

### 3.2 三态（fail-closed，基座原则一）

- **加载中（自检运行中）**：运行按钮转 `运行中…` 禁用态 + 进度指示（`LoadingSkeleton` 不适用于报告区——报告整体替换，不逐行流式）。九项探针在 daemon 内串行跑完整管线，前端等单次响应返回后整体渲染。运行期间**不显任何旧伪结果**；上次结果横幅置灰标"上次运行（陈旧）"。
- **错误态（请求失败 / daemon 不可达 / 超时）**：`ErrorState` 红色——**绝不**把"请求失败"渲染成 PASS 或 FAIL，也不渲染伪九项。明确文案"自检未能运行：控制面不可达 / 超时"。fail-closed：**不可自检 ≠ 通过**，整体判定区显示"未知（未能运行）"灰态，不显伪绿。
- **空态（从未运行过）**：`EmptyState`——"尚未运行红队自检。点击运行，对当前策略快照逐条验证。"附主操作"运行自检"。本页**无**"列表为空"语义（九项是固定集），空态仅指"本会话/本页尚未触发过"。

## 4. 交互流

### 4.1 主流程：运行自检（本页唯一写触发，且为危险动作）

虽然 `POST /v1/verify` **不改任何策略**（无 `version`、无落库副作用于策略），但它**真实发起 9 条数据面请求**走完整管线（其中含 mutate/destroy 归类的探针、临时低权凭据出示）——这对运行环境是**可感知动作**（落审计、可能触可观测告警），故按基座「危险动作」处理：**显式触发 + 确认**。

```
用户点「运行自检 ▶」
  → ConfirmDialog（危险确认，基座统一）：
      标题：运行红队自检？
      摘要预览（不是策略 diff,是动作摘要）：
        · 以临时低权 Principal 自发 9 条应被拒探针
        · 每条走完整管线 [0]→[10],逐条落审计
        · 不改任何策略(无 policy_rev 前进)
        · 反映当前快照(policy_rev <当前>)
      确认方式：显式勾选「我理解这会发起真实数据面探针并写审计」+ 点「运行」
  → 提交 POST /v1/verify（无请求体）
  → 运行中态（按钮禁用 + 进度;旧结果置灰标陈旧）
  → 响应返回：
     · 成功 all_pass=true  → 绿色整体横幅 ALL PASS(9/9);九项全绿;
                            底部提示「9 条探针已逐条落审计」+ 跳审计链接
     · 成功 all_pass=false → 红色整体横幅 VERIFY FAILED(N/9);
                            FAIL 项标红自动展开,gap_note 原样红底显示;
                            PASS 项折叠;底部跳审计链接
     · 请求失败/超时/不可达 → ErrorState(见 3.2),整体判定灰态「未知」,
                            不显伪九项,不假装通过(fail-closed)
```

**为何用 ConfirmDialog 而非 FormDrawer**：本页无表单字段、无 `version`、无策略写——不走基座「表单→摘要预览→刷新」的 CRUD 写流程。它是**无参动作触发**，故用 `ConfirmDialog`（危险确认）承载"动作摘要预览 + 显式勾选"，跳过 FormDrawer。

### 4.2 次级动作

| 动作 | 系统响应 | 预期结果 |
|---|---|---|
| 点开某 PASS 行 | 折叠区展开静态描述（探针意图 + 防线落点 + PASS 判据） | 看到该项"在验证什么"；无 `gap_note`（PASS 无缺口） |
| 点「查看本次探针在审计中的留痕」 | 深链到[审计页](02-audit.md)，预填筛选：`principal=<verify 临时低权 Principal 名>` + `since=<本次运行起始 ts>` + `decision`（八项 deny + 脱敏项 allow） | 看到 9 条探针逐条审计事件（八项 `deny` 含 `stage`+`reason`、⑨ `allow` 含 `response_digest`）——"声称拒"抬到"看到逐条拒且留痕"（详设 6.7） |
| 复制某项 `name` / `gap_note` | 一键复制（等宽文本） | 便于贴入工单 / 与详设防线条目对账 |
| 重新运行 | 同 4.1 全流程（再次确认） | 反映**最新**快照（详设 E8：重启 / 改策略后复跑，对当前策略重验，非陈旧假设） |

### 4.3 本页危险动作清单（及确认方式）

| 危险动作 | 为何危险 | 确认方式 |
|---|---|---|
| 运行自检 | 发起 9 条真实数据面探针、写审计、可能触可观测告警（虽不改策略） | `ConfirmDialog` + 显式勾选「我理解这会发起真实数据面探针并写审计」 |

> 本页**无**策略写、无 destroy 授予、无 freeze/shutdown/import——那些归各自页面。本页唯一危险动作即"运行自检"本身。

## 5. 复用的基座组件

| 组件 | 本页用途 |
|---|---|
| **AppShell** / **GlobalEmergencyBar** | 全局骨架；顶栏常驻 Freeze 闸 + 模式徽章 + 健康灯（freeze 态下整页顶部红色脉冲横幅照常生效） |
| **VerifyItemRow** | 核心——每项 PASS（绿）/ FAIL（红 + `gap_note` 指缺口防线）；折叠区挂静态描述 |
| **DecisionBadge**（或其语义变体） | 整体判定横幅 `ALL PASS` / `VERIFY FAILED` 的固定语义色（allow 绿 / deny 红） + 图标 |
| **StageChip** | 每项"预期防线落点"徽章（auth/classify/rbac 固定色 + 顺序语义；⑤/⑥/⑨ 为复合落点文案） |
| **ConfirmDialog** | 运行自检的危险确认（动作摘要 + 显式勾选） |
| **ErrorState / EmptyState / LoadingSkeleton** | 三态 fail-closed：错误不显伪数据、未运行显引导空态、运行中禁用按钮 + 进度 |
| **JsonViewer**（可选） | "查看原始报告"折叠区，等宽只读展示 `VerifyReport` JSON（`name`/`pass`/`gap_note`/`all_pass`），可复制贴工单 |

**本页特有的组合（用基座组件拼，非新原子组件）**：

- **触发区卡片** = 上次运行元信息（`ts` + `policy_rev`，等宽） + 整体判定 `DecisionBadge` + `运行自检` 主操作按钮。
- **整体判定横幅** = `DecisionBadge` 放大态 + 计数（N/9） + 一行事实文案（FAIL 时指引"看下方红项 `gap_note`"）。
- **探针结果列表** = 九个 `VerifyItemRow` 垂直堆叠（**非 DataTable / 无分页**：固定有界九项，一屏可览，基座「强制分页」约束针对无界集合查询，此处不适用）。

## 6. 正常与异常预期

### 6.1 正常操作（对照场景 §4.1-C）

| 步骤 | 操作 | 预期结果 |
|---|---|---|
| 1 | 运行自检（确认后） | daemon 以临时低权 Principal 自发九类探针，逐条走完整 `[0]→[10]`；返回 `VerifyReport`；八项 deny 项 `pass=true`、脱敏探测项 `pass=true`（放行无回显），`all_pass=true`；整体横幅绿 `ALL PASS (9/9)` |
| 2 | 九项逐条核对 | 每项绿 `PASS`，`gap_note=null` 不渲染；展开可见各项防线落点（① 越权→rbac、② 伪装写→rbac/Destroy、③ 会话篡改→classify、④ 多语句→classify、⑤ 不存在资源→rbac 且不泄露存在性、⑥ deny 无机密回显、⑦ 自报来源→auth、⑧ 错来源→auth、⑨ 脱敏探测放行无回显） |
| 3 | 查留痕 | 点跳审计页，预填 verify 临时低权 Principal + 本次时间窗，见 9 条探针逐条审计事件（八项 `deny`+`stage`+`reason`、⑨ `allow`+`response_digest`）——留痕可复核（详设 6.7、技设 13.4） |
| 4 | 改策略 / 重启后复跑（详设 E8） | 重新运行反映**当前**快照（横幅元信息 `policy_rev` 前进到当前值）；非陈旧假设；审计跨重启连续可查 |

### 6.2 异常与边界（一律 fail-closed）

| 场景 | 触发 | 预期呈现（fail-closed） |
|---|---|---|
| **某项防线有缺口（E6）** | 某探针未被按预期拒 / 拒在错阶 / 未逐条留痕 | 该项红 `FAIL` 自动展开，`gap_note` **原样**指出缺口（如"拒绝响应泄露了资源 '…' 的存在性(your_grants 含被探测代号)"、"本应在 rbac 阶被拒,实测却放行(防线漏放)"、"… deny 未逐条留痕(审计缺该项)"）；整体横幅红 `VERIFY FAILED`；`all_pass=false`。**绝不假装通过**（详设 E6：把真实强度如实暴露） |
| **请求失败 / daemon 不可达 / 超时** | 控制面桥断 / daemon 关停 / 自检超时 | `ErrorState` 红色明确文案；整体判定灰态"未知（未能运行）"；**不显伪九项、不显伪绿**；不可自检 ≠ 通过 |
| **权限不足（控制面认证失败）** | 401/403（控制面本地凭据缺失 / SO_PEERCRED 不符，详设 L-1） | `ErrorState`"无权运行控制面动作"；**不**降级为"假装跑了"；不显任何探针结果 |
| **运行中重复点击** | 用户在运行态再次点运行 | 按钮禁用态拦截；不并发触发第二次自检 |
| **越权 / 数据缺失（理论上不发生于本页）** | 报告体缺 `items` 或项数 ≠ 9 / 字段缺失 | fail-closed：渲染为整体"未知"错误态，**不**把缺字段项当 PASS；明确"报告不完整，判定无效" |
| **freeze 态下运行** | 全局处于 freeze | 自检照常可触发（freeze 收紧数据面放行，自检探针多为 deny 预期，freeze 不影响其作为应被拒探针的判定）；顶部 freeze 红横幅照常显示，提醒当前为冻结态 |

> **关键 fail-closed 纪律**：本页所有"不确定"（加载中、出错、超时、权限不足、报告残缺）**默认呈现为受限 / 未知态**，绝不乐观显绿。整体判定只有在**收到完整 `VerifyReport` 且 `all_pass=true`** 时才显绿 `ALL PASS`。

## 7. 与后端契约对齐

| 契约 | 本页落点 |
|---|---|
| **`POST /v1/verify` 无请求体、无 `version`** | 本页不走 CRUD 写流程、无乐观锁、**无 409**——它是只读自探针触发，不改策略。确认对话框承载"动作摘要"而非"策略 diff"。 |
| **分页 `page_no/page_size`（缺省 20 钳 200）** | **本页 verify 结果不分页**（九项固定有界集，一次返回，一屏可览）；分页契约**仅作用于**本页跳转的[审计页](02-audit.md)留痕查询（`GET /v1/audit` 后端分页，翻页即再请求，不前端切片）。 |
| **雪花 id 恒字符串、不丢精度** | 横幅 `policy_rev`、跳审计预填的 `principal_id`、审计事件 id 等一律**字符串渲染**，等宽显示、绝不当 number 解析。 |
| **deny 逐字段、不泄露存在性** | ⑤`default_deny_unknown_resource` 正是**主动校验**此点：FAIL 当且仅当 deny 响应 `your_grants` 含被探测代号或 `request_hint` 非空（泄露存在性）。本页对 `gap_note` **原样转述**，不二次加工、不补话术（基座原则二 + deny 渲染纪律）。 |
| **脱敏 / 匿名化在前端也成立** | ⑥`credential_zero_touch` / ⑨`redaction_probe` 的判据即"响应无真实地址 / 凭据明文回显"；本页**永不**显示真实地址 / 凭据明文 / `secret_hash`；探针 `name` 与 `gap_note` 已是脱敏文本，原样展示即合规。 |
| **可追溯（基座原则七）** | 自检不前进 `policy_rev`（非策略写），故**不**提示"policy_rev 前进 + 三联动"；改为提示"9 条探针已逐条落审计"+ 可跳 audit 复核——可追溯落在**审计留痕**而非策略修订。 |
| **真话且只说事实（原则二）** | 整体判定与每项结论**只取后端 `all_pass`/`pass`/`gap_note`**；客户端静态目录仅做 `name`→人话的描述性标注，**绝不**参与判定、绝不覆盖后端结论、绝不替系统编"修复建议话术"（缺口修复方向由人据 `gap_note` 事实裁决）。 |

---

> **设计自检小结**：本页把"主动验证而非假设"钉成一次**显式、可确认、可追溯**的动作——绿 `ALL PASS` 需完整报告 + `all_pass=true` 方成立，任何不确定一律 fail-closed 呈现为未知 / 受限；FAIL 项 `gap_note` 原样指出缺口防线，系统只陈述事实，修复由人裁决。复用基座 `VerifyItemRow`/`DecisionBadge`/`StageChip`/`ConfirmDialog`/三态组件与全部令牌，遵守统一交互模式与后端契约硬约束。
