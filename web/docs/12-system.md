# 12 · 系统：审批队列 / 设置 / 导入导出 / 关停

> 本页是控制台「系统 System」导航组里**非模式**的四块：**审批队列 Approvals**、**设置 Settings**、**导入导出 Import/Export**、**关停 Shutdown**。在 `00-设计系统与信息架构.md` 的统一基座之上展开——复用同一套令牌、组件与统一交互模式，不重定义、不偏离。模式 Mode 单列于 `11-mode.md`、顶栏应急区常驻，本页只在审批/冻结语义交叉处引用，不重复其设计。**纯设计，不含实现代码。**

---

## 1. 页面定位

**一句话**：系统页是控制台的「**配置面 + 应急闸**」——查看本就**默认关闭**的审批队列（多为空/只读，escalate 恒折叠为 deny、超时恒拒）、修订少数全局安全设置（安全 key 取最严默认）、以声明式 TOML 整体导入/导出策略（**永不含明文凭据/真实地址**），以及高危的 `shutdown` 急停。

**对应场景**：
- 场景 03（权限分配与角色管理）——声明式整体 `import`、`admin` 入口对称硬拒、并发写 409。
- 场景 06（动态权限调整与失控切断）——`approval.on_timeout` 恒 deny（`ESCALATE_FOLDS_TO_DENY`）、重启时挂起审批恒 deny、`shutdown` 急停。
- 场景 07（审计与红队自检）——`audit.fsync` / `audit.retention_days` / OTel 导出开关等观测设置，未知 key fail-closed 拒绝。

**主要 API 端点**（取自 `crates/postern-daemon/src/control/router.rs` 的 `CONTROL_ROUTES`，恰覆盖 §6.5）：

| 子页 | 端点 | 说明 |
|---|---|---|
| 审批队列 | `POST /v1/approvals` | 审批默认关闭→列表多为空/只读；本端点用于查询挂起项与（极少数开启时）人工裁决，裁决恒 deny 侧或显式 allow 单次 |
| 设置 | `GET /v1/settings` / `POST /v1/settings` | 读取全部设置（含当前值/默认/是否可写/version）；写单/多个 key（携带 version、未知 key fail-closed 拒绝） |
| 导出 | `POST /v1/export` | 导出声明式策略 TOML（瞬时态不导出，永不含明文凭据/真实地址） |
| 导入 | `POST /v1/import` | 整体导入声明式 TOML（校验失败整体拒绝、无部分 apply；覆盖=高危确认） |
| 关停 | `POST /v1/shutdown` | 优雅关停 daemon（高危确认） |

> **端点形态纪律**：router 中 `settings` 为 `GET/POST /v1/settings`（**整表读 / 写**），场景 07 文中出现的 `PUT /v1/settings/{key}` 是 **CLI 的 per-key 语法糖**，SPA 一律走 `POST /v1/settings` 提交一个 `{key, value, version}` 列表。本页据 router 实际端点设计，不臆造 per-key REST 路径。

## 3. 数据与状态

### 3.1 各 Tab 展示的字段

**审批队列**（`POST /v1/approvals` 查询返回 `Page<ApprovalItem>`，与 `AuditEvent`/`DenyResponse` 同源脱敏）：
- `id`（雪花字符串，等宽截断）、`principal`（代号）、`resource`（`ResourceCode`，永不真实地址）、`capability`（`CapabilityBadge`）、状态（恒 `escalate→deny` / `pending`，但 pending 在审批关闭下不存在）、`policy_rev`、可选 `expires_at`（`TtlBadge`）。
- **不含**：凭据值、`secret_hash`、真实地址、请求负载明文（出口已过 Sanitizer）。

**设置**（`GET /v1/settings` 返回每个 key 的 `{key, value, default, writable, version, kind/枚举约束}`）：
- 已知 key 固定集合：`approval.enabled`、`approval.on_timeout`(锁定 deny)、`audit.fsync`、`audit.retention_days`、`audit.exporter.otel.enabled` 等。
- 每个写 key 带 `version`（乐观锁来源），改动提交逐 key 回传。

**导入导出**：
- 导出返回 TOML 文本（声明式策略快照，不含瞬时态/凭据/地址）。
- 导入 dry-run 返回校验结果 + diff 摘要（新增/变更/删除计数、是否整体合法、覆盖模式将删除哪些实体的计数）。

**关停**：无展示数据，仅一个 danger 动作 + 后果说明（事实，非建议）。

### 3.2 加载 / 错误 / 空三态（fail-closed）

| 态 | 审批队列 | 设置 | 导入导出 | 关停 |
|---|---|---|---|---|
| 加载中 | `LoadingSkeleton` 表骨架 | 卡片骨架（控件灰化、禁编辑） | 区域骨架，按钮禁用 | 按钮禁用 |
| 错误 | `ErrorState`：不显伪数据；写动作（裁决）红色错误、不改本地视图 | `ErrorState`：**禁止任何写**（读不到当前 version 即不可乐观锁），全表只读 + 红错 | 导入预览失败→红错、不进入「应用」；导出失败→红错、不下载半截文件 | 关停请求失败→红错、明确「未关停」（绝不静默以为已关停） |
| 空 | `EmptyState`（§2.1，审批关闭的常态）：陈述「无挂起、escalate 即刻 deny」事实 | 不存在「空」——设置 key 是固定集合；若 `GET` 返回 0 key 视为**错误**（配置面异常）按 fail-closed 报错 | 导入未选文件=引导态；导出无可导出（空策略）=照常导出空声明 | — |

> **fail-closed 总纲**：任何加载中/出错/权限不足/version 缺失，写操作一律不可用（按钮禁用或红错），绝不乐观假装成功（基座原则 1）。设置页尤其：读不到 `version` 就不允许写——乐观锁的期望 version 唯一来源是先前读取值（场景 03 异常 F、场景 06 异常 10）。

## 4. 交互流

所有写操作遵循基座**统一写流程**：表单（RHF+Zod）→ 提交前**摘要预览** →（危险则）ConfirmDialog → 控制面三联动（事务+快照+审计）→ 失效刷新 → 成功提示（`policy_rev` 前进 + 可跳 audit）/ 失败红错（不改视图）/ 409（提示刷新重试）。

### 4.1 逐动作：动作 → 系统响应 → 预期结果

**A · 审批裁决（仅 `approval.enabled=true` 时可达；危险动作）**
- 动作：行 `[裁决]` → 打开 `FormDrawer`，展示该挂起项的脱敏事实（principal/resource/capability/reason），裁决选项 `[拒绝]`（默认）/ `[本次允许]`。
- 系统响应：提交 `POST /v1/approvals`（携带项 `id` + `version` + 裁决）；`[本次允许]` 触发 ConfirmDialog（危险=单次放行扩权）。
- 预期结果：裁决落库 → 写 `policy_change`/审批事件 → 成功提示 + 可跳 audit；该项离队。**超时/未裁的项恒 deny**（系统侧 `on_timeout=deny`，UI 不提供「超时放行」）。

**B · 修改设置（普通 key）**
- 动作：改 `audit.fsync` / `audit.retention_days` / `audit.exporter.otel.enabled` 等 → 累计到「保存改动 (n)」。
- 系统响应：点保存 → 摘要预览（哪些 key、旧值→新值）→ `POST /v1/settings`（逐 key 带 `version`）。
- 预期结果：成功提示（policy_rev 前进）；`audit.retention_days` 超界值在前端钳到 `[1, 上限]`、后端再校验；OTel 开关切换后本地 JSONL 写入不变（旁路，失败不回压）。

**C · 开启审批 `approval.enabled=true`（危险动作）**
- 动作：拨动开关 → 保存。
- 系统响应：ConfirmDialog（显式勾选「我理解：开启审批引入 pending 态；超时/重启仍恒 deny，绝不超时放行」）→ `POST /v1/settings`。
- 预期结果：成功后审批 Tab 出现 `[裁决]` 操作列；`approval.on_timeout` 仍锁定 `deny` 不可改。

**D · 导出 TOML**
- 动作：点 `[导出 TOML ↓]`。
- 系统响应：`POST /v1/export` → 返回 TOML 文本 → 浏览器下载。
- 预期结果：文件为声明式策略，**不含**凭据明文/`secret_hash`/真实地址/瞬时态；导出是读动作，无确认。

**E · 导入 TOML —— 校验（dry-run）**
- 动作：拖拽/粘贴 TOML → 选模式（合并/覆盖）→ `[校验]`。
- 系统响应：`POST /v1/import`（dry-run 标志）→ 返回 diff 摘要 + 合法性。
- 预期结果：合法→展示新增/变更/删除计数，启用 `[应用导入]`；非法→整体拒绝、列出错误、**不启用应用**（无部分 apply）。

**F · 导入 TOML —— 应用（合并 / 覆盖；覆盖=危险动作）**
- 动作：`[应用导入]`。
- 系统响应：**合并**模式→摘要预览→应用；**覆盖**模式→ ConfirmDialog（输入确认词，显示「将删除 N 个实体」计数）→应用。`POST /v1/import`（apply）。
- 预期结果：成功→policy_rev 前进 + 可跳 audit；失败/非法→整体拒绝、库未改、红错；导入含 `name="admin"` 角色→**422 整体拒绝**（入口对称硬拒，场景 03-A）。

**G · 关停 daemon（危险动作）**
- 动作：`[关停 daemon]`。
- 系统响应：ConfirmDialog（输入确认词 `shutdown`）→ `POST /v1/shutdown`。
- 预期结果：成功→daemon 进入优雅关停，UI 转「daemon 不可达」态（健康灯灭、其余写禁用）；失败→红错、明确「未关停」。

### 4.2 本页危险动作清单（一律 ConfirmDialog）

| 危险动作 | 确认方式 | 依据 |
|---|---|---|
| 审批裁决「本次允许」 | ConfirmDialog 勾选确认（单次放行扩权） | 基座危险动作清单 |
| 开启 `approval.enabled` | ConfirmDialog 勾选确认（引入 pending 态） | 场景 06 异常 9 |
| 导入「覆盖」模式 | ConfirmDialog **输入确认词** + 显示删除计数 | 基座（import 覆盖）/ 场景 03 |
| 关停 shutdown | ConfirmDialog **输入确认词** `shutdown` | 基座（shutdown）/ 场景 06-C |

> 导出、合并导入、普通设置改动**非危险**，走摘要预览即可，不强制 ConfirmDialog。

## 5. 复用的基座组件

| 组件 | 本页用途 |
|---|---|
| `AppShell` / `GlobalEmergencyBar` | 全局骨架；顶栏 Freeze 应急开关与模式徽章常驻（本页不重复实现冻结，见 `11-mode.md`） |
| `DataTable` | 审批队列表（强制分页 page_no/page_size、空态、加载骨架、行操作菜单、雪花 id 等宽截断） |
| `FormDrawer` | 审批裁决抽屉（脱敏事实 + 裁决选项 + version 乐观锁） |
| `ConfirmDialog` | 开启审批 / 导入覆盖 / 关停（输入确认词或勾选） |
| `CapabilityBadge` | 审批/导入 diff 中的动词徽章（固定配色、只读） |
| `DecisionBadge` | 审批项的 `escalate→deny` 决策态（escalate_denied 与普通 deny 可区分） |
| `TtlBadge` | 挂起项/带 TTL 的设置回退倒计时（临近过期转琥珀） |
| `JsonViewer / SqlText` | 导入 TOML 预览区、导出预览（等宽、语法高亮、只读、可复制） |
| `EmptyState / ErrorState / LoadingSkeleton` | 三态 fail-closed 呈现（审批空态为常态） |

**本页特有构成**（用基座组件拼，不新增重组件）：
- **Tab 容器**：四子页签的轻量容器（无安全逻辑，仅切换内容区）。
- **设置分组卡片**：以「分组卡片 + 行级 key 编辑」拼装（按 approval/audit 等域分组），锁定 key（`approval.on_timeout`）渲染为只读徽章 + 🔒。
- **导入 diff 摘要块**：用 `CapabilityBadge` + 计数 + `JsonViewer` 拼出「新增/变更/删除」三段预览，覆盖模式额外高亮删除段。

## 6. 正常与异常预期（对照场景规格）

### 6.1 正常操作（逐步预期）

- **审批关闭常态（场景 06）**：进入审批 Tab → `EmptyState`，陈述「无挂起、escalate 即刻 deny」；无任何裁决控件。
- **改观测设置（场景 07-D）**：`set audit.fsync relaxed` → 保存 → policy_rev 前进；本地 JSONL 写入纪律不变（relaxed 仅作用 allow 类，高价值事件恒 fsync）。开启 OTel → 本地权威不变、导出旁路镜像、失败不回压（场景 07-D2）。
- **导出（场景 03 §3.1 声明式）**：导出 TOML，内容与逐条声明等价，不含瞬时态/凭据/地址。
- **导入合并（场景 03 §3.1）**：校验通过 → 摘要预览 → 应用 → policy_rev 前进、热生效、落 `policy_change` 审计。
- **关停后重启（场景 06-E / 07-E8）**：关停 → 重启后策略原样恢复、TTL 按绝对墙钟继续计时、挂起审批恒 deny、审计跨重启连续可查。

### 6.2 异常 / 边界（一律 fail-closed）

| 边界 | 预期（fail-closed） | 依据 |
|---|---|---|
| 加载失败 | 三态 `ErrorState`，不显伪数据；写动作禁用/红错，不改本地视图 | 基座原则 1 |
| 权限不足 / 控制面不可达 | 明确报错「daemon 不可达 / 无权」；不静默以为成功（尤其关停、解冻类） | 场景 06 异常 12 |
| 审批超时 / 重启挂起 | 恒 `deny`（`on_timeout` 不可配 allow；重启挂起恒 deny），UI 无「超时放行」 | 场景 06 异常 9、`ESCALATE_FOLDS_TO_DENY` |
| 设置写未知 key | 后端 fail-closed 拒绝；前端不提供新增任意 key 的入口 | 场景 03 异常 H、场景 07 |
| 改 `approval.on_timeout` 为 allow | UI 锁定不可改；即便绕过前端，后端两处（导入校验+设置写）皆拒 | 场景 06 §2.2 |
| 导入非法 / 含 admin 角色 | 整体 422 拒绝、无部分 apply、库未改 | 场景 03-A、03 异常 C |
| 导入覆盖误触 | ConfirmDialog 输入确认词 + 删除计数二次确认 | 基座危险动作 |
| 并发改设置 / 并发裁决（乐观锁） | 第二个写携过期 version → **409**，原样呈现「他人已改、请刷新重读 version 再改」，不静默覆盖/重试 | 场景 03 异常 F、场景 06 异常 10、场景 07-E10 |
| 关停写失败 | 红错 + 「未关停」，状态保持运行（更安全侧）；解冻类同理保持更严格侧 | 场景 06 异常 14 |
| 导出/审批项越权数据缺失 | 不出现的实体即不可见（悬挂引用不进视图），不暴露其曾存在 | 场景 03 异常 G、场景 07-E5 |

## 7. 与后端契约对齐

- **分页**：审批队列走 `page_no/page_size`，缺省 20、钳 200（`PageQuery::DEFAULT_SIZE=20` / `MAX_SIZE=200`，`clamp` 越界钳不报错），返回 `Page<T>{items,page_no,page_size,total}`；翻页即再次请求，前端不全量切片（`DB_PAGINATION_MANDATORY`）。设置 key 是固定集合、**非集合查询**，不分页。
- **雪花 id 字符串**：审批项 `id`/`principal_id`/`resource_id` 全程字符串、等宽截断显示，**绝不**当 number 解析（>2^53 丢精度）。
- **乐观锁 version**：审批裁决与设置写均携带读取时的期望 `version`；不匹配→**409 Conflict** 原样呈现 + 提示重读，不静默覆盖、不静默重试（version 唯一来源是先前读取值，base 仓储不自读自比）。
- **deny / escalate 逐字段不泄露存在性**：审批/拒绝相关展示按 `DenyResponse` 逐字段渲染——`decision`/`denied`(`resource`/`capability`/`objects`)/`reason`/`your_grants`/`request_hint`/`operator_note`；`operator_note` **原样转述不加工**、未设则不显；`your_grants` 只含主体自身授权世界，不暴露 Scope 外资源/他人权限是否存在。`escalate_denied` 与普通 `deny` 可区分（`Decision::Escalate{fallback}` 折叠为 deny）。
- **脱敏 / 匿名化在前端也成立**：导出/导入/审批一切出口**永不**含明文凭据、`secret_hash`、真实地址、瞬时态（temp_grants/mode_state/审批挂起不导出）；资源只显代号。
- **`approval.on_timeout` 固定 deny**：协议/模型层不接受 allow；UI 锁定只读，是契约 `ESCALATE_FOLDS_TO_DENY` / `on_timeout=deny` 的前端体现。
- **导入整体性**：`POST /v1/import` 校验失败整体拒绝、无部分 apply；覆盖模式删除经显式确认。
- **一切错误 fail-closed 呈现**：加载/写/关停任一不确定即受限/拒绝态，绝不乐观假装成功。
