# 03 · 拒绝分析 Denials

> 本文是 postern 控制台「拒绝分析」页的详细布局与交互设计。遵循[设计基座](00-设计系统与信息架构.md)的视觉令牌、核心组件库、统一交互模式与后端契约硬约束，不重定义、不偏离。**纯设计，不含实现代码。**

---

## 一、页面定位

**一句话**：把近窗口内的 deny 事件按 `(principal × resource × stage × capability)` 聚合成倒序计数榜 + 超阈值告警列表，让运维一眼看出「谁在反复撞哪堵墙」，并据此跳转到 grants/constraints/mode 由人裁决——**只呈现聚合事实，绝不替系统编建议话术、绝不自动放行**。

- **对应场景**：[场景 7 · 审计、拒绝分析与红队自检](../../docs/examples/07-审计与红队自检.md) §1 第二问「哪些越权在反复发生」、§4.1-B、§4.2 的 E7（不自动放行）/E10（并发调整 409）。
- **主要 API**：`GET /v1/denials/summary?window=7d`（控制面读动作，经 `control.sock`，Agent 物理不可达）。
- **本页是纯只读视图**：没有任何写端点。所有"调整"都是**跳转**到对应规则编辑页（Grants/Constraints/Mode），由那些页的统一写流程承载乐观锁/确认/审计。本页自身**不产生** policy_rev 前进。

## 三、数据与状态

### 3.1 展示字段（`GET /v1/denials/summary` 响应 → 视图）

聚合维度与计数（场景 §4.1-B：「按 `(principal, resource, stage, capability)` 分组计数榜，倒序」）：

| 字段 | 来源 | 渲染 | 纪律 |
|---|---|---|---|
| `principal` | 聚合键 | 主体名 + `principal_id`（雪花字符串，等宽截断） | id 恒字符串，绝不解析为 number |
| `resource` | 聚合键 | **ResourceCodeBadge**（代号 + adapter 小图标） | 永不显真实地址 |
| `stage` | 聚合键 | **StageChip**（`auth/classify/rbac/constraint/condition/tier/transport/exec` 固定色）| 只呈现 deny 落点 |
| `capability` | 聚合键 | **CapabilityBadge**（observe…destroy 色温递增）| 只读 |
| `count` | 聚合值 | 数字，列默认倒序排序键 | 含 E3 降级聚合的 `count` 合并值 |
| `intent_digest` | 代表样本 | 等宽截断 + 复制；展开区全展 | **脱敏后摘要**，非请求原文，不含真实地址/凭据 |
| `policy_rev` 范围 | 样本 | 展开区显跨样本 rev 区间 | 提示「其间策略已变」 |
| 告警标记 | 超阈值派生 | 该行左缘琥珀条 + 告警带列项 | 来自可查询的告警事件（详设 6.9） |

窗口汇总（顶部汇总条）：`window`（回显服务端解析后的起止 UTC + 标签如 `7d`）、`deny 事件总数`、`聚合组数`。窗口选择器候选：`24h / 7d / 30d`（默认 `7d`，缺省由 daemon 取默认，前端回显其解析结果而非自行计算起止）。

**不展示**：全部 deny 流水（场景 §4.1-B「不展开全部流水」——逐条看流水去 Audit 页）；任何"建议放行"文案；任何 Scope 外资源真实地址；`secret_hash`/凭据明文/连接串。

### 3.2 三态（fail-closed，基座原则 1）

- **加载中**：聚合榜与告警带显 **LoadingSkeleton**（行骨架）；不显任何旧数据冒充新窗口结果。
- **错误**（端点不可达 / daemon 异常 / 解析失败）：整表替换为 **ErrorState**——红色，文案陈述事实（如「无法读取拒绝聚合：control 端点不可达」），**不显伪数据、不显空榜冒充"无拒绝"**。fail-closed：读不到 ≠ 没有 deny。带 `⟳ 重试`。
- **空**（窗口内确无 deny）：**EmptyState**——中性文案「该窗口内无被拒事件」。这是**真实的好消息**而非错误，与 ErrorState 视觉/语义严格区分（空态可放心呈现，错误态绝不）。空态附引导：去 Verify 页主动自检、或放宽窗口。
- **告警带**：仅当存在超阈值聚合组才出现；无告警时整带隐藏（不占位、不显"0 告警"绿条以免暗示"已审完"）。

## 四、交互流

本页是只读分析页，**自身无写操作、无危险动作**。所有"调整"是把人导向规则编辑页，由那些页承载基座的统一写流程（表单→摘要预览→危险确认→刷新→成功/失败/409）。

| # | 用户动作 | 系统响应 | 预期结果 |
|---|---|---|---|
| 1 | 进入页面 | 以默认 `window=7d` 请求 `GET /v1/denials/summary` | 渲染告警带 + 倒序聚合榜 + 汇总条；首屏即「谁撞哪堵墙」一目了然 |
| 2 | 切换窗口（`7d→30d`） | 以新 `window` 重新请求（TanStack Query 按 `window` 维度缓存/失效） | 表与汇总刷新；榜单随窗口重排；窗口标签回显服务端解析起止 |
| 3 | 点 `⟳ 刷新` | 失效当前 `window` 查询并重取 | 取近窗口最新聚合（deny 是持续写入的副产物，刷新即见增量） |
| 4 | 列排序 / 表内筛选（按 principal/stage/capability） | 客户端在**当前已分页页**内排序/筛选 | 默认按 `count` 倒序；筛选不跨页（强制后端分页，前端不取全量再切片） |
| 5 | 翻页 / 改页大小 | 携带 `page_no/page_size` 重新请求 | 后端按窗口分页返回；前端不缓存全量（对齐 `DB_PAGINATION_MANDATORY`） |
| 6 | 展开某行 `▸` | 展开只读细节面板（阶段说明 / `intent_digest` 全展 / rev 区间 / 跳转入口） | 见聚合事实与脱敏样本；**不**逐条拉流水 |
| 7 | 点告警项 `[定位↓]` | 滚动并高亮榜内对应行 | 告警与榜单同源对齐，便于先看高危组 |
| 8 | 行操作「查看 Grants 这一格 →」 | **路由跳转**到 Grants 页，预选 `agent1 × db-main` 授权格 | 离开本页；调整在 Grants 页走其统一写流程（含确认/409） |
| 9 | 行操作「复制 elevate 模板」 | 复制 `postern elevate agent1 --cap db-main:mutate --ttl <填>` 到剪贴板 | **机械生成的命令模板**（占位 TTL 需人填），非可点执行的放行按钮；放行须人去 Grants 显式写入（E7） |
| 10 | 行操作「查 agent1 全部 deny →」 | 跳转 Audit 页，预填 `principal=agent1 & decision=deny & since=窗口起` | 逐条流水在 Audit 页查（本页只给聚合，不给流水） |

**危险动作清单**：**本页无危险动作**（无 destroy/扩权/freeze/吊销/import/删除）。所有具破坏性/扩权性的动作都发生在被跳转到的目标页（如 Grants 的 `elevate`、Mode 的 freeze），由那些页的 **ConfirmDialog** 承载二次确认——本页只负责**定位问题、把人带到正确的编辑入口**，绝不在本页内放行。这正是 E7 的 UI 落地：聚合榜再醒目也只是"信号"，不是"放行依据"，系统不提供任何"一键放行高频组"的捷径。

## 五、复用的基座组件

| 组件 | 本页用途 |
|---|---|
| **AppShell / GlobalEmergencyBar** | 全局骨架与顶栏常驻应急区（freeze/mode/健康灯） |
| **DataTable** | 聚合榜主体：列排序（默认 `count` 倒序）、列筛选、**强制分页**（`page_no/page_size`，缺省 20/钳 200）、加载骨架、空态、行展开、行操作菜单；`principal_id` 列等宽截断 |
| **ResourceCodeBadge** | `resource` 列：代号 + adapter/transport 小图标，永不显真实地址 |
| **StageChip** | `stage` 列：deny 阶段固定色（`auth…exec`），顺序语义 |
| **CapabilityBadge** | `capability` 列：动词固定配色（observe 蓝灰 … destroy 红，色温递增），只读 |
| **EmptyState / ErrorState / LoadingSkeleton** | 三态：错误态不显伪数据；空态（确无 deny）与错误态严格区分 |
| **TtlBadge** | （仅在展开区 elevate 模板的 `--ttl` 占位提示中作视觉引用，无倒计时） |

**本页特有组件（均用基座组件拼装，不引入新令牌）**：

- **AlertBand（告警带）**：顶部琥珀色横条列表，每项 = `主体×资源×阶段×动词 + count + ≥阈值 + [定位]`。由基座 `--warn` 令牌 + StageChip/CapabilityBadge/ResourceCodeBadge 组合而成；仅在有超阈值组时渲染。
- **DenialGroupRow（聚合行）**：DataTable 行的内容组合 = `principal` + ResourceCodeBadge + StageChip + CapabilityBadge + `count` + 等宽 `intent_digest` 截断。
- **DenialDetailPanel（展开细节）**：只读面板，复用等宽样式呈现 `reason`（原样转述策略事实）、`intent_digest` 全展（可复制）、`policy_rev` 区间，并内嵌跳转入口与 elevate 模板复制按钮。**不**用 FormDrawer（无写操作）。

## 六、正常与异常预期

### 6.1 正常（对照场景 §4.1-B）

1. **进入/默认 7d**：返回近 7 天 deny 聚合，按 `(principal, resource, stage, capability)` 倒序，如 `agent1 × db-main × rbac × mutate = 37`、`agent1 × redis-main × rbac × manage = 31` → 榜首即最频繁被拒模式（§4.1-B.1）。
2. **读榜 + 告警**：超阈值组在告警带醒目高亮，并在榜内对应行左缘标琥珀；告警来自**可查询的告警事件**（详设 6.9）。榜单只陈述「`agent1` 反复请求 `db-main:mutate`」这一模式，**不**附判断话术（§4.1-B.2）。
3. **展开看样本**：见该组 `intent_digest` 代表样本（脱敏摘要）+ `reason`（`role=observer 不含 db-main:mutate`，原样引用策略事实）+ rev 区间，**不展开全部流水**。
4. **据此裁决**：人选择跳 Grants 做 `elevate`（确需该权限）或跳 Audit 查 agent1 调用方（疑注入/失控）；调整在目标页写入、热生效、落 `policy_change` 审计——下一窗口该组 count 应下降，闭环可观测（§4.1-B.3）。本页只把人**带到**正确入口。

### 6.2 异常 / 边界（一律 fail-closed）

| 场景 | 触发 | 本页预期（fail-closed） |
|---|---|---|
| 加载失败 | `denials/summary` 端点不可达 / daemon 异常 | **ErrorState**（红、陈述事实），不显伪数据、不显空榜冒充"无拒绝"；带重试。读不到 ≠ 无 deny |
| 真实空窗口 | 窗口内确无 deny 事件 | **EmptyState**（中性，"该窗口内无被拒事件"），与 ErrorState 严格区分；引导去 Verify 主动自检 |
| 权限 / 越权 | 控制面认证未过（基座 CONS-17） | 整页 fail-closed 拒绝呈现，不泄露任何聚合内容；引导重新认证（由 AppShell 统一处理） |
| 分页 | 聚合组数 > 页大小 | 强制后端分页（`page_no/page_size`），前端不取全量再切片；翻页即再次请求（对齐 `DB_PAGINATION_MANDATORY`） |
| 大窗口/高基数 | `30d` 组数巨大 | 仍倒序分页呈现；汇总条给总组数让人知"还有多少"；前端不试图一次拉爆 |
| 越权数据缺失 | 某组样本字段缺失（如 `intent_digest` 为空） | 展开区该字段显"—"占位，**不臆造**；其余字段照常；绝不用占位伪装成有值 |
| E7 滥用为放行依据 | 榜首醒目，想"因次数多就放行" | 本页**无任何放行控件**；只提供机械 elevate 模板（需人填 TTL）+ 跳转入口；放行须人去 Grants 显式写入并留审计。fail-closed：不确定即维持拒绝 |
| E10 并发调整 409 | 两人据同一榜首各改同一授权格 | **冲突发生在 Grants/Constraints 目标页**（携期望 `version`，第二写得 `409`，并写 `policy_change` 记录冲突）；本页是无锁读，并发查询彼此无冲突；回到本页刷新即见最新聚合 |
| E8 跨重启 | daemon 重启 | 审计 append-only 不丢历史，重启后聚合仍可查（含上一 `policy_rev`）；本页刷新即反映当前策略下的窗口聚合 |
| E3 存储降级 | 审计逼近配额，deny 按窗口聚合写 `count` | 本页 `count` 直接吃合并后的聚合计数，不因降采样丢失 deny 类信号；汇总/榜单仍可信（高价值 deny 不被降采样丢弃） |

## 七、与后端契约对齐

- **分页**：聚合组集合一律分页，`page_no/page_size` 缺省 20、上限钳 200，翻页即重新请求，前端绝不取全量再切片（对齐 `DB_PAGINATION_MANDATORY`）。窗口 `window` 落查询键（缺省由 daemon 取默认，前端回显其解析后的起止 UTC）。
- **雪花 id**：`principal_id`（及样本携带的任何 id）全程**字符串**渲染、等宽截断、绝不解析为 number（基座 §3.4）。
- **deny 逐字段、不泄露存在性**：聚合维度 `resource` 仅以**代号**呈现（ResourceCodeBadge），`intent_digest` 是**脱敏后摘要**而非请求原文；`reason` 原样引用策略事实、不加工。本页是控制面、运维有权看全自身平面的聚合，但呈现纪律与数据面一致——**永不**显真实地址/凭据明文/`secret_hash`/连接串（对齐基座硬约束、场景 E4）。注意：数据面侧"不泄露 Scope 外资源存在性"由 `DenyResponse.your_grants` 的 scope-bounded 保证（verify 项 4 校验），本页不触碰该返回。
- **不编建议话术（公理六的运维侧边界）**：面向人的机械推导（`postern elevate <principal> --cap <res>:<cap> --ttl <填>` 模板）允许，因为它是**聚合事实 + 标准命令模板**而非编造话术；TTL 留占位需人填，模板是**复制项**不是放行按钮（场景 §4.1-B.2、详设 8.9）。
- **写端点乐观锁**：本页无写端点，故本页不直接处理 `version`/409；但所有由本页跳转触发的调整，在目标页携带期望 `version`，409→提示刷新重试（E10）。本页"刷新"是无锁读，与人写互不阻塞。
- **一切错误 fail-closed**：读失败→ErrorState 不显伪数据；字段缺失→占位"—"不臆造；不确定→维持"被拒"语义，绝不乐观假装"已无 deny"。
