# 10 · 主体与凭证 Principals / Credentials

> 本页详细设计。基座（令牌/组件/信息架构/统一交互模式/后端契约硬约束）见 `00-设计系统与信息架构.md`，本页只描述本页布局、状态、交互流与所用组件/端点，**复用基座、不重定义、不偏离**。纯设计，不含实现代码。

---

## 一、页面定位

**一句话**：管理网关的**身份名册**——左半册登记每个具名主体（Agent/程序/人，`kind`），右半册登记每个主体的**网关凭证**（`local_process`/`api_key`/`token` 的元数据：可信域、有效期、吊销态），凭证只显示元数据、**永不显 `secret_hash`/明文**，`api_key` 明文仅创建时一次性显示后即不可见；**吊销是热生效的应急动作，与删除分离**。

- **对应场景**：`docs/examples/02-资源接入与认证配置.md`（主体即 agent1/agent2/agent3 等具名主体；凭证即各主体经网关认证所持的网关凭证——区别于资源侧 `vault://{code}/{tier}` 的下游账号，后者归 09 资源页）。
- **主要 API**：
  - `GET /v1/principals`、`POST /v1/principals`
  - `GET /v1/credentials`、`POST /v1/credentials`
- **页内分工**：本页是**网关身份面**（主体 ↔ 它认证用的网关凭证）。资源侧的下游凭据等级（`credential_tiers`、`auth_flow`、`vault://` 引用）属资源接入，归 09 资源页；本页不碰资源凭据明文，亦不碰真实地址。

> 心智对齐：「预设决定一切、人只修订预设」。本页全部是**修订身份名册**（持久、异步、可审），不是实时审批。唯一的"即时"动作是**吊销凭证**——它是应急闸的一部分，热生效。

## 二、布局

本页采用基座**列表页骨架**，但承载两类强相关实体（主体、凭证），故用**左右双栏 master–detail**：左栏主体列表为主，选中一个主体后右栏聚焦该主体的凭证。主操作按当前栏上下文切换。

```
┌──────────────────────────────────────────────────────────────────────────────┐
│ AppShell 顶栏：品牌 │ daemon 健康灯 │ 全局模式徽章 │ [Freeze 应急开关] │ 主题   │  ← 基座常驻
├───────────┬────────────────────────────────────────────────────────────────────┤
│ 左侧导航   │  主体与凭证 Principals / Credentials              [+ 新建主体]       │
│ ─观测      │  ┌────────────────────────┬──────────────────────────────────────┐ │
│ ─授权      │  │  主体名册 Principals     │  凭证 Credentials · {选中主体名}      │ │
│ ─接入      │  │  ┌──────────────────┐  │  ┌────────────────────────────────┐ │ │
│   资源     │  │  │ 筛选: kind▾ /搜名 │  │  │ 摘要: agent · 共 N 凭证          │ │ │
│  ▸主体凭证 │  │  └──────────────────┘  │  │  其中 1 已吊销 · 1 即将过期       │ │ │
│ ─系统      │  │  DataTable(主体)        │  └────────────────────────────────┘ │ │
│           │  │  ┌──────────────────┐  │  [+ 新建凭证]   (仅选中主体时可用)    │ │
│           │  │  │○ name │kind│凭证#│⋮│  │  ┌────────────────────────────────┐ │ │
│           │  │  │● agent1│agent│ 2 │⋮│←│  │  CredentialMetaCard 列表(该主体)   │ │ │
│           │  │  │○ svc-cr│progr│ 1 │⋮│  │  ┌──────────────────────────────┐ │ │ │
│           │  │  │○ ops   │human│ 0 │⋮│  │  │ kind=api_key  可信域=local    │ │ │ │
│           │  │  └──────────────────┘  │  │ 有效期 TtlBadge · 状态: 生效  │ │ │ │
│           │  │  分页: page_no/page_size │  │ id 7300…0123  [复制]  [⋮吊销] │ │ │ │
│           │  │  缺省20 钳200  共 T 条   │  ├──────────────────────────────┤ │ │ │
│           │  │                        │  │ kind=token   状态: 已吊销(红) │ │ │ │
│           │  │                        │  │ revoked_at … (只读,不可再吊销)│ │ │ │
│           │  │                        │  └──────────────────────────────┘ │ │ │
│           │  └────────────────────────┴──────────────────────────────────────┘ │
├───────────┴────────────────────────────────────────────────────────────────────┤
│                              右侧抽屉位 FormDrawer（新建主体 / 新建凭证）         │
└──────────────────────────────────────────────────────────────────────────────────┘
```

- **左栏（主体名册）**：基座 `DataTable`，列 = `name` / `kind` 徽章 / **凭证数**（生效计数，吊销/过期不计，悬浮拆分）/ `id`（等宽截断）/ 行操作 `⋮`。强制分页（页脚 `page_no`·`page_size`，缺省 20 钳 200）。顶部筛选条：按 `kind` 单选筛、按 `name` 搜。主操作 `[+ 新建主体]` 常驻右上。
- **右栏（凭证）**：未选主体时为引导空态（"选择左侧一个主体以查看其网关凭证"）；选中后展示该主体的 `CredentialMetaCard` 列表 + 摘要条 + `[+ 新建凭证]`。凭证以卡片（非密集表）呈现，因每条凭证字段少但状态语义重，卡片利于把「吊销态/过期临近」放到醒目位置。
- **创建经右侧 `FormDrawer`**（基座统一），不离开本页。

> 双栏取舍：主体与凭证是 1–N 强从属，且"看一个主体有哪些凭证、哪条该吊销"是本页高频任务，master–detail 比两张独立表更贴合，仍完全由基座组件拼成（两个区域各自是基座列表/卡片骨架），不引入页面级新交互范式。

## 三、数据与状态

### 3.1 主体（左栏，`GET /v1/principals` → `Page<PrincipalRow>`）

| 字段 | 来源 | 呈现 |
|---|---|---|
| `id` | snowflake，**协议恒为字符串** | 等宽截断 `7300…0123` + 悬浮全展 + 复制；**绝不当 number 解析**（丢精度红线） |
| `version` | 乐观锁基线 | 不单独显示，作为编辑/删除时回传的期望 version 隐式携带 |
| `name` | 归一化落库值 | 正文字号；左栏主标识 |
| `kind` | `agent` / `program` / `human` | 小徽章（中性语义色，区分三类身份；不复用能力动词色） |
| 凭证数（生效） | 由 `GET /v1/credentials` 按 `principal` 聚合得出（生效=未吊销且未过期） | 数字；悬浮拆"生效 N · 吊销 M · 过期 K" |

### 3.2 凭证（右栏，`GET /v1/credentials` → `Page<CredentialMeta>`，过滤到选中主体）

逐字段对齐后端 `CredentialMeta`，**`secret_hash` 字段存在于后端但本页一律不取/不显**：

| 字段 | 来源 | 呈现 |
|---|---|---|
| `principal` | 所属主体 id（字符串） | 用于过滤到右栏；不重复显示 |
| `kind` | `local_process` / `api_key` / `token` | `kind` 徽章（固定文案，只读） |
| 可信域 | 凭证适用的来源域（如 `local`/可信网段标识，元数据） | 标签；越界来源用受限凭证会在数据面被拒（场景 §4.2-⑥），本页仅声明 |
| `expires_at` | 过期时刻，可空 | `TtlBadge` 倒计时；临近过期转琥珀；`null`=长期有效（文字标注，不留空） |
| `revoked_at` | 吊销时刻，可空 | 有值=**已吊销**（红色状态徽章 + `revoked_at` 时刻）；无值=生效 |
| `secret_hash` | 后端有 | **永不显示**（基座原则六；`CredentialMetaCard` 约定不含此字段） |
| 明文/`api_key` 值 | 仅 `POST` 创建 `api_key` 的**响应一次性返回** | 一次性弹窗显示 + 复制，关闭后**不可再得**；列表/详情永不回显 |

- **凭证状态机（派生，不是独立字段）**：
  - **生效**：`revoked_at == null` 且（`expires_at == null` 或未到期）→ 绿/中性。
  - **即将过期**：`expires_at` 临近 → 琥珀 `TtlBadge`。
  - **已过期**：到期 → 灰/警示（认证层 deny，热生效）。
  - **已吊销**：`revoked_at != null` → 红，**终态**，不可再吊销、不可"取消吊销"（吊销不可逆，UI 不提供反向动作）。

### 3.3 三态（fail-closed）

- **加载中**：左栏 `LoadingSkeleton`（表格骨架行）；右栏选中主体后凭证区骨架卡。**绝不**先渲染空表佯装"无数据"。
- **错误**：`ErrorState`（红，fail-closed）——例：`GET /v1/principals` 失败 → 左栏整体错误态，**不显任何伪行**；`GET /v1/credentials` 失败 → 右栏错误态，主体名册仍可用但凭证区显式标"凭证加载失败，无法确认吊销/有效状态"。错误态下**不展示任何凭证为'生效'**（不确定即按受限呈现）。
- **空**：
  - 主体名册空 → `EmptyState`（带 `[+ 新建主体]` 引导）。
  - 选中主体但无凭证 → 右栏 `EmptyState`："该主体暂无网关凭证，新建后该主体方能经网关认证"（如实陈述后果，不编建议话术）。
  - 未选主体 → 右栏中性引导态（非错误）："选择左侧一个主体查看其网关凭证"。

## 四、交互流

所有写操作走基座**统一写流程**：表单（RHF+Zod）→ 提交前**摘要预览** →（危险则）`ConfirmDialog` → 失效刷新 → 成功提示（policy_rev 前进 + 可跳 audit）/ 失败红色错误（不改本地视图）/ 409 冲突（提示刷新重试）。

### 4.1 新建主体（`POST /v1/principals`）

1. 点 `[+ 新建主体]` → 右侧 `FormDrawer` 打开。
2. 表单：`name`（必填，归一化校验）+ `kind`（单选 `agent`/`program`/`human`，必选）。
3. 提交前**摘要预览**："将登记主体 `agent1`（kind=agent）。该主体初始**无任何凭证、无任何授权**——存在但默认拒绝一切（公理一）。"
4. 提交 → 成功：drawer 关、左栏失效刷新、toast"主体已登记，policy_rev → N，可查看 audit"（链接 `policy_change` 事件）；新行高亮。
5. **非危险**动作（仅登记一个空主体，不扩权），不需 `ConfirmDialog`。

### 4.2 新建凭证（`POST /v1/credentials`，仅在选中主体时）

1. 右栏 `[+ 新建凭证]` → `FormDrawer`，已绑定当前 `principal`（只读显示所属主体）。
2. 表单：`kind`（`local_process`/`api_key`/`token`）+ 可信域 + 有效期（可选，空=长期有效）。
   - **`local_process`**：通常无明文录入（进程上下文凭证），表单据 `kind` 自适应不要求 secret。
   - **`token`**：录入既有令牌值（明文经控制面转发、本地不留存、提交后不回显）。
   - **`api_key`**：由 daemon 生成，**表单不录入明文**。
3. **明文零接触纪律（基座原则六 + 场景公理四）**：任何 secret 字段**不回显、不落本地、提交后即清**；表单永不显示也不接收 `secret_hash`。
4. 提交前**摘要预览**："为主体 `agent1` 新建 `api_key` 凭证，可信域=local，有效期=30d。"
5. 提交 → 成功：
   - **`api_key` 特例（一次性显示）**：成功响应**一次性**返回新 `api_key` 明文 → 弹出 `ConfirmDialog` 样式的**一次性展示框**（等宽 + 复制 + 醒目提示"此值仅显示一次，关闭后不可再获取，请立即妥存"）。关闭即不可再得；列表只显元数据。
   - 其余 `kind`：直接 toast 成功。
   - 右栏凭证列表失效刷新、左栏该主体"凭证数"刷新、toast"凭证已创建，policy_rev → N，可查看 `credential_event`"。
6. **危险性**：创建凭证本身是扩面动作但不破坏既有；**不强制 `ConfirmDialog`**（摘要预览足够），仅 `api_key` 因明文一次性而走一次性展示框。

### 4.3 吊销凭证（应急动作，热生效）—— 本页**最高危**动作

> 吊销 ≠ 删除。吊销是把凭证置为 `revoked_at != null`，**认证层即时拒绝**（场景 §4.2-⑤：已吊销凭证认证层拒），热生效、不依赖扫描器、不可逆。

1. 凭证卡片行操作 `⋮` → "吊销凭证"（红色危险样式）。
2. 弹 `ConfirmDialog`（基座危险确认，需**显式勾选/输入确认**，防误触）：
   - 摘要："吊销主体 `agent1` 的 `api_key` 凭证（id `7300…0123`）。"
   - 后果直述（只说事实，不安抚）："**热生效**：吊销后该凭证一切认证**即时被拒**，正在进行的会话按 fail-closed 处理。**不可撤销。** 此操作不删除凭证记录（区别于删除），仅置吊销态。"
   - 不显示 `secret_hash`/明文。
3. 确认 → `POST /v1/credentials`（吊销变更，携该凭证期望 `version`）→ 成功：卡片转**已吊销**红态（终态）、`revoked_at` 显示、行操作收起（不可再吊销）、左栏"凭证数"减、toast"凭证已吊销，热生效，policy_rev → N，可查看 `credential_event`"。
4. 失败：红色错误，**卡片状态不变**（仍显原状态，不佯装已吊销）。

### 4.4 删除主体 / 删除凭证（逻辑删除，与吊销区分）

- 后端写路径支持逻辑删除（`delete_flag`，乐观锁）。**删除主体**：行操作 `⋮` → "删除主体"，`ConfirmDialog` 需确认；摘要含"将逻辑删除主体及其归属判定，**不等于吊销其凭证**——若需立即切断认证，请先吊销其凭证"。删除主体若其下仍有生效凭证，提示先吊销（fail-closed，避免留下'已删主体仍可认证'的悖态）。
- **删除凭证**：与吊销并列但语义不同——删除是名册清理（逻辑删除记录），**吊销才是应急切断**。`ConfirmDialog` 明示二者区别："删除从名册移除该凭证记录；若意图是**立即停止认证**，应使用'吊销'（热生效）。"

### 4.5 本页危险动作清单（一律 `ConfirmDialog`）

| 动作 | 端点 | 确认方式 | 可逆性 |
|---|---|---|---|
| **吊销凭证**（应急、热生效） | `POST /v1/credentials` | `ConfirmDialog` + 显式勾选；摘要含"热生效·不可撤销·非删除" | 不可逆 |
| 删除主体（逻辑删除） | `POST /v1/principals` | `ConfirmDialog`；下有生效凭证则提示先吊销 | 逻辑删除（受撤销策略约束，本页按不可逆呈现） |
| 删除凭证（逻辑删除） | `POST /v1/credentials` | `ConfirmDialog`；明示≠吊销 | 同上 |

> 创建主体/创建凭证为非破坏性，走摘要预览即可，不入危险清单（`api_key` 的一次性展示框是信息提示，非危险确认）。

## 五、复用的基座组件

| 基座组件 | 本页用途 |
|---|---|
| **AppShell / GlobalEmergencyBar** | 全局骨架；顶栏 Freeze 应急开关与模式徽章常驻（与本页吊销互补：吊销切单凭证，Freeze 切全局） |
| **DataTable** | 左栏主体名册：`kind` 筛选、`name` 搜、**强制分页**（`page_no`/`page_size` 缺省 20 钳 200）、空/错/加载三态、行操作菜单、雪花 id 等宽截断 |
| **CredentialMetaCard** | 右栏每条凭证：`kind`/可信域/有效期/吊销态；**约定不含 `secret_hash`/明文** |
| **TtlBadge** | 凭证 `expires_at` 倒计时，临近过期转琥珀 |
| **FormDrawer** | 新建主体 / 新建凭证（RHF+Zod、摘要预览、409 提示、成功 policy_rev 前进） |
| **ConfirmDialog** | 吊销凭证 / 删除主体 / 删除凭证；危险确认需显式勾选 |
| **EmptyState / ErrorState / LoadingSkeleton** | 三态 fail-closed（左栏表态、右栏凭证区态、未选主体引导态） |
| 雪花 id 呈现纪律（基座 §3.4） | 主体 id、凭证 id 等宽截断 + 悬浮全展 + 复制；全程字符串 |

**本页特有组件构成（用基座组件拼，不新增范式）**：

- **PrincipalCredentialPanel（双栏壳）**：左 `DataTable`（主体）+ 右凭证区，纯布局容器，选中态联动；不含安全逻辑。
- **CredentialStatusBadge**：派生状态徽章（生效/即将过期/已过期/已吊销），由 `CredentialMetaCard` 内嵌，复用基座语义色（生效=allow 绿/中性、临近=warn 琥珀、吊销/过期=deny 红）。
- **ApiKeyRevealDialog**：`api_key` 创建后的**一次性明文展示框**——以基座 `ConfirmDialog` 容器 + 等宽 + 复制拼成，单按钮"我已妥存（关闭后不可再得）"。仅展示创建响应回传的明文，不读列表、不读 `secret_hash`。

## 六、正常与异常预期（对照场景 02）

### 6.1 正常操作每步预期

| 步骤（场景对应） | 操作 | 预期结果 |
|---|---|---|
| 登记主体（场景前置：agent1/2/3、svc-*） | 新建主体 `agent1`(kind=agent) | 左栏出现该主体，凭证数=0；右栏空态"暂无凭证"；toast policy_rev 前进；该主体此刻**默认拒绝一切**（公理一），无凭证不可认证 |
| 录入网关凭证 | 为 `agent1` 新建 `api_key` | 一次性展示框显示明文（仅此一次）；列表只显元数据（kind/可信域/有效期/吊销态）；**`secret_hash` 不可见**；toast 落 `credential_event`（只记 kind/主体，不记 secret） |
| 有效期声明 | 设 `expires_at`=30d | `TtlBadge` 倒计时；临近转琥珀；空=长期有效（文字标注） |
| 应急切断（场景 §4.2-⑤ 的正向操作） | 吊销 `agent1` 的某凭证 | 确认后转红"已吊销"终态，`revoked_at` 显示；**热生效**——该凭证认证即时被拒；toast 落 `credential_event` |

### 6.2 异常 / 边界（一律 fail-closed）

| 编号 | 场景 | 预期（fail-closed） |
|---|---|---|
| **加载失败** | `GET /v1/principals` 或 `/v1/credentials` 失败 | 对应区 `ErrorState`，**不显伪行/伪卡**；凭证区错误时**不把任何凭证渲染为'生效'**（不确定即受限）；提供重试 |
| **凭证加载失败但需判断吊销** | 右栏 `GET /v1/credentials` 失败 | 右栏显式"无法确认凭证有效/吊销状态"，不佯装生效；左栏"凭证数"显示"—"而非 0 |
| **权限不足 / 越权** | 控制面拒绝（本页应仅经 `control.sock`，正常运维可达；若返回 403/受限） | 受限态呈现，不暴露细节；不显任何凭证字段（fail-closed） |
| **分页边界** | `page_no` 越界 / `page_size` 超 200 | 后端 `clamp`（`page_no`→≥1，`page_size`→[1,200]）；前端回显**钳后真实 `page_no/page_size/total`**，不假装请求值生效 |
| **乐观锁 409**（场景 E7） | 两运维并发改同一主体/凭证，或 SPA 与 CLI 并发；吊销时他人已先改 | 后写者收 `409 Conflict` → `FormDrawer`/`ConfirmDialog` 内红色"他人已修改该记录，请刷新后重试"，**不静默覆盖**；刷新取最新 `version` 再操作；吊销失败时凭证**保持原状态** |
| **越权数据缺失 / Scope 外** | 引用不存在或越界的主体/凭证 | **不泄露存在性**——不回"该 id 不存在"这类可探测信息；按受限/空呈现（对齐 deny 不暴露 Scope 外资源存在性） |
| **吊销不可逆边界** | 试图"取消吊销"或重复吊销 | UI **不提供**反向动作；已吊销卡片行操作收起；重复吊销路径不存在 |
| **机密面未解锁（场景 E6）** 的旁证 | 凭证创建/校验依赖机密面，未解锁时后端 fail-closed | 写操作失败红色错误"机密保险箱未解锁，无法写入凭证"，**不降级为明文落库**，本地视图不变 |

## 七、与后端契约对齐

- **分页**：`GET /v1/principals`、`GET /v1/credentials` 一律分页（`DB_PAGINATION_MANDATORY`）。请求 `page_no`（1-based）/`page_size`；缺省 `page_size=20`，**钳 [1,200]**（`PageQuery::clamp`，越界钳不报错）。响应信封 `Page<T> = { items, page_no, page_size, total }`——前端**以响应回显的钳后值**为准渲染页脚。
- **雪花 id 全程字符串**：`principal.id`、凭证 `id` 在协议里恒为字符串（`>2^53`），前端**绝不当 number 解析**，等宽截断 + 悬浮全展 + 复制。
- **写端点乐观锁**：编辑/吊销/删除携带先前读取的期望 `version`；不匹配 → `409 Conflict`，提示刷新重试（期望 version 唯一来源是先前读取值，不臆造）。
- **凭证字段纪律**：`CredentialMeta` 含 `secret_hash` 字段，但 UI **永不取、永不显**；凭证明文（含 `api_key`）只在创建响应一次性返回，列表/详情零回显（基座原则六 + 公理四）。
- **吊销热生效**：`revoked_at != null` 即认证层即时拒绝（重新检 wall clock，不依赖扫描器）；UI 据此将吊销呈现为**应急、热生效、不可逆终态**，与逻辑删除分离。
- **写三联动可追溯**：每个写端点 = 一次事务 + 快照重建 + 审计三联动；成功提示 **policy_rev 前进**并可跳对应 `policy_change`/`credential_event`（`credential_event` 只记 kind/主体/tier，**绝不记 secret 值**）。
- **deny 不泄露存在性**：越界/不存在的主体或凭证不回可探测的"存在/不存在"信号，按受限/空 fail-closed（对齐 deny 渲染逐字段、不暴露 Scope 外资源存在性）。
- **控制面隔离**：本页全部经 `control.sock`（`0600`，Agent 物理不可达），凭证明文经控制面转发由 daemon/机密面处置，**SPA 本地不存储/不派生明文**。
