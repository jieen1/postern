# postern — 本地安全网关

> AI Agent 与一切基础设施之间的唯一通道。**Agent 的 IAM + 防火墙，跑在你自己掌控的信任域内。**

postern 是一个本地安全网关：它站在 AI Agent 与真实基础设施（服务器、数据库、容器、API、集群等）之间，Agent 一侧只能看到被授予的"能力"；凭据、IP、实例标识、拓扑、协议细节全部存在于另一侧，对 Agent 不可见。所有访问由系统强制鉴权，默认拒绝，权限可动态调整，全程审计。

**当前状态**：本仓库处于设计阶段，核心资产是 `docs/` 下的三份设计文档（产品层 / 系统层 / 实现层）；产品代码实现尚未开始，仓库已就位 Stele 契约基础设施（`contract/` + `build.rs` + `tests/contract/`）以在第一行实现代码之前就锁住数据库层与安全核心的不变量。

## 要解决的问题

让 Agent 安全地操作真实基础设施，核心矛盾有四：

1. **凭据暴露** — Agent 的上下文会读取不可信内容，任何进入上下文的密钥都可能被提示注入窃取。凭据必须永不进入 Agent 可达范围。
2. **权限失控** — Agent 可能误用高权限操作，或被注入诱导执行破坏性动作。权限必须分级、精确、由系统强制，而非依赖 Agent 自觉或文档约定。
3. **信息泄露** — 服务器 IP、实例 ID、内网拓扑本身即敏感资产，也是攻击踏板，不应进入 Agent 视野。
4. **人不在环** — Agent 操作高频且常在无人值守时进行。安全机制不能依赖"人实时审批"才成立。

## 设计公理

任何具体决策若与公理冲突，以公理为准：

| # | 公理 | 含义 |
|---|---|---|
| 一 | **默认拒绝** | 未被显式授予的操作一律拒绝；授权是白名单 |
| 二 | **失败即关闭** | 一切不确定状态（配置缺失、解析失败、凭证无法验证、审批超时、依赖不可用）解析为拒绝，绝不放行 |
| 三 | **系统强制，而非约定** | 边界由系统在执行路径上强制；只靠 Agent 配合才安全的边界视为不存在 |
| 四 | **凭据零接触** | 凭据、真实地址、拓扑标识永不离开网关进程，永不出现在返回给 Agent 的任何数据中 |
| 五 | **单一信任域，边界可放大** | 始终运行在用户掌控的信任域内（本机或组织内网）；不提供公共云托管形态 |
| 六 | **网关说真话，且只说事实** | 返回给 Agent 的一切要么是策略与系统的客观事实，要么是资源所有者亲笔预写的内容，绝不代为生成建议或话术 |
| 七 | **入口对称，语义一致** | 同一类操作无论经 MCP、HTTP、CLI 还是 SDK 进入，都走同一条求值与执行管线，安全语义完全一致 |

## 核心模型

系统的全部行为建立在五个概念之上：

| 概念 | 含义 | 要点 |
|---|---|---|
| **Principal** | 谁在请求 | 每个 Agent 实例、程序、用户都是独立具名主体；授权与审计的基本单位 |
| **Resource** | 被接入的东西 | 对 Agent 匿名化：只见代号（如 `db-main`），代号↔真实地址映射仅网关内部持有 |
| **Role** | 信任等级 | 一组 Capability 的集合，可继承；保留角色 `admin` 在模型层硬约束不可授予任何 Principal |
| **Scope** | 辖区 | 授权作用于哪些 Resource，与 Role 正交 |
| **Capability** | 权限原子动词 | `observe / query / mutate / execute / manage / destroy`，正交、逐项授予 |

授权关系：`Principal —绑定→ Role × Scope —展开→ (Resource × Capability × 细则)` 判定矩阵。每个授权格可附加细则（表白名单、容器前缀、参数化命令模板、路径规则）与条件谓词（限流、时间窗、模式、临时授权 TTL）。

## 系统如何运作

单一 headless daemon 承载全部逻辑，控制面与数据面物理隔离：

```
   ┌─ 数据面(Agent 与程序) ─┐        ┌─ 控制面(人类管理) ─┐
   MCP 外壳   HTTP 外壳   SDK         RPC ── CLI / 管理前端
      │         │         │            │
      └────┬────┴─────────┘            └──────┬───────────
           ▼                                  ▼
   ┌──────────────────────────────────────────────────────┐
   │                  网关核心 daemon                       │
   │   数据面请求内核 · 策略引擎 · 连接管理 · 传输 · 适配器  │
   │   身份与凭证 · 机密保险箱 · 匿名化映射 · 审计与观测     │
   └──────────────────────────────────────────────────────┘
                              │
                   任意 Transport(长/非长连接)
                              ▼
                 Resources(服务器 / DB / 容器 / API / ...)
```

- **控制面**（人类管理）默认只绑定进程间通信，不监听网络端口——Agent 物理上触达不到策略、审计、凭据与映射表。
- **数据面**（Agent 与程序）的每个请求都完整走过统一求值管线：

```
归一化 → 认证(含凭证可信域) → 语义归一化(协议感知) → RBAC 判定
→ 细则校验 → 条件谓词 → allow/deny/escalate → 获取连接 → 执行
→ 响应脱敏 → 审计(allow 与 deny 都记)
```

任一环节判定拒绝即短路。语义归一化必须协议感知：对 SQL 做语法树解析，识破查询外壳下隐藏的写操作（如 `WITH x AS (DELETE …) SELECT`）；`execute` 类能力只接受参数化模板的实例化，不存在自由命令通道。

## 权限哲学

> **预设决定一切，拒绝引导一切，人只修订预设。**

绝大多数请求由预设（角色 + 作用域 + 细则）毫秒级确定裁决，完全不依赖人在场。人对权限的介入方式是**修订规则**（持久、异步、不阻塞），而非**批准请求**（单次、同步、阻塞）。审批默认关闭，`escalate` 折叠为 `deny`，绝不超时放行。

拒绝不是终点，而是网关与 Agent 之间的结构化对话：

```json
{
  "decision": "deny",
  "denied": { "resource": "db-main", "capability": "mutate" },
  "reason": "role=observer 不含 db-main:mutate",
  "your_grants": { "db-main": ["query"], "app-prod": ["observe"] },
  "request_hint": "<申请该权限的标准命令，默认带 TTL>",
  "operator_note": "(可选：资源所有者亲笔预写的指引，网关原样转述)"
}
```

Agent 据此自我修正、依现有授权另寻路径，或向用户陈述事由并附上现成的提升命令——人方便时一条命令授权（默认带 TTL，到期自动回收）。模式（观察/维护/冻结）可一键切换；发现失控时吊销凭证或冻结模式，均热生效。

## 匿名化与脱敏

- **请求侧**：Agent 只能以资源代号发起请求，无法枚举 Scope 之外的资源；代号到真实地址的解析只发生在传输层内部。
- **响应侧**：一切返回内容（含错误与拒绝响应）经脱敏，擦除内网 IP、主机名、实例 ID、连接串、密钥令牌及资源所有者声明的业务敏感字段。

两侧合拢，Agent 无论正常工作、被注入诱导还是主动探测，都无法获得资源真实身份与凭据。

## 可扩展性

可替换/可扩展点均为接口隔离的插件，新增实现不触及核心：

| 插件点 | 已知形态 |
|---|---|
| **Transport**（到达资源，统一抽象为本地 socket） | SSM、SSH、Tailscale、Cloudflare Tunnel、直连 |
| **Adapter**（理解资源协议，映射动词与对象） | postgres / mysql（SQL 语法树解析）、容器运行时、HTTP、键值存储、集群编排 |
| **Credential Provider**（资源凭据来源） | 静态保险箱（OS 密钥链/本地加密存储）、动态签发、证书 |
| **审计导出器**（可选，默认关闭） | 对齐开放遥测规范（OpenTelemetry 语义）向外部投影 |
| **远端探针**（可选） | 资源侧轻量只读组件，仅用于本机才能取的能力 |

## 存储与信任域

- **策略状态**是系统管理的权威状态（嵌入式库，事务读写、热生效），不是静态文件；声明式文件只是可选的导入/导出表示，且永不含明文凭据与真实地址。
- **审计**是 append-only 事件流，本地自给自足；allow 与 deny 都记录。
- **机密**（资源凭据、代号↔真实地址映射）隔离持有，数据面无任何读取路径。

信任域覆盖**单机**（个人桌面）与**组织内网**（自部署、团队共享）两档；放大时替换认证器、状态存储、审计写入等显式抽象的隔离点，核心模型与求值逻辑不变。不做公共云托管——将用户的密钥与生产访问能力托管于第三方，与本产品的安全定位根本矛盾。

## 仓库结构

```
postern/
├── docs/
│   ├── 本地安全网关-核心设计记录.md     # 产品层
│   ├── 本地安全网关-技术设计文档.md     # 系统层
│   ├── 本地安全网关-详细设计文档.md     # 实现层
│   ├── modules/                        # 模块级详细设计（00 索引 + 每 crate 一篇）
│   └── examples/                       # 配置与授权示例
├── contract/                          # Stele 契约（不变量的锁定声明，受保护）
│   ├── main.stele
│   ├── proposals/agent-additions.stele # 经受限通道提案并锁定的契约（DB 层 + 安全核心）
│   └── sql-exceptions.json            # 裸 SQL 豁免登记（受保护，初始为空）
├── build.rs                           # 契约扫描器接线：扫描工作区并喂 fixture（受保护）
├── tests/contract/                    # 由契约生成的测试（受保护，不手工编辑）
├── src/                               # Rust 源码（占位）
├── stele.config.json                  # Stele 配置（受保护）
└── CLAUDE.md                          # Agent 协作行为准则与契约保护说明
```

本仓库由 Stele 契约机制保护（Stele 是独立项目，非本仓库的一部分）：不变量声明在 `contract/` 下，`build.rs` 把工作区扫描结果喂给生成的契约测试，受保护文件经 SHA-256 哈希锁定，Agent 无法静默修改它们——这本身就是"系统强制，而非约定"公理的工程实践。每条规则都附带一条反例自检（teeth）以保证扫描器真实生效。
