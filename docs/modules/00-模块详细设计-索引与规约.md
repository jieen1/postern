## 每篇模块文档的统一结构

每篇模块文档必须含以下小节，顺序一致：

1. **定位（一句话）** — 这个 crate 在系统中是什么。
2. **承载领域与职责范围** — 对应第八部分哪些领域；本 crate 负责什么（封闭列举）。
3. **支持的功能** — 该 crate 对外提供的能力清单（按对外接口组织）。
4. **明确边界（不做什么）** — 显式排除项，每项指明"归哪个 crate / 领域"。
5. **对外接口** — 该 crate 暴露给其他 crate 的类型与 trait（设计级签名，非实现）；标注哪些是定义、哪些是实现。
6. **与相邻模块的交互** — 对每个有依赖关系的相邻 crate，逐一写明：**方向**（谁调谁）、**内容**（传什么类型/数据）、**时机**（在求值管线/启动序列的哪一步）、**失败语义**（出错时如何 fail-closed）。
7. **必守不变量** — 该 crate 必须守住的规则，标注由哪条 Stele 契约 / clippy lint / 构造函数签名强制。
8. **验收标准** — 实现者按"做到什么程度算这个模块完成"的**可验收清单**（见下）。

## 验收标准（第 8 节）的写法规约

第 8 节是该模块的**验收基准**——拿这份清单可逐条判定开发实现的"功能写全没、逻辑对不对"。**合格验收标准的铁律：每条对当前代码只有"通过/不通过"一个答案，无歧义、可复现。** 禁止"应该没问题/基本可用/大体实现了"这类无法判定的描述。

**统一结构**（样板见 `01-postern-core.md` §8）：

- 开头一句说明：每条 = 要求 + 通过判定；答案二元、可复现；判定方式按条目而定（不强求都是测试）。
- 按三组组织，每组一张表，列为 `编号 | 要求 | 通过判定`：
  - **一、功能完整性**（判断：该有的功能都写了吗、行为对吗）—— 逐条对应第 3 节功能（必要时含第 5 节接口的存在性）。要求=必须实现什么；通过判定=该功能存在且"给定输入→产出确切可观察结果"。
  - **二、逻辑正确性**（判断：关键逻辑、边界、失败处理对不对）—— 覆盖关键规则、边界、以及全部关键 fail-closed 失败路径（错误→拒绝、不可建→拒绝、不可记→不放行、过期→即刻失效等）。通过判定="触发某条件→行为恰为某可观察结果"。
  - **三、边界与不变量**（机器强制，绿/红即答案）—— 逐条对应第 4 节边界与第 7 节不变量中**可被机器判定者**，通过判定直接写"契约 `<名>` 绿 / `cargo tree|deny` / `clippy -D warnings` 退出码 0"。
- **每条"通过判定"必须可观察、二元**：给确切输入/操作 + 确切预期结果（如"返回 `Deny` 且 `stage=auth`""契约 X 绿""`clamp(201)==200`"）；不写过程描述。
- **优先挂到既有产物**：24 条 Stele 契约、场景规格（`docs/examples/02~07` 的正常/异常预期）、red-team `postern verify` 九项、依赖图 `ARCH_FORBIDDEN_EDGES`。
- 无法机器判定、只能靠行为观察或人工 checklist 的项，**如实标注**，且仍写成"满足某确切条件即过"的二元形式（人工项拆成逐项 yes/no，全 yes 才过）。
- 以 **通过定义（DoD）** 收尾：一句话——三组每一条都通过即该模块完成，任一条不过即不通过。
- 纯设计：描述"要验证什么、怎么算过"，不写测试实现代码。

## 权威依赖图（唯一事实来源）

源自《详细设计文档》3.2，由契约 `ARCH_FORBIDDEN_EDGES` 强制。各模块文档的"交互"小节必须与此图一致，不得描述被禁止的依赖。

```
                 postern-core         ← 不依赖工作区内任何 crate（零 IO）
                /   |    |   \
       postern-store |  postern-transports  postern-adapters
                postern-secrets
                     \   |   /
                  postern-daemon       ← 唯一组装点（二进制 posternd）
                      |
                 postern-cli           ← 仅依赖 core（共享类型）+ HTTP/UDS 客户端
```

**允许的依赖边**：
- 所有 lib crate（store/secrets/transports/adapters）→ `core`（消费领域类型与插件 trait 定义）。
- `daemon` → core / store / secrets / transports / adapters（唯一组装点，可依赖全部）。
- `cli` → core（共享请求/响应/分页类型）+ HTTP-over-UDS 客户端依赖。

**禁止的依赖边**（违反即 `ARCH_FORBIDDEN_EDGES` 红）：
- `adapters` ↛ `secrets` / `transports` / `store`
- `transports` ↛ `store`
- `cli` ↛ `store` / `secrets`
- `core` ↛ 任何工作区 IO crate

## 通用约定

- 接口签名是**设计承诺**（与《详细设计文档》第四部分一致），实现可调内部细节但不得违背签名与不变量。
- 机密类型（`ResolvedTarget`/`ResourceCredential`/`PresentedCredential`）：不可 Clone/Serialize、`Debug=REDACTED`、无 `Display`；构造点受 `SEC_CONSTRUCTION_SITES` / `SEC_SECRET_TYPE_DISCIPLINE` 约束。
- 一切失败沿 fail-closed 解析为拒绝（公理二）；求值路径禁吞错（`EVAL_NO_ERROR_SWALLOWING`）。
- 文档语言：模块文档为中文；代码/标识符/接口名为英文（与上层一致）。
