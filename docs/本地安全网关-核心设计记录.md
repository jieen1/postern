# 本地安全网关(Local Security Gateway for AI Agents)— 核心设计记录

> 本文档沉淀产品讨论的全部核心结论,作为后续技术设计的依据。

---

## 1. 问题起源与实践验证

产品构想来自一套真实落地的实践:让 AI Agent(Claude Code,运行于 WSL)安全访问一台零公网暴露的 AWS 生产服务器(Docker 部署,SSM-only 接入)。

最终落地的手工方案:

- **信任边界分离**:凭据、AWS CLI、SSM 隧道全部在 Windows 侧;Agent 在 WSL 侧,通过 wsl.conf 关闭 automount 与 interop,使 Agent 物理上无法读取凭据或执行 Windows 程序
- **传输**:SSM 端口转发将服务器能力映射到本地端口(业务 HTTP、数据库、日志 API),由保活脚本维持
- **只读强制**:数据库专用 SELECT-only 账号(数据库引擎层强制);日志通过自建 40 行只读 log-api(白名单容器前缀、无 inspect、剥离 ANSI 码),不暴露 docker.sock 能力
- **Agent 说明书**:CLAUDE.md 只描述 localhost 入口与用法,不含任何服务器信息;明确"通道断开即报告,不自行诊断"
- **执行类需求的化解**:Agent 在本地 venv 跑项目代码 + 连只读库,替代"在生产容器内执行脚本",生产侧零执行通道

关键经验教训(产品功能的直接来源):

1. 凭据放环境/边界外,文档只放引用 —— 凭据永不进 Agent 上下文
2. docker 组 = root;docker.sock 的 :ro 不限制 API 操作 —— 权限必须在正确的层强制
3. SSM 默认 ssm-user 带完整 sudo —— 默认配置往往不安全
4. Dozzle 无对外 API 且新版含 shell/actions 功能 —— 引入组件须验证其真实能力面
5. "默认不会"≠"不能":提示注入与 Agent 自主探索是真实攻击面,安全须系统强制而非约定
6. 端口未发布到宿主机等环境问题是接入的常见坑 —— 产品应自动探测并代修

## 2. 市场结论

- **无现成端到端产品**。MCP 网关(Docker MCP Gateway / Lasso / Open Edison 等)假设后端可达,不解决隧道/传输;零信任访问产品(Teleport / Boundary / Tailscale 等)面向人、需在服务器装组件、不懂 Agent/MCP;现成只读 MCP server 只是零件
- **核心空白**:无人把"传输层生命周期管理"与"面向 Agent 的受限工具面"接在一起
- 关键抽象:**万物收敛于本地 socket,传输方式(SSM/SSH/WS/Tailscale/CF Tunnel…)只是插件** —— 这是与现有 MCP 网关的根本差异点
- 需求证据:Dozzle 社区存在"为 AI Agent 提供日志 REST API/MCP"的公开 feature request

## 3. 产品定位

**本地安全网关**:AI Agent 与一切基础设施之间的唯一通道。

- Agent 一侧只看到"能力"(capabilities);凭据、IP、实例 ID、region、拓扑、协议细节全部在另一侧不可见(资源匿名化)
- 所有权限由系统强制、默认拒绝、动态可调、全程审计
- 只读不是关键,**严格、分级、动态的权限控制**才是产品本体
- 一句话:**"Agent 的 IAM + 防火墙,跑在你本地。"**

目标用户:独立开发者 / 小团队全栈工程师(1~5 台生产机器,Docker 为主,使用 Claude Code/Cursor);次级:中小公司平台工程师(为团队 Agent 统一发受控访问)。

## 4. 核心模型

| 概念 | 含义 | 要点 |
|---|---|---|
| **Principal** | 谁在请求(每个 Agent 实例独立身份) | token 认证;授权与审计主体 |
| **Resource** | 被接入的东西(服务器/DB/K8s/API) | 对 Agent 匿名化:只见代号(如 db-main),网关持有代号↔真实地址映射 |
| **Role** | 信任等级(observer/debugger/operator…) | 可继承;admin 硬编码不可授予 Agent |
| **Scope** | 辖区(该 Principal 管到哪些 Resource) | 与 Role 正交:Role 定动作,Scope 定范围 |
| **Capability** | 权限原子动词 | observe / query / mutate / execute / manage / destroy;每格可挂细则(表、容器前缀、命令模板、路径规则) |

授权关系:`Principal —绑定→ Role × Scope —展开→ (Resource × Capability × 细则) 矩阵`

权限体系 = **RBAC 为骨(角色组织授权)+ Capability 动词为原子(执行时精确判断)+ 条件层为筋(限流/时间窗/Mode/TTL,ABAC 性质)**。

## 5. 架构分层

```
Agent们(MCP / localhost HTTP,各带 Principal 身份)
─────────────────────────────────────────────
本地安全网关
  控制面:策略引擎 + 管理 CLI/控制台(与数据面端口隔离,Agent 物理够不到)
  数据面:协议适配器(postgres/docker/http/k8s/redis…)
          请求解析 → 策略求值 → 执行 → 响应脱敏
  传输面:Transport 插件(SSM/SSH/Tailscale/CF Tunnel/直连…)
          隧道生命周期:按需拉起、保活、重连
  机密面:凭据保险箱(OS keychain/加密存储)+ 代号↔真实地址映射表
  观测面:审计流(append-only)、denials 分析、告警
─────────────────────────────────────────────
Resources(任意 transport 接入;可选只读远端探针,网关自动部署)
```

强制性设计原则:

- **协议感知校验**:SQL 必须过 AST 解析(识破 `WITH x AS (DELETE…) SELECT` 类伪装);shell 命令只支持参数化模板,绝不自由字符串
- **双向脱敏**:不仅过滤请求,更过滤返回内容(日志中的内网 IP、连接串、token 等擦除)
- **fail-closed 贯穿全部**:没配=拒绝、超时=拒绝、引擎故障=拒绝
- **网关自保**:控制面与数据面隔离;策略与审计文件对 Agent 不可读;审计 append-only;内置红队自检(postern verify 模拟越权请求逐条确认被拒)

## 6. 权限哲学(最重要的结论)

> **预设决定一切,拒绝引导一切,人只修订预设。**

### 6.1 审批的定位(经反复确认)

- 人不在场是常态,Agent 操作高频,任何"挂起等人"的设计都不可作为承重墙
- 决策三值:`allow / deny / escalate`
- **审批默认关闭**;关闭或超时时 escalate ≡ deny(fail-closed),绝不超时放行
- 审批仅作为"人在场模式"的可选便利功能,开启常带 TTL
- 人的介入方式是**改规则(策略,持久、异步、不阻塞)**,而非**批请求(单次、同步、阻塞)**

### 6.2 结构化拒绝:网关与 Agent 的对话协议

拒绝响应必须可被 Agent 消费与推理,且**网关说的每句话要么是策略事实,要么是人亲笔写的,绝无系统代拟**(动态建议由 Agent 基于事实自行推理,网关不编造 alternatives 文案):

```json
{
  "decision": "deny",
  "denied": {"resource": "db-main", "capability": "mutate"},
  "reason": "role=observer 不含 db-main:mutate",
  "your_grants": {"db-main": ["query"], "app-prod": ["observe", "execute:diag_*"]},
  "request_hint": "postern elevate <principal> --cap db-main:mutate --ttl 15m",
  "operator_note": "(可选,资源所有者在配置 deny_notes 中亲笔预写,网关原样转述)"
}
```

拒绝后的三级自动分流(人最少介入):

1. Agent 误用 → 读 reason 自我修正【人无感知】
2. 存在替代路径 → 基于 your_grants 自行推理绕行【人无感知】
3. 确需权限 → Agent 在对话中向用户陈述事由 + 现成 elevate 命令 → 人异步一条命令授权(默认带 TTL)【人粘贴一次】

安全细则:拒绝信息只描述该 Principal 自己的世界,不暴露系统内其他资源/角色/权限(防拓扑探测)。

### 6.3 动态策略管理

- 全部热生效(毫秒级),不重启不断隧道
- **临时授权默认带 TTL,到期自动回收**(安全口子自动愈合);永久变更是例外
- Mode 预设(观察/维护/冻结)一键切换,可带 TTL;`postern freeze` 一键全局拉闸
- 策略变更本身进审计
- `postern denials --suggest`:被拒日志 → 策略调整建议 → 预设持续进化(闭环)
- 与 Agent 能力正向共生:Agent 越聪明,自动分流消化比例越高,人被打扰越少(与审批制相反,为理念护城河)

## 7. 核心流程

**流程 1 — 请求求值(毫秒级)**:认证 Principal → RBAC(Role×Scope 格子存在?)→ 细则(AST/模板/路径)→ 条件(限流/时间窗/Mode/TTL)→ allow 则经 transport 执行 → 响应脱敏 → 审计(allow 与 deny 都记)

**流程 2 — 拒绝分流**:deny + 结构化响应 → Agent 自行修正 / 绕行 / 向人申请

**流程 3 — 策略管理**:`postern grants / elevate --ttl / revoke / role edit / freeze / denials --suggest`

**流程 4 — 资源接入(5 分钟体验)**:`postern init` → 选 transport → 探测资源(发现并代修端口未发布等问题)→ 生成只读账号/部署探针 → 凭据入保险箱 → 自动生成 CLAUDE.md 片段 → `postern verify` 红队自检

## 8. 商业与生态判断

- open-core:单机全功能开源(安全工具不开源无人信任);收费线在团队与合规(多人共享、集中审计、SSO、审批流、托管控制面)
- 竞争挤压来自三个方向:MCP 网关补 transport、网络产品(Tailscale 类)做 Agent 策略层、Claude Code 原生能力扩张;护城河在"开箱即用的组合 + 对踩坑路径的理解"
- 窗口期判断:12~18 个月
