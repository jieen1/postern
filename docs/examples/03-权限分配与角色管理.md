# 场景 3 · 权限分配与角色管理

> 把已接入的异构资源（`svc-*`/`docker-A`/`db-main`/`redis-main`/`mq-main`，见示例一）的差异化权限灵活分配给 `agent1`/`agent2`/`agent3`——用**资源无关的角色阶梯** + **标签 Scope** + **对象细则** + **凭据等级**四轴组合表达，且随时可调、热生效。本场景钉死「如何把一句"agent2 管 docker、读 db、管 redis"的意图，无歧义地落成系统强制的授权矩阵」的预期。

本规格的授权模型、命名、矩阵以《技术设计文档》3.3/3.4/3.5、《详细设计文档》5.2 schema、5.2bis 授权落地为准；与示例一（01-多资源多Agent授权示例）的资源代号、Agent、角色、tier 完全一致。七公理为最高准绳。

---

## 1. 用户需求

> 我（资源所有者/运维）已经把两台机器上的资源接入网关（场景 2）。现在我要给三个 Agent 配差异化权限，意图很明确，但我**不想为每种资源类型各造一套角色**，也不想逐条手写"agent2 在 docker-A 上能 manage、在 db-main 上只能 query……"这种 N×M 的格子。我要的是：

- **agent1**：A、B 上**所有服务只读**——日志、状态、数据都能看，但一个字都不能改。
- **agent2**：A 的 **docker 管理**（重启业务容器）+ B 的**数据库只读**（看业务表，PII 脱敏）+ B 的 **redis 管理**（管 `cache:` 前缀键）。
- **agent3**：**业务服务管理（用系统跑业务流程，即提交表单/推进流程）** + **docker 查日志**（只看不重启）+ B 上**所有服务日志**。

且我要求：

1. **"管理"在不同 Agent 处指向不同资源面**——agent2 的"docker 管理"是重启容器（`docker-A:manage`），agent3 的"业务服务管理"是提交业务表单（`svc-*:mutate`），二者绝不能串味。
2. 用**角色阶梯**（信任等级递进）+ **标签选择器**（"prod 全部""A 上 docker""B 上全部"一句话圈定）表达，新增同类资源打对标签即自动纳入。
3. 同一资源对不同 Agent 给不同动词、不同对象细则、不同引擎账号（tier）。
4. **随时可调**：改一条 binding 或 constraint，下一次求值即按新策略，无需重启 daemon、无需 Agent 配合。
5. **绝对不能**把 `admin`（改网关自身配置的能力）授予任何 Agent。

**本场景覆盖的系统核心功能**：RBAC 授权（Principal —绑定→ Role × Scope —展开→ Resource × Capability × 细则 × tier，《技术设计》3.5、《详细设计》5.2bis）；标准角色阶梯 observer/operator/maintainer（资源无关、经继承，5.2bis ①）；标签 Scope 选择器（`{all:[{key,value}]}`，5.2bis ②）；对象细则（`http_route`/`container_prefix`/`key_prefix`/`table_allow`/`column_mask`/`command_class`/`command_template`，5.2bis ④）；tier 选择（动词→凭据等级，10.5）；`admin` 不可授予的模型层硬约束（《技术设计》3.3、《详细设计》8.1/8.10）。

## 2. 预期实现路径

postern 把"灵活分配差异化权限"落成一个**网格上的点的集合**：授权 = `(信任等级 Role) × (辖区 Scope) × (动词 Capability) × (对象细则 constraint) × (条件 condition) × (引擎账号 tier)`（5.2bis）。本场景只动其中前四轴 + tier，条件轴（time_window 等）见场景 6。

**这是一组控制面动作**（写 `policy.db`），全部经 `control.sock` 上的控制面 API（6.5）；每次写入 = **一次事务 + 快照重建（policy_rev 递增）+ 审计事件**（6.2、5.3 `policy_change`），改完即热生效——数据面下一次 `Evaluator::evaluate` 读到新 `Arc<PolicySnapshot>` 即按新策略判定（6.2 无锁原子替换）。本场景不产生任何数据面动作（Agent 的实际请求见场景 4），但每条授权的**正确性以求值管线 [3] RBAC / [4] 细则 / [6] tier 选择能据此判定**为验收口径。

逐步落地路径：

1. **声明标准角色阶梯（资源无关，经继承）**（5.2bis ①、5.2 `roles`/`role_inherits`/`role_capabilities`）。角色只描述动词集，动词含义由各 Adapter 解释（`observer` 给 DB 是 SELECT、给 docker 是看日志、给 http 是 GET）：
   - `observer` = {observe, query}
   - `operator` = `observer` ⊕ {mutate, execute}（经 `role_inherits` 显式继承，不是隐式蕴含，《技术设计》3.4）
   - `maintainer` = `operator` ⊕ {manage}
   - `destroy` 不进任何标准角色，须 `(资源 × destroy)` 单格 + TTL 显式授予（本场景三个 Agent 均不授予 destroy）。
   - `admin` **不可声明、不可授予**：`roles.name CHECK(lower(trim(name)) <> 'admin')` + `Capability` 枚举无 Admin 变体（双重硬约束，5.2、8.1 契约 `SEC_ADMIN_NOT_GRANTABLE`）。

2. **给每个 Principal 挂 binding（role × scope）**（5.2 `bindings`/`binding_scope`、5.2bis ②）。scope 两种 `kind`：
   - `selector`：按 `resource_labels` 标签匹配，如 `{all:[{key:'env',value:'prod'}]}`（"prod 全部"）、`{all:[{key:'host',value:'A'},{key:'kind',value:'docker'}]}`（"A 上 docker"=精确 `docker-A`）、`{all:[{key:'host',value:'B'}]}`（"B 上全部"）。
   - `resource`：枚举具体资源代号（如 `db-main`、`redis-main`）。
   - 差异化（三个 Agent 三种意图）靠**多条 binding** + **不同 scope** 表达，而非为每种资源造角色（避免组合爆炸，5.2bis ①）。

3. **挂对象细则收窄动词作用面**（5.2 `grant_constraints`、5.2bis ④，控制面 `POST /v1/resources/{code}/constraints`）。细则 `kind` 由对应 Adapter 声明支持哪些：
   - `docker-A`：`container_prefix=app-`（只限业务容器）。
   - `db-main`：`table_allow`（业务表白名单）+ `column_mask`（PII 列脱敏）。
   - `redis-main`：`key_prefix=cache:` + `command_class`（容许的命令类）。
   - `svc-*`：`http_route`（method+path 收窄到业务流程接口）+ `mask_fields`（PII 字段脱敏）。
   - 细则挂在**资源 × 动词**上（资源级），对所有在该格被授权的 Principal 生效；同格多行**取交集**（同 `kind`）/恒 `AND`（不同 `kind`），fail-closed（5.2 约束合并语义）。

4. **tier 选择由策略引擎按动词自动完成**（10.5、6.3、求值步骤 [6]/[7]）。运维只在资源上声明 tier→动词映射（场景 2 已做），本场景**不直接选 tier**：求值时引擎按本次动词选承载它的 tier（observe/query→只读 tier，manage→admin tier……），无承载该动词的 tier→deny。本场景的验收点是"展开矩阵后每格落到正确 tier"。

5. **授权矩阵展开校验**：选择器在**快照构建时**展开为当时匹配的具体资源集（5.2bis ②、6.2），授权空间（绑定 × 角色继承）随之物化进 `PolicySnapshot`（8.11 内容清单）。运维用 `postern grants <principal>` / `GET /v1/grants/{principal}`（与 Agent 侧 `your_grants` 同源，6.5）确认展开结果与意图一致。

6. **随时可调、热生效**（6.2）：改 binding/constraint/role → 控制面写事务 COMMIT → 同一写锁内重建快照 → 数据面无锁读新快照。无需重启、无需 Agent 配合（公理三：系统强制不靠 Agent 配合）。

> **领域归属**（《详细设计》第八部分）：角色/绑定/细则的语义合法性裁决归领域模型与控制面（8.10）；RBAC 展开与 tier 选择归策略引擎（8.3，纯函数）；快照构建归存储层（8.11）；CLI/SPA 只发起与渲染、零安全逻辑（8.12、07-postern-cli §4）。

## 3. 页面与操作

三类入口中，本场景主要落在**控制面**（CLI / SPA）——权限分配是运维的管理动作。数据面（Agent）侧仅作"分配结果如何被 Agent 看到"的旁证。

### 3.1 CLI（`postern ...`，人/脚本用）

对应控制面端点见 6.5 与 07-postern-cli §3。

- **声明角色阶梯**（一次性，或经 import）：
  ```bash
  postern role add observer   --cap observe --cap query
  postern role add operator   --inherits observer   --cap mutate --cap execute
  postern role add maintainer  --inherits operator   --cap manage
  # admin 无法声明：postern role add admin ... → 控制面 422，模型层硬拒（见 4.2-A）
  ```
- **绑定 role × scope**（`role` 命令组的 `bind` 子命令，落 `POST /v1/bindings`，07-postern-cli §3）：
  ```bash
  # agent1：prod 全部只读
  postern role bind agent1 --role observer \
      --scope-selector 'all=env:prod'
  # agent2：A 上 docker 管理 + db 只读 + redis 管理（三条 binding）
  postern role bind agent2 --role maintainer --scope-selector 'all=host:A,kind:docker'
  postern role bind agent2 --role observer   --scope-resource db-main
  postern role bind agent2 --role maintainer --scope-resource redis-main
  # agent3：业务服务 operator + docker 查日志 + B 上全部日志（B 上是纯"看日志"=observe，用单格授予而非 observer 角色，避免带进 query）
  postern role bind agent3 --role operator  --scope-selector 'all=env:prod,kind:business'
  postern role bind agent3 --role observer  --scope-resource docker-A
  postern role bind agent3 --cap observe    --scope-selector 'all=host:B'   # 单格 observe（角色阶梯与单格授予可混用，5.2bis ①/3.4）
  ```
- **挂对象细则**（落 `POST /v1/resources/{code}/constraints`）：
  ```bash
  postern constraint add docker-A   --cap manage  --kind container_prefix --spec 'prefix=app-'
  postern constraint add docker-A   --cap observe --kind container_prefix --spec 'prefix=app-'
  postern constraint add db-main     --cap query   --kind table_allow  --spec 'tables=public.orders,public.customers,public.invoices'
  postern constraint add db-main     --cap query   --kind column_mask  --spec 'fields=public.customers.email,public.customers.phone'
  postern constraint add redis-main  --cap query   --kind key_prefix   --spec 'prefix=cache:'
  postern constraint add redis-main  --cap mutate  --kind key_prefix   --spec 'prefix=cache:'
  postern constraint add redis-main  --cap manage  --kind command_class --spec 'classes=@read,@write,@keyspace'
  postern constraint add svc-order   --cap mutate  --kind http_route   --spec 'routes=POST:/api/orders,POST:/api/orders/*/submit'
  postern constraint add svc-order   --cap mutate  --kind mask_fields  --spec 'fields=customer.email,customer.phone'
  # svc-billing / svc-crm 同构（从略）
  ```
- **展开校验与确认**：
  ```bash
  postern grants agent2          # 渲染 agent2 的 (资源×动词) 授权视图（your_grants 同源）
  postern role list --principal agent2   # 列出 agent2 的全部 binding（含 scope 展开后的资源集与 version）
  postern constraint list redis-main     # 列出 redis-main 上的全部细则
  postern verify                 # 红队自检：含"提权类操作→模型层不存在该授权"等项（6.7）
  ```
- **声明式整体导入**（与逐条等价，6.6）：`postern import policy.toml`（片段形态见示例一 §4），导入校验失败整体拒绝、无部分 apply。

### 3.2 管理前端（SPA，人用，控制面 API 的视图）

SPA 是控制面 API 的视图（D4、CONS-17：默认经桌面壳 webview over UDS，浏览器从不直接触达 `control.sock`）。本场景相关页面：

- **角色阶梯页**：以"动词集 + 继承链"可视化展示 observer⊂operator⊂maintainer；新建/编辑角色时勾选动词；`admin` 名称在前端即被禁用输入（前端便利，**真正的硬拒在 daemon**，公理三）。
- **Principal 授权页**（agent1/2/3 各一）：左侧列出该 Principal 的全部 binding（role + scope，scope 旁实时显示"展开为哪些资源代号"），右侧是**展开后的授权矩阵**（资源 × 动词网格，每格标 ✅/❌ 与所选 tier），与 `your_grants` 同源。运维在此添加/删除 binding、调整 scope 选择器，矩阵实时重算预览。
- **资源细则页**（每个资源一个）：展示该资源上各动词的 constraint（如 docker-A 的 `container_prefix=app-`），可增删；删除细则走显式 `delete_flag=1`（限制性表禁 enable_flag，5.2），并提示"删除细则=扩大动词作用面"。
- **授权审计页**：渲染 `policy_change` 事件流（谁、何时、改了哪条 binding/constraint、写后 version），供事后核对与对账（5.3、6.5 `GET /v1/audit?kind=policy_change`）。

> SPA 自身**零安全逻辑**：所有矩阵展开、tier 落点、合法性校验都在 daemon 完成，前端只渲染 daemon 回报的事实（8.12）。SPA 上的每个写动作都携带读取时拿到的 `version`，参与乐观锁（见 4.2-F）。

### 3.3 数据面（Agent 用，MCP/HTTP 外壳）

本场景**不直接由 Agent 操作**（权限分配是控制面动作，Agent 无控制面路径，5.5/8.10），但分配结果对 Agent 的可见面是：

- **`postern_grants`**（MCP 固定工具，6.8）：Agent 调用即得自身 `your_grants`（自己的授权世界投影），用于自我修正与分流。
- **`postern_surface`**（MCP 固定工具，6.8）：返回该 Principal Scope 内**已授权能力面**（授权快照的投影），**绝不触达 `Adapter::discover`**、不泄露 Scope 外资源是否存在（CONS-20）。
- Agent 据此能验证"我被分配了什么"，但**无任何路径修改自己的授权**（控制面与数据面物理隔离）。

CLI 的 `init` 向导可附带在客户端本地生成 `CLAUDE.md` 片段（把"该 Principal 经哪个 MCP 端点、有哪些已授权动词"渲染为可粘贴文本，07-postern-cli §3）——纯文本渲染便利，内容只含授权事实，不编造话术（公理六）。

## 4. 使用方式与预期结果

### 4.1 正常操作

以"为三个 Agent 完成差异化授权"为主线，每步给「操作 → 可验收的预期结果」。

**步骤 1 · 声明角色阶梯（继承）**
- 操作：`postern role add observer --cap observe --cap query`；`postern role add operator --inherits observer --cap mutate --cap execute`；`postern role add maintainer --inherits operator --cap manage`。
- 预期结果：三条 `roles` 行 + 对应 `role_capabilities` + 两条 `role_inherits` 写入权威库，各返回带 `version=0` 的实体。`postern role list` 显示 `operator` 的有效动词集为 {observe, query, mutate, execute}（含继承展开）、`maintainer` 为 {observe, query, mutate, execute, manage}。每条写入产生一条 `policy_change` 审计（actor=运维、policy_rev 递增、表名+行 id+写后 version）。`destroy`、`admin` 不在任何角色动词集内。

**步骤 2 · 给 agent1 绑定 observer × env:prod**
- 操作：`postern role bind agent1 --role observer --scope-selector 'all=env:prod'`。
- 预期结果：一条 `bindings`（principal=agent1, role=observer）+ 一条 `binding_scope`（kind=selector, spec=`{all:[{key:'env',value:'prod'}]}`）写入。快照重建时该选择器展开为全部带 `env:prod` 标签的资源（`svc-order`/`svc-billing`/`svc-crm`/`docker-A`/`db-main`/`redis-main`/`mq-main`）。`postern grants agent1` 返回每个资源仅 {observe} 或 {observe, query}（docker-A 无 query 语义→仅 observe），无任何写/管/毁格。policy_rev 递增。

**步骤 3 · 给 agent2 绑定三条 binding（不同辖区不同信任等级）**
- 操作：`postern role bind agent2 --role maintainer --scope-selector 'all=host:A,kind:docker'`；`... --role observer --scope-resource db-main`；`... --role maintainer --scope-resource redis-main`。
- 预期结果：三条 binding 写入。展开后 `postern grants agent2` 返回：
  - `docker-A`：{observe（→logs tier）, manage（→admin tier）}，无 destroy（不在 maintainer）。
  - `db-main`：{observe, query（→ro tier）}，无 mutate（observer 不含）。
  - `redis-main`：{observe, query, mutate, manage（→admin tier）}，无 destroy。
  - **未被任一 scope 选中的资源**（`svc-*`、`mq-main`）：完全不出现在 agent2 的授权视图中（默认拒绝，公理一）。policy_rev 递增，每条 binding 一条 `policy_change`。

**步骤 4 · 给 agent3 绑定三条 binding（operator 业务 + observe docker 日志 + observe B 全部）**
- 操作：`postern role bind agent3 --role operator --scope-selector 'all=env:prod,kind:business'`；`... --role observer --scope-resource docker-A`；`... --cap observe --scope-selector 'all=host:B'`（B 上意图是"看日志"=纯 observe，按示例一 §3.3 注用 `(资源 × observe)` 单格授予，而非 `observer` 角色——避免 `observer` 的 query 把"读数据"也带进来，体现"日志=observe"与"读数据=query"的动词正交，5.2bis ①/3.4）。
- 预期结果：`postern grants agent3` 返回：
  - `svc-order`/`svc-billing`/`svc-crm`：{observe, query, mutate}（operator 含 mutate；execute 在 http 资源无对应格，无害落空，5.2bis ①）。
  - `docker-A`：仅 {observe}（observer 角色落在 docker 的 observe 格），**无 manage/destroy**——agent3 能查容器日志但不能重启，与 agent2 精确区分。
  - `db-main`/`redis-main`/`mq-main`：仅 {observe}（单格 observe 授予，query 标 ❌）——agent3 在 B 上只看日志/状态，不读数据（与 agent2 在 db-main 的 {observe, query} 区分）。
- 验收点：同一资源 `docker-A` 对 agent2 给 manage、对 agent3 只给 observe——"管理"在两 Agent 处指向不同动词，互不串味（需求 1 满足）；agent3 在 B 上的 observe-only 与 agent2 在 db-main 的 query 精确正交（动词级差异化）。

**步骤 5 · 挂对象细则（收窄动词作用面）**
- 操作：执行 §3.1 的全部 `postern constraint add`（docker container_prefix、db table_allow/column_mask、redis key_prefix/command_class、svc http_route/mask_fields）。
- 预期结果：各 `grant_constraints` 行写入（`CHECK(enable_flag=1)`，限制性表，5.2）。验收：
  - agent2 的 `docker-A:manage` 只对 `app-` 前缀容器生效（求值步骤 [4] `container_prefix` 校验，非 `app-` 容器→deny）。
  - agent2 的 `db-main:query` 只命中白名单表且 PII 列脱敏（[4] `table_allow` + 输出 `column_mask`）；agent3 在 db-main 上无 query 格（仅 observe），该细则对 agent3 无作用面。
  - agent3 的 `svc-order:mutate` 只对 `POST /api/orders`、`POST /api/orders/*/submit` 路由生效（[4] `http_route`）。
  - 细则挂在资源级，对所有**在该格被授权**的 Principal 一致生效（如 db `column_mask` 同时作用于 agent1/agent2 的 db-main query；agent3 在 db-main 无 query 格则不涉及）。每条 `policy_change` 入审计。

**步骤 6 · 展开校验与确认**
- 操作：`postern grants agent1`/`agent2`/`agent3`；`postern role list --principal agent2`；`postern constraint list redis-main`；`postern verify`。
- 预期结果：三个授权视图与示例一 §3 的 (资源×动词) 矩阵逐格一致（✅ 标授予并注明 tier、❌ 标默认拒绝）。`role list` 显示每条 binding 的 scope 展开结果与当前 `version`（供后续修改回传）。`verify` 全项通过（含"提权类操作→模型层不存在该授权"项，6.7），无防线缺口。整个分配过程在审计中形成完整 `policy_change` 链，可逐行与权威库对账（5.3）。

**步骤 7 · 随时调整（热生效）**
- 操作（举例：把 agent3 在 svc-crm 上的 mutate 收回）：把 binding scope 从 `kind:business`（含三个 svc）改窄，或对 svc-crm 单独建更窄 binding；携带读取时的 `version` 提交。
- 预期结果：控制面事务 COMMIT → 同一写锁内重建快照（policy_rev 递增）→ 数据面下一次求值即按新策略（agent3 对 svc-crm 的 mutate 立刻 deny）。**无需重启 daemon、无需 Agent 配合、无需在飞请求等待**（6.2）。变更入 `policy_change` 审计。

### 4.2 异常场景与预期结果

每条写「触发 → 系统行为 → 预期结果」，一律 fail-closed；凡返回 Agent 的内容均匿名化（代号）、脱敏、只说事实。

**A · 试图授予 `admin`（模型层硬拒）**
- 触发：运维（或被诱导的脚本）执行 `postern role add admin --cap manage`，或 `postern role bind agent2 --role admin ...`，或 import 一份含 `name="admin"` 角色的 TOML。
- 系统行为：控制面在领域模型层即拒——`roles.name CHECK(lower(trim(name)) <> 'admin')` 拦截建角色（含 `Admin`、` admin `带空白的大小写/空白绕过，因入库前归一化 trim+大小写策略，5.2 入库归一化）；`Capability` 枚举根本无 Admin 变体，无法在类型上表达"授予 admin"（8.1 契约 `SEC_ADMIN_NOT_GRANTABLE`）；import 校验整体拒绝、无部分 apply（6.6）。
- 预期结果：返回统一 `{error:{code,message}}`（如 422），`message` 为常量安全文案（"admin 不可作为可授予角色"），**不写入任何行**。`admin` 只存在于控制面人类操作，永不是可分配角色（《技术设计》3.3、8.10）。无论 CLI 还是 import 入口，硬拒一致（公理七：入口对称）。该尝试入审计（`policy_change` 失败或控制面拒绝记录）。

**B · 标签选择器展开为空集（不授予、不报错）**
- 触发：运维绑定一条 `--scope-selector 'all=env:staging'`，但当前无任何资源带 `env:staging` 标签；或选择器语法可解析但匹配不到任何资源。
- 系统行为：快照构建时该选择器展开为**空集**——该 binding 不授予任何资源（5.2bis ②、示例一 §4 注）。这是 fail-closed 的"安全方向"：空集=授予面为空，绝不放行任何资源。
- 预期结果：binding 写入成功（语法合法），但 `postern grants <principal>` 中该 binding 贡献 0 个授权格；不报错、不放行（既不误授全部、也不抛异常）。SPA 的授权矩阵对该 binding 显示"展开为 0 个资源（无匹配标签）"的事实提示。若后续有资源被打上 `env:staging` 标签，下一次快照重建即自动纳入（标签选择的弹性）。

**C · 选择器语法不可解析（fail-closed 拒绝）**
- 触发：import 一份 selector spec 非法（如 `{all:[{key:'env'}]}` 缺 value，或非 JSON）的 TOML，或 API 直传非法 selector。
- 系统行为：标签语法无法解析 → 该绑定不授予任何资源（5.2bis ②，fail-closed）；import 校验失败则整体拒绝（6.6）。
- 预期结果：import 整体返回错误、无部分 apply；逐条 API 写入返回 422 并不落库。绝不"尽力解析"出一个意外宽的授权面（公理二：不确定即拒绝）。

**D · 同资源不同 Agent 不同动词（精确正交，非异常但需验收）**
- 触发：`docker-A` 同时被 agent2（maintainer→manage）与 agent3（observer→observe）绑定。
- 系统行为：RBAC 展开按 (Principal, Resource, Capability) 逐格独立判定（5.2bis、求值步骤 [3]）——agent2 的 `docker-A:manage` 格存在，agent3 的 `docker-A:manage` 格**不存在**。两者互不影响。
- 预期结果：agent2 重启 `app-order` 容器→放行（走 admin tier，限 `app-` 前缀）；agent3 重启同容器→deny（observer 不含 manage，求值 [3] 无格短路）。`postern grants agent2`/`agent3` 在 docker-A 上分别显示 {observe, manage} / {observe}。同一资源、同一物理目标，差异化授权精确正交（需求 1/3 满足）。

**E · 细则冲突合并（同 kind 取交集，fail-closed）**
- 触发：运维对 `db-main:query` 先后挂两条 `table_allow`：①`tables=public.orders,public.customers`，②`tables=public.customers,public.invoices`（同格、同 kind 多行）。
- 系统行为：同格（同 resource+capability）同 `kind` 多行**取交集**（全部须满足，fail-closed，5.2 约束合并语义）；不同 `kind` 之间恒 `AND`。
- 预期结果：`db-main:query` 的有效表白名单 = ①∩② = {`public.customers`}（交集，不是并集）。验收：query `public.orders` → deny（不在交集）；query `public.customers` → 允许且 PII 列经 column_mask 脱敏（不同 kind 恒 AND）。合并取交集是"宁可更窄"的 fail-closed 方向——绝不因多挂一条细则而意外扩权。若某 kind 确需并集语义，须在该 kind 的 spec 文档显式声明其扩权特性（默认不取并集）。

**F · 并发改策略的乐观锁冲突（409）**
- 触发：两个运维（或一个运维两个标签页）同时读到 agent2 的某 binding（`version=3`），各自修改后先后提交；或 SPA 与 CLI 并发改同一 constraint。
- 系统行为：更新/删除端点**必须携带期望 `version`**；第一个提交成功（version→4）；第二个携带过期的 `version=3` 提交 → 不匹配 → 返回 `409 Conflict` 并写 `policy_change` 审计（记录冲突，5.2 乐观锁端到端语义、6.5）。base 仓储不自读自比，期望 version 唯一来源是调用方先前读取值（5.1-②）。
- 预期结果：第二个写入**不生效**、不静默覆盖、不静默重试；CLI/SPA 原样呈现 409 并提示"重新读取最新 version 再改"（07-postern-cli §6.2）。两次修改不会互相吞掉——避免"后写者悄悄覆盖前写者的扩权/收权"。系统协调写（sweeper 幂等回收）不走乐观锁，与用户意图写区分（5.2）。

**G · 绑定/细则引用已逻辑删除的资源（悬挂引用不可见，fail-closed）**
- 触发：`svc-crm` 被逻辑删除（场景 2 下线），但 agent3 仍有一条 scope 含 svc-crm 的 binding，或 svc-crm 上仍有 http_route 细则。
- 系统行为：父行（resources）逻辑删除时，同一事务内级联把直接子行（credential_tiers/binding_scope/grant_constraints/...）置 `delete_flag=1`（5.2 级联逻辑删除规则）；即便级联遗漏，快照构建按引用链可见性过滤——父行不可见 ⇒ 该行不可见（5.2 快照构建兜底，fail-closed）。
- 预期结果：`postern grants agent3` 中 svc-crm **完全不出现**（悬挂引用不进快照、不放行）；不暴露 svc-crm 曾存在。不提供 undelete（逻辑删除是终态）；若需恢复，新增同名资源（partial unique 允许重建）后重新绑定。绝不出现"删了资源但旧授权悬空生效"的 fail-open。

**H · 给不存在的 Principal 或动词非法（配置缺失，fail-closed）**
- 触发：`postern role bind agent9 --role observer ...`（agent9 未注册），或 `--cap frobnicate`（非 6 动词之一），或 `--role superuser`（角色未声明）。
- 系统行为：控制面校验外键与枚举——principal/role 不存在→拒绝写入；capability 非 {observe,query,mutate,execute,manage,destroy} 之一→`role_capabilities` 的 6 动词 `CHECK` 拒绝（5.2）；CLI 端 `--cap` 还做本地字面量语法校验（07-postern-cli §6.1，仅语法、不做授权判断）。
- 预期结果：返回 422/明确错误、不落库。配置缺失绝不被静默忽略或"尽力创建"（公理二）。daemon 启动时若 settings 出现未知 key 亦 fail-closed 拒绝加载（5.2 settings 语义）——配置面整体不容忍未知/缺失。

**I · 试图经数据面修改自己的授权（无路径）**
- 触发：agent1（已被诱导）尝试经 MCP/HTTP 数据面调用某"提权"工具，或直连 `control.sock`。
- 系统行为：数据面外壳只暴露固定动词工具（`postern_query`/`postern_mutate`/.../`postern_grants`/`postern_surface`，6.8），**无任何修改授权的工具**；`control.sock=0600` 仅属主可连，Agent 运行于独立 uid 物理不可达（5.5 硬前置条件②、8.10）；即便同 uid 也叠加控制面认证（5.5）。`postern verify` 含"从模拟 Agent uid 连 control.sock 必须失败"红队项。
- 预期结果：Agent 经数据面**无任何路径**触达授权变更；`postern_surface` 只返回已授权能力面投影、不泄露 Scope 外资源存在性（CONS-20、6.7 verify 项 4）。提权只能经控制面由人发起（场景 5 申请提权 / 场景 6 临时授权）。控制面与数据面物理隔离是结构性保证，非"建议"（公理三）。

**J · daemon 重启后授权恢复（持久权威）**
- 触发：完成全部分配后 daemon 重启（或崩溃恢复）。
- 系统行为：策略状态持久于 `policy.db`（权威库）；启动序列解锁保险箱、加载权威库、重建 `PolicySnapshot`（6.2、8.11）。瞬时状态（temp_grants 及 TTL）不在本场景（见场景 6）；本场景的 binding/role/constraint 全是持久策略，原样恢复。
- 预期结果：重启后 `postern grants agent1/2/3` 与重启前逐格一致，policy_rev 连续可对账。授权分配是配置、是持久权威，不因重启丢失（《技术设计》"预设决定一切"）。审批挂起类才在重启时 fail-closed（不在本场景，6.10）。
