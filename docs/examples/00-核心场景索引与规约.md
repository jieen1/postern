# postern 核心场景规格 · 索引与规约

> 本目录用一组**场景规格**把系统的核心功能"钉死预期"——每个场景明确：用户需求、预期实现路径、页面/操作、以及正常与异常操作的预期结果。这些规格既是功能定义，也充当实现的**验收基准**（正常操作预期 + 异常操作预期）。
>
> 与上层文档冲突时，以《技术设计文档》七公理与《详细设计文档》第八部分领域裁决为准。本目录纯设计，不含实现代码、阶段划分或进度状态。

## 每个场景文件的统一结构

每个场景文件**必须**含以下四节，标题与顺序一致：

```
# 场景 N · <标题>

> 一句话场景概述。

## 1. 用户需求
用第一人称/具体口吻写清用户想达成什么（像真实诉求那样具体，可含具体资源/Agent/动作）。
点明该场景要解决的痛点与它覆盖的系统核心功能。

## 2. 预期实现路径
postern 如何满足该需求：涉及的领域/机制/流程，按步骤展开，**引用设计文档具体章节**
（如「求值步骤[1]~[10]」「详细设计 6.13 会话来源」「5.2bis 授权模型」）。
说明哪些是控制面动作、哪些是数据面动作、哪些热生效。

## 3. 页面与操作
用户**实际怎么操作**，三类入口都要覆盖到（与该场景相关的）：
- **CLI**：`postern ...` 的具体命令与参数（人/脚本用）。
- **管理前端（SPA）**：对应的页面/视图与其上的操作（人用；SPA 是控制面 API 的视图，按功能描述页面）。
- **数据面（Agent 用）**：MCP 工具（如 `postern_query`/`postern_mutate`/`postern_surface`）或 HTTP 外壳调用。
（不涉及某入口的场景可注明「不适用」。）

## 4. 使用方式与预期结果
### 4.1 正常操作
逐步骤写「操作 → 预期结果」，每步预期结果要**可验收**（具体到返回什么、状态如何变、是否进审计）。
### 4.2 异常场景与预期结果
列出该场景的关键异常/边界/攻击，每条写「触发条件 → 系统行为 → 预期结果」，
**一律体现 fail-closed**（不确定即拒绝、凭据零接触、只说事实）。异常要覆盖：
认证失败/凭证过期吊销、越权、伪装攻击、连接不可建、会话过期、配置缺失、并发冲突、daemon 重启恢复 等与本场景相关者。
```

## 约定

- 中文；命令/标识符/接口名英文。
- 预期结果必须**可验收、可观察**（避免「应该没问题」这类含糊表述）。
- 异常预期一律 fail-closed；凡涉及返回 Agent 的内容，须体现匿名化（代号）与脱敏。
- 资源代号、Agent 名、角色名尽量沿用 01 示例（`svc-*`/`db-*`/`docker-*`/`agent1..3`/`observer`/`operator`/`maintainer`），跨场景一致。
- 敏感项（账号、密码、token、真实地址）一律以 `vault://` 引用或代号出现，不写明文。
