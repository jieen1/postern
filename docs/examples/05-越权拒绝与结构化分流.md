# 场景 5 · 越权拒绝与结构化分流

> Agent 越权时，网关不是甩一句"forbidden"了事，而是回一份**可推理的结构化拒绝**——告诉它被拒的事实、它自己还能做什么、以及（确需扩权时）申请权限的现成命令。Agent 据此**自我修正 / 走替代路径 / 向人申请提权**三级分流，人最少介入、无同步审批阻塞。

> 本场景资源代号、Agent、角色沿用 [01-多资源多Agent授权示例](01-多资源多Agent授权示例.md)（`docker-A`/`db-main`/`svc-*`/`agent1..3`/`observer`/`operator`/`maintainer`）。一切真实地址、账号、令牌均以 `vault://` 引用或代号出现，绝不写明文（公理四、技术设计 11.2 不变量）。冲突时以《技术设计文档》七公理与《详细设计文档》第八部分为准。

## 1. 用户需求

> "我的 agent3 被授权'查 docker-A 的业务容器日志'（observer），但它在排障时常常想顺手**重启容器**——那是 maintainer 才有的 `manage`，它没有。我不想让它一撞到墙就卡死，也不想每次都跳出来等我点'批准'。我希望：
> - Agent 撞墙时，网关**讲清楚为什么**、**它自己还能干什么**，让它自己绕过去；
> - 它**确实**需要重启权限时，把'要做什么、为什么、要什么权限'连同一条**现成的提升命令**摆到我面前，我**方便时**粘一下、带个 TTL，自动到点回收；
> - 网关回给 Agent 的东西**只能是事实或我亲笔写的注记**，绝不替我编话术，也绝不借机让 Agent 摸清我系统里还有哪些资源、别的 Agent 有什么权限。"

**痛点**：传统"同步审批流"要求人实时在场守候——Agent 操作越多，人被打扰越频繁；而粗暴的 `403` 又让 Agent 盲目重试或彻底卡死。

**覆盖的系统核心功能**：

- **结构化拒绝响应**（技术设计 6.4 协议 / 详细设计 4.1 `DenyResponse` 结构体）：`decision` / `denied` / `reason` / `your_grants` / `request_hint` / `operator_note`，只含**策略事实**或**人亲笔预写**（公理六）。
- **三级自动分流**（技术设计 6.3）：①误用自我修正 ②低权限替代路径绕行 ③申请提权（Agent 向人陈述 + 贴 `request_hint`，人异步一条命令授权）。
- **拒绝进审计**（详细设计 5.3，被拒是最有分析价值的信号，附完整 `stage`+`reason`）。
- **拒绝只述自身授权世界 / 防拓扑探测**（公理四、技术设计 6.4 安全边界、详细设计 6.7 verify 项 4）。
- **escalate 默认折叠为 deny**（审批默认关闭，`Decision::Escalate{fallback=deny}`，详细设计 6.10 / `decision="escalate_denied"`）。

## 2. 预期实现路径

越权请求与正常请求**走同一条求值管线**（公理七），只是在某一步判定拒绝即短路，进入"拒绝响应组装"。

### 2.1 越权如何被判定并短路（求值管线 [0]~[10]，详细设计 6.1）

以 agent3 重启容器为例（`docker-A` 上只授了 `observe`/b3-2）：

| 步骤 | 动作 | 本场景判定 |
|---|---|---|
| [0] 外壳归一化 | MCP/HTTP 外壳 → `NormalizedRequest{presented=agent3, resource="docker-A", intent=manage 负载}` | 自此与外壳无关（公理七） |
| [1] 认证+可信域 | `Authenticator` 校验 agent3 凭证有效 | → `PrincipalId(agent3)` |
| [2] 语义归一化 | `docker` Adapter.classify：`restart` → `Capability=Manage`，objects=[container:app-order] | 归类成功 |
| [3] **RBAC** | 快照查表：agent3 在 `docker-A` 是 `observer`（observe+query），`(docker-A, manage)` 格**不存在** | → **deny**，`stage=rbac`（公理一），**短路** |
| [4]~[8] | 不执行 | manage 是有副作用动词，但在 deny 短路下**确未执行**，无 intent 事件副作用（详细设计 6.1 [7a]） |
| [9] 脱敏 | `DenyResponse` 经**同一 Sanitizer** 出口（拒绝响应也脱敏，详细设计 6.4 作用面） | `denied.objects` 经匿名化与脱敏 |
| [10] 审计 | `AuditSink.record`：`kind=request, decision=deny, stage=rbac, reason="role=observer 不含 docker-A:manage"` | 进审计（详细设计 5.3） |

越权可能在不同 stage 短路（均 deny，公理二）：认证失败 `stage=auth`；归类失败 `stage=classify`；**无格 `stage=rbac`**；细则不过 `stage=constraint`；条件不满足 `stage=condition`；无承载该动词的 tier `NO_TIER_MATCH_DENIED`。本场景主线落在 `stage=rbac`。

### 2.2 拒绝响应如何组装（技术设计 6.4 协议；详细设计 4.1 结构体 / 8.3 策略引擎事实组装，公理六）

`Evaluator` 产出 `Decision::Deny(DenyResponse)`，由 `EvalTrace` 提供"在哪步、因何拒"；`DenyResponse` 各字段全部取自单一权威快照（详细设计 8.3 策略引擎"DenyResponse 的事实组装：reason/your_grants/request_hint/operator_note 全部取自快照"）。字段逐项来源：

- `reason`：引用**策略事实**（如 `role=observer 不含 docker-A:manage`），非话术。
- `your_grants`：从**该 Principal 自身**授权快照投影（与控制面 `GET /v1/grants/{principal}` 同源，详细设计 6.5）——agent3 在它**有授权的资源**上还能做什么。
- `request_hint`：由策略**机械生成**的 `postern elevate ...` 命令（不是网关编造的建议）。
- `operator_note`：**仅当**资源所有者为该 `(资源, 动词)` 在 `deny_notes` 里**亲笔预写**才出现，原样转述；否则字段不出现（`skip_serializing_if = "Option::is_none"`，详细设计 4.1）。
- `denied`：已匿名化、已脱敏的 `DeniedFacts`（出口脱敏走同一 Sanitizer，详细设计 6.4 作用面）。

### 2.3 三级分流如何发生（技术设计 6.3）

拒绝不是终点，是网关与 Agent 的一次结构化对话：

1. **①误用自我修正**：Agent 读 `reason` 理解"我本欲查询却发了修改/重启" → 改用自己有的 `observe`/`query` 重试。**人无感知。**
2. **②低权限替代路径**：Agent 依 `your_grants` 推理出可行替代（如不重启而读日志诊断） → 完成任务。**人无感知。**
3. **③申请提权**：确需更高权限 → Agent 向用户**陈述**（要做什么、为什么、需何权限）并附 `request_hint` → 人异步看到，**一条命令授权（默认带 TTL）**。**人粘贴一次。**

### 2.4 提权的热生效与回收（控制面动作）

第③级里人执行 `postern elevate agent3 --cap docker-A:manage --ttl 30m`：

- 控制面 `POST /v1/grants/temp` 写 `temp_grants` → **同一写锁内重建 `PolicySnapshot`（Arc 原子替换）热生效**（详细设计 6.2、6.5），无需重启 daemon、无需 Agent 重连。
- 求值时"当前生效授权" = `bindings×role_capabilities` ∪ `temp_grants(expires_at >= now 且 ended_at 为空)`（详细设计 5.2）——agent3 重试时新增的 `(docker-A, manage)` 格已生效。
- **TTL 到点自动回收**：`expires_at < now` 在**求值时刻按墙钟即刻失效**（不依赖后台时序，详细设计 6.2）；`sweeper` 事务性把过期记录写 `ended_at`/`end_reason=expired` 并写审计（口子自动愈合的可见记录，详细设计 5.2 终态字段、6.1）。

### 2.5 审批关闭时 escalate 即 deny（详细设计 6.10）

某些 `(资源, 动词)` 可在 `role_capabilities.action` 标 `escalate`。审批开关默认关闭（`settings: approval.enabled=false`，详细设计 5.2 已知 key）：

- 关闭时 `escalate ≡ deny`——`Decision::Escalate{fallback}` 即取 `fallback`（恒 deny），**立即 deny**，不进挂起队列、不阻塞等待人。
- 审计 `decision=escalate_denied`（与普通 deny 可区分，详细设计 5.3）。
- 这正是"安全不依赖实时审批"：人不在场，escalate 也 fail-closed，绝不悬挂等待（公理二）。

**控制面动作 / 数据面动作划分**：越权请求的判定、拒绝组装、审计在**数据面内核**（每请求，热路径）；`postern elevate` / `postern deny-note set` 是**控制面动作**（写策略，热生效）；TTL 回收由 `sweeper` 在控制面侧事务回收。

## 3. 页面与操作

### 3.1 CLI（`postern ...`，人/脚本用）

| 命令 | 端点（详细设计 6.5） | 在本场景的用途 |
|---|---|---|
| `postern elevate agent3 --cap docker-A:manage --ttl 30m` | `POST /v1/grants/temp` | 第③级：人收到 Agent 申请后**粘贴一次**，给 agent3 临时 `manage`，带 30 分钟 TTL |
| `postern revoke-grant <id>` | `DELETE /v1/grants/temp/{id}` | 提前撤回（不等 TTL 到点） |
| `postern grants agent3` | `GET /v1/grants/{principal}` | 查看 agent3 当前授权（与 `your_grants` 同源），确认提权已生效/到点已回收 |
| `postern deny-note set docker-A manage --note "..."` | `POST /v1/resources/{code}/deny-notes` | 资源所有者**亲笔预写**拒绝注记，令越权时返回 `operator_note`（公理六） |
| `postern denials --window 7d` | `GET /v1/denials/summary?window=7d` | 拒绝聚合分析：看"agent3 反复请求 docker-A:manage"的模式，据此决定是否固化为常驻角色（详细设计 6.5 拒绝分析闭环） |
| `postern audit --principal agent3 --kind request` | `GET /v1/audit?...` | 查 agent3 的 deny 记录（含 `stage`+`reason`） |
| `postern settings get approval.enabled` | `GET /v1/settings/{key}` | 确认审批开关状态（默认 `false`，escalate 即 deny） |

### 3.2 管理前端（SPA，人用；控制面 API 的视图）

- **拒绝分析页**：`denials/summary` 的可视化——按 `(Principal × 资源 × stage)` 聚合的越权热点榜，一眼看出"谁反复撞哪堵墙"；每条可下钻到具体 deny 审计记录（`reason`/`stage`/`policy_rev`）。这是"预设进化闭环"的人用入口（详细设计 6.5）。
- **临时授权页**：列 `temp_grants`（principal/cap/`expires_at`/剩余 TTL/`end_reason`），可发起 `elevate`、提前 `revoke-grant`；提交即热生效。读端点返回 `version`，更新/删除回传期望 `version`（乐观锁，详细设计 6.5）。
- **拒绝注记页**：按 `(资源, 动词)` 维护 `deny_notes`，所见即越权时 Agent 会收到的 `operator_note` 原文。
- **审批挂起队列页**：当 `approval.enabled=true` 时展示挂起 escalate 项并裁决（`approve`/`deny`）；**本场景默认关闭，此页通常为空**（escalate 已在数据面即时折叠为 deny）。
- **审计查询页**：按 principal/decision 过滤 deny 记录。

### 3.3 数据面（Agent 用：MCP 工具 / HTTP 外壳）

- **触发越权的工具**：固定动词工具（详细设计 6.8），如 `postern_manage(resource="docker-A", request={action:"restart", container:"app-order"})`——越权时**拒绝响应原样作为工具结果返回**，Agent 据此分流。
- **自我修正/替代用的工具**：`postern_observe`（agent3 在 docker-A 上有 `observe`，可读 `app-` 前缀容器日志诊断）、`postern_query`（agent3 对 `svc-*` 有 `query`，可读业务接口；docker 资源无 `query` 语义，故此工具不落在 docker-A 上）。
- **`postern_grants`**：返回该 Principal 的 `your_grants` 事实（与拒绝响应里的 `your_grants` 同源）——Agent 可主动查"我能做什么"再行动。
- **`postern_surface`**：该 Principal **Scope 内已授权能力面**（授权快照投影），**不触达 `Adapter::discover`、不暴露未授权资源**（详细设计 6.8、CONS-20）——Agent 探索替代路径时只看见自己的授权世界。
- 控制面 elevate / deny-note **不在数据面入口**：Agent 永远不能自己提权或改注记（公理三，控制面 `control.sock=0600`，Agent 不可连）。

## 4. 使用方式与预期结果

### 4.1 正常操作

#### 步骤 0 · 前置：所有者亲笔预写拒绝注记（一次性，可选）

- **操作**：资源所有者执行
  `postern deny-note set docker-A manage --note "重启业务容器属维护操作；确需重启找 @owner，会按窗口临时授 manage。只读排障用 observe 看日志即可。"`
- **预期结果**：`deny_notes(resource=docker-A, capability=manage)` 写入并热生效（同写锁内重建快照）；写 `policy_change` 审计。此后**任何**对 `docker-A:manage` 的越权拒绝都会附带该 `operator_note` 原文（公理六——这是**人写的**，不是网关编的）。

#### 步骤 1 · agent3 越权重启容器 → 收到结构化 deny

- **操作**：agent3 经 MCP 外壳调
  `postern_manage(resource="docker-A", request={action:"restart", container:"app-order"})`
- **预期结果**：
  - 求值在 **[3] RBAC** 短路（`(docker-A, manage)` 格不存在），**确未执行**任何重启；
  - 工具结果是结构化拒绝响应（已脱敏）：

    ```jsonc
    {
      "decision": "deny",
      "denied": { "resource": "docker-A", "capability": "manage", "objects": ["container:app-order"] },
      "reason": "role=observer 不含 docker-A:manage",         // 策略事实，非话术
      "your_grants": {                                         // agent3 自身授权世界投影
        "docker-A": ["observe"],                              // 能看业务容器日志，不能 manage
        "svc-order":  ["observe","query","mutate"],
        "svc-billing":["observe","query","mutate"],
        "svc-crm":    ["observe","query","mutate"],
        "db-main":   ["observe"],
        "redis-main":["observe"],
        "mq-main":   ["observe"]
      },
      "request_hint": "postern elevate agent3 --cap docker-A:manage --ttl 30m",  // 策略机械生成
      "operator_note": "重启业务容器属维护操作；确需重启找 @owner，会按窗口临时授 manage。只读排障用 observe 看日志即可。"  // 步骤0 预写则出现
    }
    ```

  - 进审计：`kind=request, decision=deny, stage=rbac, reason="role=observer 不含 docker-A:manage", tier=null, response_digest=null`，并带 `policy_rev`/`intent_digest`（可聚合，详细设计 5.3）。

#### 步骤 2 · 第①级：误用自我修正【人无感知】

- **操作**：agent3 读 `reason`，意识到本意是"看容器为何异常"而非重启；改调
  `postern_observe(resource="docker-A", request={action:"logs", container:"app-order"})`
- **预期结果**：该请求 RBAC 命中 `(docker-A, observe)` 格（b3-2，限 `app-` 前缀容器）→ 经 `logs` tier 读只读日志 API → 脱敏后返回容器日志（真实地址/凭据被 ScrubSet 擦除）；`decision=allow` 进审计。**全程无人介入**。

#### 步骤 3 · 第②级：低权限替代路径绕行【人无感知】

- **操作**：另一情形——agent3 本想重启 `app-order` 以"清掉卡死连接"，读 `your_grants` 后发现自己对 `redis-main` 有 `observe`、对 `svc-order` 有 `observe/query/mutate`；它推理出替代：不重启容器，而是经业务接口走幂等重试 / 读 redis 状态确认积压是否自行消解。改调
  `postern_query(resource="svc-order", request={...})` 等它**有权**的动作。
- **预期结果**：替代动作在各自资源上 RBAC 命中并放行（agent3 对 `svc-order` 是 operator，对 `redis-main` 是 observer），任务以更低权限完成；**无人介入**。要点：替代方法由 **Agent 依事实推理**得出，网关只给 `your_grants` 事实、不编"你可以试试 redis"这类建议（公理六）。

#### 步骤 4 · 第③级：确需提权 → Agent 向人陈述 + 贴命令【人粘贴一次】

- **操作**：确认非重启不可（如容器进程级死锁，日志和业务接口都无法解）。agent3 在对话中向用户**陈述**：要做什么（重启 `app-order` 容器）、为什么（进程死锁，已试 observe 排障与业务接口绕行均无效）、需何权限（`docker-A:manage`），并**原样附上** `request_hint`：
  `postern elevate agent3 --cap docker-A:manage --ttl 30m`
- **预期结果**：Agent 给出的申请文本里，权限事实与命令来自拒绝响应（机械生成），陈述理由由 Agent 自己组织——**网关不参与撰写**。人**异步**看到（无需实时在场）。

#### 步骤 5 · 人执行提升命令 → 热生效

- **操作**：人方便时**粘贴一次**那条命令（或在 SPA 临时授权页点一下）：
  `postern elevate agent3 --cap docker-A:manage --ttl 30m`
- **预期结果**：
  - 控制面 `POST /v1/grants/temp` 写 `temp_grants(principal=agent3, resource=docker-A, capability=manage, expires_at=now+30m)`；
  - **同一写锁内重建快照、Arc 原子替换热生效**（详细设计 6.2）——无需重启 daemon、agent3 无需重连；
  - 写 `policy_change` 审计（actor=操作者、`policy_rev` 递增、行 id + 写后 version）；
  - 返回 temp_grant 的 `id`（雪花 id 字符串）供日后 `revoke-grant`。

#### 步骤 6 · agent3 重试 → 通过

- **操作**：agent3 重新调
  `postern_manage(resource="docker-A", request={action:"restart", container:"app-order"})`
- **预期结果**：
  - [3] RBAC 此次命中——当前生效授权含 `temp_grants` 的 `(docker-A, manage)` 格；
  - [4] 细则 `container_prefix=app-` 通过（`app-order` 命中前缀）；
  - manage 是有副作用动词 → **两阶段审计**：先落 `intent` 事件（写不进则执行前 deny，确未执行），经 `admin` tier 建连执行重启，后落 `outcome` 事件（与 intent 同请求 id 关联）；
  - 脱敏后返回成功结果（无真实地址/socket 路径回显）。

#### 步骤 7 · TTL 到点自动回收

- **操作**：无需人操作，等待 30 分钟。
- **预期结果**：
  - `expires_at < now` 后，求值时刻墙钟二次校验即令该格**即刻不可见**——agent3 再次 `restart` 立刻回到步骤 1 的 `stage=rbac` deny（即便此间无任何策略写入、快照未重建，也即刻失效，详细设计 6.2）；
  - `sweeper` 事务性回收：写 `temp_grants.ended_at` + `end_reason=expired`，写 `policy_change` 审计（与人工 `revoked` 语义可区分）；
  - `postern grants agent3` 显示 `docker-A` 又只剩 `observe`——口子自动愈合，且回收留痕可见。

### 4.2 异常场景与预期结果

> 一律 fail-closed：不确定即拒绝、凭据零接触、只说事实、匿名化代号、响应脱敏。

#### A · 拒绝响应不泄露 Scope 外资源是否存在（防拓扑探测，verify 项 4）

- **触发**：agent3 请求一个它 **Scope 外**（完全无 binding）的资源代号——无论该代号在系统里**真实存在但未授权给 agent3**（如 agent2 才有 binding 的某资源），还是**根本不存在**的臆造代号 `db-secret-xyz`。（注意：对 agent3 **有 binding 但动词不足**的资源——如 `mq-main` 它有 observe 却请求 manage——属步骤 1 主线的常规 `stage=rbac` deny，此时 `mq-main:["observe"]` 会如实出现在 `your_grants` 中，那是 agent3 自身授权世界的事实，不算泄露。）
- **系统行为**：两类 Scope 外请求都在 RBAC 短路为 deny；`your_grants` **只投影 agent3 自身有授权的资源**，绝不列出任何 Scope 外资源（无论其真假）；`reason` 不暗示"该资源存在但你无权"与"该资源不存在"之别。
- **预期结果**：**对不存在资源的请求与对无权限资源的请求，其拒绝响应不可区分**——`decision=deny`、`your_grants` 只含自身世界、`reason` 不泄露存在性。拒绝响应不能成为拓扑/权限探测工具（公理四、技术设计 6.4 安全边界、详细设计 6.7 verify 项 4，契约 `DENY_RESPONSE_SCOPE_BOUNDED`）。审计侧网关可记真实事实（origin、resource_id），但那是**网关侧可观测面**，不返回 Agent（详细设计 5.3）。

#### B · `denied.object` 经脱敏，不泄露真实地址/凭据

- **触发**：agent3 越权请求里嵌入了诱导回显的对象名（如把真实主机串塞进 `container` 字段），或对象本身关联真实地址。
- **系统行为**：`DenyResponse` 与正常响应**走同一 Sanitizer 出口**（详细设计 6.4 作用面含拒绝响应的 `denied.objects`）；系统级 ScrubSet 擦除真实地址/IP/凭据串。
- **预期结果**：返回的 `denied.objects` 只含**代号化、已脱敏**的对象引用（如 `container:app-order`），任何真实地址/凭据/socket 路径被擦除（公理四）。即便对象名被构造来诱导泄露，输出端也不回显真实拓扑标识。

#### C · 审批关闭下 escalate 立即 deny（不挂起、不等人）

- **触发**：某 `(资源, 动词)` 在 `role_capabilities.action` 标 `escalate`，agent3 触发它；当前 `approval.enabled=false`（默认）。
- **系统行为**：`Decision::Escalate{fallback}` 即取 `fallback`（恒 deny），**立即 deny**，不进挂起队列、不启动超时定时器、不阻塞等待人裁决（详细设计 6.10）。
- **预期结果**：Agent 同步收到结构化拒绝（与普通 deny 同形，可正常分流）；审计 `decision=escalate_denied`，与普通 `deny` 可区分（详细设计 5.3）。"安全不依赖实时审批"——人不在场也不会有请求悬挂或被 fail-open（公理二）。`approval.on_timeout` 固定 `deny`，导入/设置层禁止配为 allow。

#### D · `operator_note` 仅当所有者预写才出现，网关不编造

- **触发**：越权命中的 `(资源, 动词)` **没有**对应 `deny_notes` 记录（如所有者从未为 `db-main:mutate` 写注记）。
- **系统行为**：`DenyResponse.operator_note` 为 `None`，序列化时**整字段不出现**（`skip_serializing_if = "Option::is_none"`，详细设计 4.1）。网关绝不"补一句友好提示"。
- **预期结果**：拒绝响应里**没有** `operator_note` 字段；`reason`/`your_grants`/`request_hint` 仍是机械生成的事实。**所有面向 Agent 的非事实文本只能来自人亲笔预写**（公理六）——无预写则无该文本，网关不代为生成话术。

#### E · `request_hint` 由策略机械生成，不编造

- **触发**：对某越权请求生成提权提示。
- **系统行为**：`request_hint` 从**策略状态机械生成**目标 `(principal, res:verb)` 对应的标准 `postern elevate` 命令（技术设计 6.4，详细设计 4.1 字段语义）；若该动词在模型层根本**不可授予**（如 `admin` 性质能力——`Capability` 无 `Admin` 变体、`roles.name` 硬约束禁 `admin`），则**不生成**一条无意义/不可执行的提权命令。
- **预期结果**：`request_hint` 要么是一条**真实可执行**的标准提权命令，要么（对不可授予能力）**为 `null`/不出现**——绝不编造一条无法落地的命令误导 Agent（公理六、详细设计 6.7 verify 项 7：提权类操作模型层不存在该授权）。

#### F · Agent 反复越权刷审计（DoS 防护）

- **触发**：agent3 在自我修正/分流失败后**高频重试**同一越权请求，意图刷满审计存储。
- **系统行为**：deny 类审计按 `(Principal, 类型)` 窗口**聚合计数**（写一条带 `count` 的聚合记录，详细设计 5.3）；审计存储有独立配额与水位监控，逼近上限触发可感知降级（告警/强制轮转/降采样）而非让数据面瘫痪。
- **预期结果**：反复越权**不会**把审计打满成全局 DoS；每次请求仍 fail-closed deny（功能不退化），越权热点经 `denials/summary` 可见供人决策（详细设计 6.9 告警、6.5 拒绝分析）。

#### G · 凭证过期/吊销时的越权请求（认证层先于 RBAC）

- **触发**：agent3 凭证已过期或被吊销，又发来越权（或任何）请求。
- **系统行为**：[1] 认证阶段 `credentials.expires_at`/`revoked_at` 按求值时刻墙钟二次校验即失效（契约 `EXPIRED_OR_REVOKED_CREDENTIAL_DENIED`）→ `stage=auth` deny，**根本不进 [3] RBAC**（详细设计 6.2 墙钟二次校验、5.2 credentials schema；技术设计第七部分 身份与凭证）。
- **预期结果**：拒绝 `stage=auth`，`your_grants` 不返回（身份未确立，无授权世界可投影）；不泄露"若你凭证有效本可做什么"。**凭据零接触**——拒绝响应不含任何凭证细节（公理四）。

#### H · 提权热生效边界：已在飞的越权请求不被"追认"

- **触发**：人执行 `elevate` 的**同时**，agent3 上一条越权 `manage` 请求已在 [3] 短路为 deny 并返回。
- **系统行为**：快照 Arc 原子替换是**事件点**生效；已返回 deny 的请求是**确未执行**的终态，不会因随后提权而被"追认成功"。Agent 须**重试**才走到新快照。
- **预期结果**：提权只对**重试**（步骤 6）生效；旧的 deny 不被翻转成 allow，无"无审计痕迹的已执行操作"（详细设计 6.1 时序不变量、6.2 在飞语义）。决策确定性成立：相同请求 + 相同快照 → 相同决策。

#### I · daemon 重启后的恢复（TTL 与挂起项）

- **触发**：人执行 `elevate`（带 TTL）后，daemon 重启；或审批开启时有挂起 escalate 项，daemon 重启。
- **系统行为**：`temp_grants` 及剩余 TTL 是状态存储的一部分，daemon 重启后**恢复并继续计时**（求值时刻墙钟校验，详细设计 6.2；技术设计 6.2 临时授权 TTL）；而**挂起的 escalate 项跨重启恒 deny（fail-closed）**，不"复活"一个待批的危险操作（详细设计 6.10）。
- **预期结果**：重启不会丢失合法的临时授权（口子按 TTL 正常愈合），也不会让一个未决的危险提权悄悄越过重启被放行——两类瞬时态都按 fail-closed 方向恢复。

#### J · 配置缺失/快照不可得时的越权请求

- **触发**：极端情形——快照构建遇引用链悬挂、settings 出现未知 key、或快照不可得。
- **系统行为**：未知 settings key → daemon **拒绝加载/启动**（fail-closed，详细设计 5.2 已知 key 注册表）；引用链上父行不可见 ⇒ 子授权不可见（详细设计 5.2 快照构建兜底）；快照不可得 → boot fail-closed 拒绝启动（模块 06-postern-daemon boot 启动序列）。
- **预期结果**：任何不确定状态下，授权**只会更少不会更多**——配置异常绝不被解析为放行，越权请求在最坏假设下仍 deny（公理二）。

---

> **覆盖小结**：本场景钉死了"结构化拒绝 → 三级分流 → 申请提权（带 TTL，热生效，自动回收）"的完整闭环，并以 4.2 的 A~J 把"拒绝响应只述自身授权世界、防拓扑探测、`denied`/`operator_note`/`request_hint` 的公理六边界、escalate 默认 deny、凭证态先于 RBAC、热生效不追认在飞、重启 fail-closed 恢复"逐条钉成可验收预期。验收基准与技术设计 6.3（三级分流）/6.4（结构化拒绝协议与安全边界）、详细设计 4.1（`DenyResponse` 结构体）/8.3（事实组装）/6.4（Sanitizer 出口）/6.10（审批挂起与 escalate 折叠）/5.3（审计）/6.1（求值时序）/6.2（热生效与墙钟二次校验）及 6.7 verify 项 4/7/9 直接对应。
