# 场景 2 · 资源接入与认证配置

> 运维把服务器 A（SSH）、服务器 B（SSM）、数据库 `db-main` 与业务系统 `svc-*` 纳入网关：声明传输/适配器/凭据等级，配好认证（数据库静态账号 + 业务系统账号密码/会话登录，含可能的 2FA），生成只读账号与凭据等级，把真实地址对 Agent 匿名化，接入后跑红队自检确认严格。本场景是后续所有授权与数据面场景的地基——**资源、tier、认证、匿名化在此一次性钉死**。

本场景资源代号、机器、角色沿用 [01 多资源多 Agent 授权示例](01-多资源多Agent授权示例.md)：服务器 A 经 `ssh` 到达（跑 `svc-order`/`svc-billing`/`svc-crm` 业务服务 + `docker-A` 运行时），服务器 B 经 `ssm` 到达（跑数据库 `db-main` 与缓存 `redis-main`）。数据库资源代号统一沿用示例 §1 的 `db-main`，与场景 03~07 一致。下游授权由场景 03 承接，数据面运行由场景 04 承接。

## 1. 用户需求

我是这台机器上 AI Agent 集群的运维。手里有两台服务器和一批业务系统，想交给 postern 统一看管，让 agent1/agent2/agent3 之后能经网关受控使用，而**真实地址、账号、密码一概不让 Agent 碰到**。具体诉求：

- **把异构资源接进来**：服务器 A 经 SSH 到达，上面跑 3 个业务服务（HTTP 接口/页面）和一套 docker 运行时；服务器 B 经 AWS SSM 会话到达，上面有数据库 `db-main`。每个资源要声明清楚"经哪条 Transport 到达、由哪个 Adapter 解析能力、有哪些凭据等级"。
- **配好认证**：数据库 `db-main` 用**静态账号**（直配用户名密码）；业务系统 `svc-*` 要用**账号密码登录拿会话**（绝大多数是表单/Basic 登录，登一次复用），其中可能有系统强制 **2FA**——我接受 2FA 只在接入时人来按一次，但绝不要每次 Agent 请求都来按。
- **生成只读账号与凭据等级**：每个资源至少要有一个**只读 tier**（数据库 SELECT-only 账号、业务系统只读业务账号），写/管类动词若需要再单配更高 tier；后续 agent1 这类只读 Agent 自动落到只读 tier。
- **真实地址对 Agent 匿名化**：`10.0.3.x`、SSM instance-id、业务系统域名一律不让 Agent 看见，Agent 眼里只有 `db-main`/`svc-order` 这种代号。
- **接入后做红队自检**：我不想"相信它严格"，我想"验证它严格"——接入完跑一遍 `postern verify`，确认伪装写、越权、地址回显、Scope 外探测全被拒。

**痛点与覆盖的核心功能**：手工逐项配资源、记地址、贴账号既易错又泄密。本场景覆盖系统核心功能——**资源接入（transport/adapter/credential_tiers 声明）、资源系统认证生命周期（详细设计 6.13 三档：账号密码默认/token/OAuth-2FA；接入期引导）、匿名化建立（代号↔真实地址映射入机密面 5.4 `targets`）、能力面发现（发现≠授权，技术设计 10.1）、接入期 tier⊆账号真实权限校验（6.3）、`CLAUDE.md` 片段生成（07 CLI §3）、`postern verify` 红队自检（6.7）**。

## 2. 预期实现路径

接入是**纯控制面动作**（经 `control.sock`，技术设计 4.6 / 详细设计 6.5），不经数据面，不触求值管线。整套流程是 daemon 把"探测 + 校验 + 落库 + 入 vault"在服务端完成，CLI/SPA 只发起与呈现（07 CLI §4）。

**① 资源声明与传输/适配器绑定（5.2 schema、5.2bis ③、技术设计 10.1/10.2）**
一个资源 = `(代号 + Adapter + Transport + 标签 + 凭据等级集合 + engine_enforced)`。`svc-*` = `http` 适配器 + `ssh` 传输；`db-main` = `postgres` 适配器 + `ssm` 传输；`docker-A` = `docker` 适配器 + `ssh` 传输。`engine_enforced`：postgres=`true`（SELECT-only 账号引擎兜底）、http=取决于业务系统 RBAC（声明）、docker=`true`（账号即 socket 权限）。声明经 `POST /v1/resources`（6.5），落 `resources` 表。

**② 能力面发现（控制面 `Adapter::discover`，发现≠授权，技术设计 10.1、详细设计 6.8 CONS-20）**
向导触发 `POST /v1/resources/{code}/discover`，daemon 经该资源的 Transport 真实连上、由 Adapter 探测能力面（postgres 列 schema/表/列，http 从接口描述列可用路径，docker 列容器），回报 `CapabilitySurface`。**关键边界：发现只产出"有哪些对象"的事实，不授予任何访问**；探测结果供运维在控制面圈选，绝不暴露给数据面 Agent（数据面只有 `postern_surface`，是授权快照投影，不触达 `discover`）。

**③ 凭据等级声明与认证流程（auth_flow）落地（5.2 `resource_credential_tiers`、6.13）**
- **`db-main`（静态账号来源）**：声明 `ro` tier=SELECT-only 账号（capabilities=`observe,query`），`rw` tier=读写账号（按需，`mutate`）。账号是静态密码，无会话状态——`CredentialProvider` 直接取用（6.13 与会话来源并列的静态来源）。
- **`svc-*`（会话来源，6.13）**：声明 `ro` tier=只读业务账号、`op` tier=操作账号。其账号须经业务系统**登录**拿会话，非敏感配置（流程类型、认证端点、注入方式、刷新策略、是否需 2FA）写入 `resource_credential_tiers.auth_flow`（JSON 列）；敏感项（账号、refresh token / 长效会话）一律 `vault://` 引用。**持久凭据三档**（6.13）：①账号密码（默认、接入期零交互、过期用存着的账号密码无人值守重登）、②API token、③OAuth/强制 2FA（接入期人在场完成 OTP，换取可无人值守续期的长效会话）。

**④ 接入期 tier⊆账号真实权限校验（6.3、技术设计 10.5）**
凭据录入后，daemon 用该账号登录/连上**试探一次只读请求**，探测账号实际权限，与 tier 声明的 capabilities 比对：**声明 ⊆ 真实**才接入；声明动词超出账号真实权限 → 拒绝接入或降级 tier（一个声明 `ro` 却实际能写的账号必须报缺口）。本场景两类资源的探测手段：`db-main` 用 postgres 的 `has_table_privilege`/默认权限探测账号实际是否 SELECT-only；`svc-*` 用只读业务账号登录后试探一次只读接口、确认其实际可达接口与声明 tier 自洽（其它适配器同构，如 redis 6+ 用 `ACL GETUSER`，本场景不涉及）。这是"引擎层兜底"成立的前提，**不自动成立、需被验证**。

**⑤ 匿名化建立（5.4 `targets`、技术设计 11.3、详细设计 6.4）**
真实地址（A 的 SSH host、B 的 SSM instance-id/region、业务系统域名端口）写入机密面 vault 的 `targets` 表，键=资源代号，值=真实坐标；`transport_config` 在 policy.db 只存 `host_ref = "vault://{code}/target"`，**明文永不入 policy.db、永不出导出文件**（11.2 不变量）。机密面据 `targets`/`secrets` 构造系统级 **ScrubSet** 句柄，作为响应脱敏的兜底（6.4）。自此 Agent 侧只见代号，真实地址不进 Agent 可控的输入/输出回路（公理四）。

**⑥ 凭据落地（机密面、公理四）**
账号密码经 `POST /v1/credentials` 提交，daemon 经机密面写入 vault 为 `vault://{code}/{tier}` 引用，加密持久（XChaCha20-Poly1305，5.4）。CLI/SPA 只转发人类输入，**不在本地经手明文存储或派生**（07 CLI §4）。

**⑦ `CLAUDE.md` 片段生成（07 CLI §3，纯客户端文本渲染）**
向导可在客户端本地把"该 Principal 经哪个 MCP 端点、有哪些已授权动词"渲染成可粘贴文本。**只渲染事实**（已授权动词、MCP 端点位置），不编造引导话术（公理六）；来源是控制面回报的授权事实。此步零安全逻辑。

**⑧ 红队自检（6.7、技术设计 9）**
接入完触发 `POST /v1/verify`：daemon 以一个临时低权 Principal 自我发起一组**应被拒绝**的数据面请求（伪装写经只读 tier、Scope 外资源访问、地址回显诱导、提权尝试……），逐条确认 deny 且 deny 进审计。**任一项未被拒绝 → verify 失败并指出防线缺口**。这把"我相信严格"变成"我验证过严格"（公理三在运维层的延伸）。

**热生效边界**：资源声明、tier、auth_flow、凭据均为控制面写入，"一次事务 + 快照重建 + 审计事件"三联动（6.5）；新资源接入后即对求值快照可见，无需重启 daemon。

## 3. 页面与操作

### CLI（`postern ...`，人/脚本用）

- **接入向导（编排式）**：`postern init` —— 交互向导，把多步接入编排为一串控制面调用：建资源 → 触发 `discover` 探测 → 呈现候选对象供圈选 → 写 tier/细则/绑定 → 可选生成 `CLAUDE.md` 片段（07 CLI §3）。
- **单资源接入**：
  - `postern resource add db-main --adapter postgres --transport ssm`（声明资源；真实地址、账号随后经凭据录入入 vault，不上命令行明文）
  - `postern resource add svc-order --adapter http --transport ssh --auth session`（`--auth session` 标明走 6.13 会话来源，触发 auth_flow 引导）
  - `postern resource discover db-main` / `postern resource discover svc-order`（触发控制面 `discover` 接入侧探测，呈现能力面候选）
  - `postern resource list` / `postern resource disable <code>`
- **凭据录入**：`postern credential add`（提交账号密码等，CLI 仅转发给控制面，由 daemon 写入 vault；命令交互中输入的明文不回显、不落 CLI 本地）。
- **凭据等级声明**：`postern constraint add` / 经 `init` 圈选回写为资源的 `credential_tiers` 与 `constraints`（细则收窄到具体表/路由）。
- **红队自检**：`postern verify`（触发 `POST /v1/verify`，执行在 daemon）。
- **导出复核**：`postern export <file.toml>`（产物全 `vault://` 引用，无明文，用于复核接入结果）。

### 管理前端 SPA（人用；控制面 API 的视图）

- **资源接入页**：表单选 Transport（ssh/ssm/direct）、选 Adapter（postgres/http/docker…）、填代号与标签；提交即 `POST /v1/resources`。真实地址字段提交后即转为 vault 引用，页面回显代号而非明文。
- **能力面发现视图**：对某资源点"探测"，调 `POST /v1/resources/{code}/discover`，把 daemon 回报的候选对象（表/列、路由、容器）列成可勾选清单，运维**圈选**哪些纳入授权——页面明示"发现 ≠ 授权"，未圈选对象一律默认拒绝。
- **认证配置视图**：对 `svc-*` 配置 auth_flow（流程类型、认证端点、会话注入方式、刷新策略、是否需 2FA）；敏感账号字段提交即入 vault。需 2FA 的资源在此页一次性完成 OTP（接入期人在场）。
- **接入校验结果视图**：呈现 tier⊆账号真实权限校验结果（通过/缺口），缺口项高亮并阻断接入完成。
- **红队自检视图**：一键触发 verify，逐项展示九类探测的 PASS/FAIL（八项判据为被拒、第 8 项脱敏探测判据为放行且响应擦净）；任一项 FAIL 红色告警并指出缺口防线。

### 数据面（Agent 用，MCP 工具 / HTTP 外壳）

**不适用于接入配置本身**——接入是控制面动作，Agent 不参与。与本场景相关的数据面接触面仅两点（均为接入成果的下游体现，运行细节归场景 04）：

- 接入完成后，Agent 经 `postern_surface` 看到的是**授权快照投影**（其 Scope 内已授权能力面），**不触达 `Adapter::discover`、不见真实地址**（6.8 CONS-20）。
- Agent 经 `postern_query`/`postern_mutate` 等固定动词工具操作时，只以资源代号寻址（如 `resource="db-main"`），凭据/地址全程零接触（公理四、七）。

## 4. 使用方式与预期结果

### 4.1 正常操作

> 以"接入 `db-main`（静态账号）+ `svc-order`（账号密码会话登录，可能 2FA）"为主线，逐步走查；每步给可验收预期。

**步骤 1 · 声明资源与传输/适配器**
- 操作：`postern resource add db-main --adapter postgres --transport ssm`；`postern resource add svc-order --adapter http --transport ssh --auth session`。
- 预期结果：daemon 经 `POST /v1/resources` 写入 `resources` 表，返回资源代号 + `version`（乐观锁基线）。`db-main` 的 `engine_enforced=true`、`svc-order` 的 `engine_enforced` 按业务系统 RBAC 声明。资源此刻**无任何凭据、无任何授权格**——存在但默认拒绝一切（公理一）。写入产生一条 `policy_change` 审计事件。

**步骤 2 · 录入真实地址（匿名化建立）**
- 操作：在接入向导/SPA 资源页提供 `db-main` 的真实坐标（host/port）、`svc-order` 的域名端口；提交。
- 预期结果：真实地址写入机密面 vault 的 `targets`（键=`db-main`/`svc-order`），policy.db 的 `transport_config` 只存 `host_ref="vault://db-main/target"` 等引用。`postern export` 产物里**这些字段一律为 `vault://` 引用，无任何明文地址**（11.2 不变量可验收）。机密面据此更新 ScrubSet 句柄。

**步骤 3 · 录入凭据（凭据零接触）**
- 操作：`postern credential add` 录入 `db-main` 的 SELECT-only 账号密码、读写账号密码；录入 `svc-order` 只读业务账号、操作账号的用户名密码。
- 预期结果：明文经控制面交由机密面加密写入 vault 为 `vault://db-main/ro`、`vault://db-main/rw`、`vault://svc-order/ro`、`vault://svc-order/op`。CLI 交互中输入的明文不回显、不落本地、不入 policy.db。一条 `credential_event` 审计落地，**只记"哪个账号/tier"，绝不记账号密码值**（6.13）。

**步骤 4 · 声明凭据等级（生成只读账号/tier）**
- 操作：经 `init` 圈选或声明，为 `db-main` 配 `ro`(capabilities=`observe,query`,`vault://db-main/ro`) 与 `rw`(`mutate`,`vault://db-main/rw`)；为 `svc-order` 配 `ro`(`observe,query`,`vault://svc-order/ro`) 与 `op`(`mutate`,`vault://svc-order/op`)。
- 预期结果：写入 `resource_credential_tiers` 表，`(resource,tier)` 在未删除集内唯一。每个资源**至少有一个只读 tier**。后续策略引擎按本次动词选 tier，只读动词自动落 `ro`（场景 03/04 承接）。

**步骤 5 · 配置业务系统认证流程（auth_flow，含可能的 2FA）**
- 操作：对 `svc-order` 声明 `auth_flow`：`flow=form_login`、`auth_endpoint=/api/login`、`session_injection={kind:cookie,cookie_name:JSESSIONID}`、`csrf` 提取/回填、`refresh={kind:session_renew,...,on_hard_expire:fail_closed}`、`credential_ref=vault://svc-order/op`；若系统强制 2FA 则 `twofa={required:true,stage:onboarding_only,method:otp}`，此刻运维在控制面一次性完成 OTP。
- 预期结果：非敏感配置入 `resource_credential_tiers.auth_flow`（JSON），敏感账号仍 `vault://` 引用（6.13 schema 落点）。**第①档（账号密码）路径下接入期零交互**，过期由存着的账号密码无人值守重登；仅第③档强制 2FA 时人在场按一次 OTP，换取可数据面无人值守续期的长效会话写入 vault（加密）。一条 `credential_event` 记接入引导，不记 OTP/令牌值。

**步骤 6 · 能力面发现（发现≠授权）**
- 操作：`postern resource discover db-main`（或 SPA "探测"）。
- 预期结果：daemon 经 ssm 通路真实连上 B、postgres 适配器探测，回报 `CapabilitySurface`（schema/表/列清单）。清单仅为"有哪些对象"的**事实**，**不授予任何访问**；运维在其上圈选业务表纳入 `query` 细则（`table_allow`），PII 列纳入 `column_mask`。未圈选对象一律默认拒绝（公理一）。探测结果只在控制面呈现，**绝不进入数据面 / Agent 视野**。

**步骤 7 · 接入期 tier⊆账号真实权限校验**
- 操作：录入凭据后由 daemon 自动执行（向导/SPA 呈现结果）。
- 预期结果：daemon 用 `db-main/ro` 账号连上探测实际权限（`has_table_privilege` 等），确认账号**实际只有 SELECT**、与 `ro` 声明的 `observe,query` 自洽；用 `svc-order/ro` 登录试探一次只读请求，确认会话可建立、账号 tier 与声明动词集自洽（6.3、6.13）。**声明 ⊆ 真实 → 校验通过、接入完成**；该校验项同时纳入 `postern verify` 常规项。

**步骤 8 · 生成 `CLAUDE.md` 片段**
- 操作：向导询问是否生成片段，或后续 `postern grants <principal>` 取授权事实自行渲染。
- 预期结果：客户端本地渲染出"agent1 经哪个 MCP 端点（`data.sock` 的 `/mcp`）、对 `db-main`/`svc-order` 有哪些已授权动词"的可粘贴文本。**内容只含事实**（已授权动词、端点位置），不含编造的引导话术（公理六）；片段不含真实地址、不含凭据。

**步骤 9 · 红队自检确认严格**
- 操作：`postern verify`。
- 预期结果：daemon 以临时低权 Principal 逐条发起红队请求，返回**九类探测逐条 PASS 且均进审计**的报告（与场景 07 §4.1-C 九项表同源，6.7 逐项）——其中**八项的 PASS 判据是被拒**（含 `stage`+`reason`），**第 8 项脱敏探测的 PASS 判据是放行但响应经擦净**（非 deny）：① 伪装写经只读 tier 被拒（归类/引擎层）、② CTE 包裹 DELETE 经 mutate 授权被拒（按最高危写节点归 destroy）、③ 模板外自由命令拒、④ Scope 外资源代号访问拒且不泄露该资源是否存在、⑤ 已吊销/已过期凭证认证层拒、⑥ 可信域外来源用受限凭证拒、⑦ 提权类（请求 admin 性质能力）模型层无该授权、⑧ 地址回显诱导请求**放行但返回内容经擦净**（无真实地址/凭据回显）、⑨ 审批关闭下 escalate 标注操作立即 deny。此外接入相关的两类常规校验项同样纳入 verify：tier⊆账号真实权限（6.3，见步骤 7/E5）、`data.sock` 可连 uid 不含 daemon 自身 uid（06 daemon §3.1 同 uid 前置条件）。**任一项未被拒 / 任一校验项不通过 → verify 失败并明确指出防线缺口**（6.7）。verify 通过即"接入严格性已验证"。

### 4.2 异常场景与预期结果

> 一律 fail-closed：不确定即拒绝、凭据零接触、只说事实、匿名化代号、响应脱敏。接入异常多在**控制面**暴露给运维（不进数据面），但凡涉及返回内容一律脱敏。

**E1 · 业务系统账号密码错误 / 会话登录失败（接入期）**
- 触发：`svc-order` 录入的账号密码错误，或登录端点返回认证失败。
- 系统行为：步骤 7 的接入期试探登录失败；daemon 不完成接入、不写 `ro/op` 凭据为"已校验可用"，把失败如实回报为控制面错误码。
- 预期结果：接入**中止**（fail-closed），向导/SPA 提示"会话登录失败，账号或认证流程配置不正确"，**不回显业务系统返回的原始错误串/地址**（7.2-1 跨边界先脱敏）。已录入的明文不被当作可用凭据生效；运维更正后重试。绝不"先接入再说"。

**E2 · 需 2FA 但接入期无人值守 / OTP 未完成**
- 触发：`svc-order` 为第③档强制 2FA，但接入由脚本无人值守发起，或 OTP 未在窗口内输入。
- 系统行为：2FA 是 `stage=onboarding_only` 的接入期、人在场成本（6.13）；无人完成 OTP 则换不到长效会话，daemon 无法完成接入期校验。
- 预期结果：接入**不完成**（fail-closed），明确提示"该资源强制二次验证，须在控制面由人在场完成 OTP 方可接入"。**设计不假装能绕过 2FA**——若该系统每次登录都强制 2FA 且不提供任何可刷新长效会话，则如实记录该约束，标注"网关只能在单次会话有效期内运作，硬过期后数据面 fail-closed 拒绝并提示重新接入"（6.13 诚实边界）。不静默降级、不在数据面塞 2FA 路径。

**E3 · 探测发现端口未发布 / 通路不可建（代修）**
- 触发：`postern resource discover svc-order` 时，业务服务 80 端口未经 ssh 转发发布、或 ssm 会话目标不可达。
- 系统行为：daemon 经 Transport 试连失败，`Adapter::discover` 拿不到 Channel；连接管理层把失败如实回报为缺口（07 CLI §4 "代修"：daemon 报回缺口，向导呈现并发起修正调用）。
- 预期结果：探测**返回失败 + 具体缺口**（如"目标端口不可达，疑未发布转发"），向导提示并可发起"代修"——把缺口转译为后续控制面修正调用（如补 `transport_config` 转发声明），**修正动作的语义合法性裁决仍在 daemon**。缺口未消解前资源不进入可用接入状态（fail-closed）。回报的错误**经脱敏，不含真实地址/端口明文**（7.2-1，`connection refused to 10.0.3.17` 一类绝不外泄）。

**E4 · 真实地址泄漏尝试（接入物料 / 导出 / 数据面回显）**
- 触发：① 运维误把真实地址写进资源代号或细则文本；② `postern export` 后检查是否漏明文；③ 接入后 Agent 经数据面诱导回显连接串/IP。
- 系统行为：①真实地址只允许进 vault 的 `targets`，policy.db 字段只存 `vault://` 引用；导出端点对一切敏感项强制 `vault://` 引用（11.2 不变量）；③数据面一切返回字节过同一 Sanitizer + ScrubSet（6.4、7.2-3）。
- 预期结果：①若把明文塞进非引用字段，导入/写入校验拒绝或该明文不被解析为地址（不进通路）；②`export` 产物**全程无明文地址/凭据**，可逐字段验收；③Agent 任何"回显地址/连接串"的请求，返回内容经 ScrubSet 擦除，**Agent 拿不到真实地址**（`postern verify` 第 8 项覆盖）。匿名化 + 凭据零接触是上游保证，脱敏是兜底，三重不依赖单点。

**E5 · tier 声明动词 > 账号真实权限（接入校验失败）**
- 触发：把 `db-main` 的 `ro` tier 声明为含 `mutate`，但绑定的 vault 账号实际是 SELECT-only（或反之：声明 `ro` 却绑定了一个实际能写的账号）。
- 系统行为：步骤 7 接入期主动探测账号实际权限并与声明比对（6.3）。"声明 ⊆ 真实"不成立即失败。
- 预期结果：**接入被拒或要求降级 tier**（fail-closed），报缺口"`ro` 声明含 `mutate`，但账号无写权限"或"声明 `ro` 的账号实际拥有写权限，违反只读兜底前提"。运维须改声明或换账号方可接入。该项纳入 `postern verify` 常规项，发布前必跑（6.3、6.7）。绝不让一个名实不符的 tier 进入可用状态——否则"引擎层兜底"形同虚设。

**E6 · 机密保险箱未解锁（vault locked）**
- 触发：录入凭据 / 接入期登录试探 / 探测时，机密面保险箱尚未解锁。
- 系统行为：机密面无法取 `targets`/`secrets`，`CredentialProvider.credential_for` 与地址解析均失败（5.4、6.3）。
- 预期结果：相关接入步骤 **fail-closed 失败**，提示"机密保险箱未解锁，无法写入/校验凭据"。daemon 不退化为"无凭据接入"或明文落库；解锁保险箱后重试。数据面同理：保险箱未解锁时任何需建连的请求在步骤[7b] deny（公理二）。

**E7 · 接入并发冲突（乐观锁）**
- 触发：两名运维同时改同一资源的 tier/auth_flow，或 SPA 与 CLI 并发写。
- 系统行为：控制面更新/删除端点必须携带期望 `version`，不匹配返回 `409 Conflict` 并写 `policy_change` 审计（6.5 乐观锁）。
- 预期结果：后写者收到 `409 Conflict`（可验收），需重新读取最新 `version` 再改；不会发生静默覆盖。期望 version 的唯一来源是先前读取值。

**E8 · 能力面发现被误当授权 / 数据面试图触发 discover**
- 触发：① 运维以为"探测出来的表就能查了"；② Agent 经数据面试图调用接入侧 `discover`。
- 系统行为：①发现产出的对象在被显式纳入授权格细则前一律默认拒绝（技术设计 10.1，发现≠授权）；②`Adapter::discover` 只接受控制面触发，数据面 `postern_surface` 是快照投影、**禁止触达 `discover`**（6.8 CONS-20，术语边界由命名规范固化）。
- 预期结果：①未圈选的表/路由/容器 Agent 一律 deny（场景 04/05 体现）；②Agent 无任何路径触发接入侧探测——`postern_surface` 只返回其 Scope 内已授权能力面，不连任何底层资源、不列未授权对象、不泄露真实地址。

**E9 · daemon 重启后接入状态恢复**
- 触发：接入配置完成后 daemon 重启。
- 系统行为：资源/tier/auth_flow/匿名化映射均为持久状态（policy.db + vault），启动序列「开库 → 重建快照 → 解锁保险箱 → 注册插件 → 开放数据面」逐步推进（06 daemon §3.1 boot），重建首份策略快照后数据面最后开放。
- 预期结果：重启后已接入资源、tier、auth_flow、`targets` 映射**完整恢复**，无需重新接入；保险箱未解锁前数据面 fail-closed（同 E6）。live-session 缓存（内存、Zeroizing）不持久——重启后首次建连按持久凭据形态重新登录（第①档无人值守重登 / 第③档刷新或重新接入），符合 6.13 运行期物化。

**E10 · 接入期审计写失败**
- 触发：录入凭据/接入引导/探测时 `credential_event`/`policy_change` 审计写失败。
- 系统行为：写入端点遵循"一次事务 + 快照重建 + 审计事件"三联动（6.5）；审计相对状态变更是安全不变量（7.2-5）。
- 预期结果：接入写入**不被确认为成功**（fail-closed），向导/SPA 报"审计降级，接入未完成"，不留"已生效但无痕"的接入。运维据错误码处置后重试。绝不让接入成为唯一不留痕的状态变更路径。
