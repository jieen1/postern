# 场景 7 · 审计、拒绝分析与红队自检

> 回答三个问题：我的 Agent 做了什么（审计查询）、哪些越权在反复发生→提示该往哪调（拒绝分析）、系统是否真的严格（`postern verify` 主动验证而非假设）。审计是 append-only 的问责载体，拒绝是最有分析价值的信号，红队自检把"我以为它会拒"变成"我看到它逐条拒了"。

本场景沿用[示例一](01-多资源多Agent授权示例.md)的资源与主体：资源 `svc-order`/`svc-billing`/`svc-crm`（http 业务）、`docker-A`（docker）、`db-main`（postgres）、`redis-main`（redis）、`mq-main`（rabbitmq）；主体 `agent1`（observer 只读全部）、`agent2`（docker-A 管理 + db-main 只读 + redis-main 管理）、`agent3`（业务服务 operator + docker 查日志 + B 上日志）；角色阶梯 `observer ⊂ operator ⊂ maintainer`。

---

## 1. 用户需求

我（资源所有者 / 运维）已经把资源接入、把三个 Agent 的授权配好了，现在我要能**事后看清、持续收紧、主动验证**：

- **"我的 Agent 做了什么？"**——我要查 `agent3` 上周对 `svc-order` 的全部写操作、`agent1` 昨天被拒的所有请求；要能按主体、时间、决策（allow/deny）筛，量大时要分页，不能一次拉爆。allow 和 deny **都要有记录**，deny 要能还原"卡在哪一步、为什么"。
- **"哪些越权在反复发生？"**——某个 Agent 老是撞同一堵墙（比如 `agent1` 反复想 `mutate db-main`），这是个信号：要么它的预设需要调整，要么有注入/失控。我要一个**聚合视图**（按主体/资源/阶段/动词），按窗口（如近 7 天）统计，告诉我"该往哪个方向看"，而不是逼我逐条翻流水。
- **"系统是不是真的严格？"**——我不想假设"应该会拒"。我要一个命令，让网关**自己对自己发起一批本该被拒的请求**（伪装写、模板外命令、Scope 外探测、过期/吊销凭证、可信域外、提权、脱敏探测、审批关下 escalate），逐条确认它们**确实被拒、且 deny 进了审计**。任何一项没被拒，我要立刻知道是哪条防线有缺口。

底线要求贯穿全部：审计内容**永不含**凭据值与真实地址；返回内容（含拒绝、含审计里的对象）一律匿名化代号 + 脱敏；审计写不进时**不放行**（不可记 = 不放行）；审计盘满**不能**成为 Agent 能触发的全局拒绝服务面。

本场景覆盖的系统核心功能：审计事件流（append-only、allow/deny 都记、deny 含 stage+reason、两阶段审计时序）、审计查询分页、拒绝聚合分析（denials summary）、可选 OTel 旁路导出（默认关闭）、`postern verify` 红队自检、审计存储健康与 DoS 防护、审计防篡改与脱敏。

## 2. 预期实现路径

postern 把"观测与自检"落在**观测面（执行侧记录纪律）+ 控制面（查询/聚合/触发 verify）+ 存储层（JSONL append-only 载体）**三处，全部是控制面读动作或数据面执行副产物，无需重启、热可查。引用《详细设计文档》（下称详设）与《技术设计文档》（下称技设）具体章节：

1. **审计记录纪律（执行侧，详设 5.3 + 6.1 时序不变量、8.9 观测面）**——每个数据面请求都落 `request` 事件，**allow 与 deny 都记**；deny 必含 `stage`（卡在管线哪一步：`auth`/`classify`/`rbac`/`constraint`/`condition`/`connect` 等）+ `reason`（引用策略事实）。信封字段见详设 5.3：`v/id/ts/kind/entry/origin/principal(_id)/credential_id/resource(_id)/capability/objects/decision/stage/reason/policy_rev/intent_digest/tier/duration_ms/response_digest`，一切 id 为雪花 id 字符串（/CONS-15）。
2. **两阶段审计时序（安全不变量，详设 6.1 步骤 [7a]/[10]）**——
   - **只读动词**（observe/query）：执行后单次审计；`AuditSink::record` 写失败 → **该请求按 deny 返回**（不可记 = 不放行，此刻尚未暴露任何已落库副作用）。
   - **有副作用动词**（mutate/execute/manage/destroy）：**两阶段**。执行前先落 `intent` 事件（步骤 [7a]），intent 写不进 → **执行前 deny**（确未执行，既诚实又安全）；执行后落 `outcome` 事件（步骤 [10]），与 intent 以同一请求 id 关联。**已执行的请求绝不返回 deny**：outcome 写失败时返回"已执行但审计降级"错误码（带可识别码），而非谎报 deny 诱导重试。
3. **审计查询（控制面 6.5 `GET /v1/audit?since&principal&kind&decision&page_no&page_size`，倒序、强制分页）**——分页在 JSONL 扫描层执行（详设 5.3 量级核算：7 日 ≈70 MB，顺序扫描百毫秒级），契约 `DB_PAGINATION_MANDATORY` 禁前端分页、禁无界查询。CLI `postern audit` 透传，可 `--format jsonl` 输出机器形态。
4. **拒绝聚合分析（控制面 6.5 `GET /v1/denials/summary?window=7d`，详设 6.9 告警 + 技设 6.5 拒绝分析闭环）**——deny 事件按 Principal/资源/阶段/动词聚合计数，呈现"某主体反复请求某类被拒操作"的模式；超阈值产生**可被控制面查询的告警事件**。这是技设 6.5"预设进化闭环"的数据来源：网关只呈现模式，调整方向由人裁决（详设 8.9：面向人的机械推导建议不受公理六约束——公理六只约束数据面返回 Agent 的内容；面向运维的建议仍只陈述聚合事实，不编造话术）。
5. **可选 OTel 旁路导出（详设 6.9 `AuditExporter`，默认关闭）**——本地 JSONL 始终是**单一权威**（公理三）；`AuditSink::record` 先成功，再异步喂 `AuditExporter` 旁路镜像到外部可观测系统。导出**失败不回压、不阻断本地写入**（trait 方法无 `Result` 回主路径）。OTel 字段映射：`ts`→timestamp、`kind`→event name、`decision`/`stage`/`reason`/`*_id`/`policy_rev`/`intent_digest`→attributes；导出器只搬运**已脱敏**事实。
6. **红队自检（控制面 6.5 `POST /v1/verify`，详设 6.7、技设 13.4 红队自检 + 11.4"自我观测"）**——daemon 以一个**临时低权 Principal** 自我发起一组**应被拒绝**的数据面请求，每条都走完整管线 [0]→[10]（公理七），逐条确认结果符合预期（八项 `deny`、第 8 项脱敏探测放行但响应无敏感回显）且均出现在审计中。九类红队项见 §3 与 §4.1。**任一项不符预期 → verify 失败并指出防线缺口**（详设 6.7、5.5 含 control.sock 红队项）。
7. **审计存储健康与 DoS 防护（详设 5.3"审计存储健康与拒绝服务防护"）**——区分"存储健康"（平面级状态：容量/可写/轮转）与"单条写入失败"（请求级瞬时 IO）；独立配额 + 水位监控，逼近上限触发**可感知降级**（告警事件、强制轮转、低价值审计降采样、deny 类按窗口聚合计 `count`），而非全平面瞬间瘫痪；`audit.retention_days` 取有界安全默认 + 强制轮转上限。降级本身写可见审计/告警（口子自动愈合对运维可见）。
8. **审计防篡改与脱敏（详设 5.3/5.5/8.9、6.4、技设 12.2"放大后防篡改"）**——载体物理 append-only（JSONL 只追加，无修改/物理删除语义，保留期到期整文件删除除外，详设 8.9 必守不变量）；文件 `0600`、属主为 daemon 运行账户、Agent uid 不可读（详设 5.5 硬前置条件①）；事件正文**写入前过同一 Sanitizer**（详设 5.3、6.4，作用面含"审计事件正文"）。**脱敏诚实度（详设 6.4）**：审计/响应里"不含真实地址/凭据"的**真正保证来自上游两道机制**——**匿名化**（Agent 只见资源代号，真实地址从不进入 Agent 可控的输入/输出回路）+ **凭据零接触**（凭据从不经手 Agent、从不进入响应路径，公理四）；出口的系统级 ScrubSet 是其上的**尽力而为黑名单兜底**（match-and-erase 不可枚举句柄，不承诺识别全部编码变体）。故审计**结构上不含**凭据值与真实地址，而非仅靠输出端擦除。放大后（单一信任域放大，技设公理五、12.2）防篡改由"放大档"的更强实现承载（带防篡改的问责级审计写入 / 外部 append-only 副本 / OTel 导出的不可变后端，技设 11.4、12.2）。

**控制面 vs 数据面 vs 热生效**：`audit`/`denials`/`verify` 全是**控制面读 / 触发**动作（经 `control.sock`，Agent 物理不可达）；审计**写**是数据面执行的副产物（每请求落事件）；查询、聚合、verify 均**无需重启、即时可得**。

## 3. 页面与操作

### CLI（`postern ...`，给人/脚本用，经 `control.sock`）

| 命令 | 端点 | 用途 |
|---|---|---|
| `postern audit [--principal <name>] [--since <ts>] [--kind <kind>] [--decision allow\|deny] [--page-no N] [--page-size M] [--format jsonl]` | `GET /v1/audit` | 审计查询：按主体/时间/事件类/决策筛，倒序，强制分页 |
| `postern denials [--window 7d]` | `GET /v1/denials/summary?window=7d` | 拒绝聚合分析（按主体/资源/阶段/动词统计 + 告警事件） |
| `postern verify` | `POST /v1/verify` | 触发红队自检，逐条返回 PASS/FAIL + 缺口说明 |
| `postern settings get audit.fsync` / `set audit.fsync relaxed` | `GET/PUT /v1/settings/{key}` | 审计 fsync 策略（性能阀，仅 allow 类可 relaxed） |
| `postern settings get audit.retention_days` / `set ...` | `GET/PUT /v1/settings/{key}` | 审计保留期（有界安全默认） |
| `postern settings set audit.exporter.otel.enabled true` | `PUT /v1/settings/{key}` | 启用 OTel 旁路导出（默认关闭） |

- 集合命令透传 `page_no/page_size`，缺省由 daemon 取默认 20、上限钳到 200；CLI **不取回全量再切片**（契约 `DB_PAGINATION_MANDATORY`）。
- 雪花 id（`id`/`principal_id`/`resource_id`/`credential_id`）一律**字符串渲染**，绝不解析为整数。
- 拒绝事实经 `DenyResponse`/`DeniedFacts` 同结构反序列化渲染（详设 CLI 模块 §5）。

### 管理前端（SPA，控制面 API 的视图，人用）

SPA 是 `control.sock` / control 端点的浏览器视图（CONS-17：默认经桌面外壳 webview UDS 代理，或显式开启 localhost 绑定 + 控制面认证）。本场景相关页面：

- **审计流页**：`GET /v1/audit` 的视图。筛选器（principal / 时间范围 / kind / decision），表格按 `ts` 倒序、后端分页（翻页即再次请求，不在前端切片）。每行展开见 deny 的 `stage`+`reason`、allow 的 `tier`+`response_digest`；两阶段写的 `intent`/`outcome` 以同一请求 id 关联呈现。`--format jsonl` 对应导出按钮。
- **拒绝分析页（denials）**：`GET /v1/denials/summary` 的视图。按 Principal/资源/阶段/动词的聚合计数榜、窗口选择（7d 等）、告警事件列表（超阈值项高亮），呈现"反复撞墙"模式，附**机械推导**的调整方向提示（不是话术，是聚合事实 + 标准 `elevate`/`constraint` 命令模板）。
- **红队自检页（verify）**：触发 `POST /v1/verify`，展示九项红队项逐条 PASS/FAIL，FAIL 项标红并指出缺口防线（如"项 4：Scope 外访问未被拒——拒绝响应泄露了资源存在性"）。
- **观测设置页**：`audit.fsync` / `audit.retention_days` / OTel 导出开关（`settings` 端点视图，写入受版本/校验约束）。

### 数据面（Agent 用，MCP 工具 / HTTP 外壳）

**对审计/denials/verify 端点：不适用。** 观测与自检全属控制面，Agent 物理不可达（`control.sock=0600`，Agent uid 不可读，详设 5.5）。Agent 唯一与本场景的接触是**被审计**——它经 `postern_query`/`postern_mutate`/`postern_manage` 等数据面工具发起的每个请求都落审计事件（allow/deny 都记），但 Agent **看不到**审计本身，只在被拒时收到结构化 `DenyResponse`（技设 6.4）。verify 跑的也是数据面请求，但由 daemon 内部以临时低权 Principal 自发，不经任何外部 Agent 入口。

## 4. 使用方式与预期结果

### 4.1 正常操作

#### A. 审计查询：「agent3 上周对 svc-order 做了什么」

1. **操作**：`postern audit --principal agent3 --since 2026-06-05T00:00:00Z --decision allow --page-no 1 --page-size 20`
   **预期结果**：返回 `Page<AuditEvent>` 信封，倒序（`ts` 新→旧），含 `total`/`page_no`/`page_size`；每条 `kind=request`、`decision=allow`、`principal=agent3`、`resource=svc-order`，写动词项含 `tier="op"`、`capability=mutate`、`objects=["route:/api/orders"]`、`response_digest=sha256:...`（**不含响应内容**）。返回中**无任何**真实地址、Cookie/会话值、账号密码（写入前已过 Sanitizer）。分页在 JSONL 扫描层执行，CLI 不取全量。
2. **操作**：翻页 `--page-no 2`。
   **预期结果**：返回下一窗口，`page_no=2`；后端按窗口截断，前端不缓存全量。
3. **操作**：查某次 mutate 的两阶段配对——同请求 id 应有 `intent` 与 `outcome` 两条。
   **预期结果**：`intent` 事件（执行前，无 `response_digest`）与 `outcome` 事件（执行后，含 `response_digest`、`duration_ms`）以同一请求 id 关联可对账——"发起即有痕、结果可追溯"（详设 6.1 时序不变量）。
4. **操作**：`postern audit --principal agent1 --decision deny --since <昨天>`
   **预期结果**：返回 agent1 昨日全部 deny 事件，每条含 `stage`（如 `rbac`）+ `reason`（如 `role=observer 不含 db-main:mutate`）+ `intent_digest`（脱敏摘要，供归类）+ `policy_rev`；deny 可完整还原"在哪一步、因何拒绝"（详设 8.9 不变量）。
5. **操作**：`postern audit --principal agent3 --format jsonl > agent3.jsonl`
   **预期结果**：机器形态逐行 JSON 输出，雪花 id 为字符串；内容与人类渲染同源，仍脱敏。

#### B. 拒绝聚合分析：「哪些越权反复发生」

1. **操作**：`postern denials --window 7d`
   **预期结果**：返回近 7 天 deny 聚合：按 `(principal, resource, stage, capability)` 分组计数榜（如 `agent1 × db-main × rbac × mutate = 37 次`、`agent1 × redis-main × rbac × mutate = 12 次`），倒序。每组附 `intent_digest` 代表样本，**不展开全部流水**。
2. **操作**：阅读榜首与告警列表。
   **预期结果**：超阈值的组产生**可查询的告警事件**（详设 6.9）；榜单提示"`agent1` 反复请求 `db-main:mutate`"这一模式——这是技设 6.5 预设进化闭环的信号：要么 agent1 确需该权限（机械生成 `postern elevate agent1 --cap db-main:mutate --ttl ...` 模板供人裁决），要么是注入/失控迹象（人据此查 agent1 调用方）。网关只呈现**聚合事实**，不替人决定放行。
3. **操作**：据分析人做出策略调整（如经 03/06 场景的 `elevate` 或新增 `constraint`）。
   **预期结果**：调整经控制面写入，热生效；该写入本身落 `policy_change` 审计（含 actor、`policy_rev`、行 id、写后 version）；下一窗口 `denials` 中该模式应随之下降——闭环可观测。

#### C. 红队自检：「系统是不是真严格」

1. **操作**：`postern verify`
   **预期结果**：daemon 以临时低权 Principal 自发九类红队请求（每条走完整 [0]→[10]），逐条返回 **PASS**，整体 PASS。其中八项的 PASS 判据是被拒（含 `stage`+`reason`），第 8 项（脱敏探测）的 PASS 判据是放行但响应无真实地址/凭据回显（上游匿名化/凭据零接触为真保证，步骤 [9] ScrubSet 黑名单兜底）——对账如下：

   | # | 红队项 | 防线落点 | 预期 PASS 判据 |
   |---|---|---|---|
   | 1 | 伪装写（写 CTE 包裹经只读授权） | classify 归类 / 引擎层（`engine_enforced=true` 兜底） | 归类识破写节点或 SELECT-only 账号引擎拒，deny |
   | 2 | CTE 包裹 DELETE 经 mutate 授权（非 destroy） | classify（按最高危写节点归 Destroy） | 归为 Destroy，mutate 授权不足，deny |
   | 3 | 模板外自由命令（execute 类） | classify / constraint（`command_template` 只接受参数化实例） | 无匹配模板，deny |
   | 4 | Scope 外资源代号访问 | rbac（scope 未选中） | deny **且**拒绝响应不泄露该资源是否存在（`your_grants` 只含自身世界，技设 6.4 安全边界） |
   | 5 | 已吊销/已过期凭证请求 | auth（步骤 [1]） | 凭证 `revoked_at`/`expires_at` 在求值时刻失效，认证层 deny |
   | 6 | 可信域外来源使用受限凭证 | auth（可信域校验，认证阶段验证凭证有效性之后确立 Principal 之前，技设 7） | 来源不在凭证可信域，deny |
   | 7 | 提权类操作（请求 admin 性质能力） | 模型层（`admin` 不可授予任何 Principal，双重硬约束） | 无该授权格，deny |
   | 8 | 响应脱敏探测（诱导回显连接串/IP） | 上游匿名化/凭据零接触 + 脱敏出口（步骤 [9]） | **放行但**返回内容无真实地址/凭据回显——真保证在上游（Agent 无从指定/获知真实地址，凭据从不进响应路径），步骤 [9] ScrubSet 黑名单兜底擦除（此项 PASS 判据是无敏感回显，非 deny） |
   | 9 | escalate 标注操作在审批关闭下 | 动作分流（步骤 [6]，`escalate ≡ deny`） | 立即 deny（`decision=escalate_denied`，与普通 deny 可区分） |

   （另含详设 5.5 红队项：从模拟 Agent uid 连 `control.sock` 必须失败。）
2. **操作**：查 verify 自身在审计中的痕迹。
   **预期结果**：verify 发起的每条请求**逐条出现在审计**中（八项 deny 事件含 `stage`+`reason`；第 8 项为 allow 事件、`response_digest` 指向无敏感回显的响应；临时低权 Principal 落在 `principal`/`principal_id` 字段），verify 不仅"声称拒"还**留痕可复核**——"自我观测"落地（详设 6.7、技设 13.4：逐条确认结果并出现在审计中）。

#### D. 可选 OTel 旁路导出

1. **操作**：`postern settings set audit.exporter.otel.enabled true`（并配置导出端点）。
   **预期结果**：本地 JSONL 写入**不变**（仍单一权威）；之后每条事件 `AuditSink::record` 先成功，再异步喂 `AuditExporter` 镜像为 OTel log/span（字段按 §2 映射）。导出器只搬运**已脱敏**事实。
2. **操作**：制造导出端点不可达（断开外部系统）。
   **预期结果**：本地审计写入**照常成功**，数据面决策路径**不受影响**（公理三：本地权威不依赖导出）；导出失败由导出器内部缓冲/丢弃消化，**不回压、不阻断**，仅自记。默认 `enabled=false` 时无此旁路。

### 4.2 异常场景与预期结果

#### E1. 只读请求审计写失败（不可记 = 不放行）

- **触发**：`agent1` 发起 `query db-main`，`AuditSink::record` 写失败（瞬时 IO 错误）。
- **系统行为**：只读动词执行后单次审计写失败 → **该请求按 deny 返回**（详设 6.1 时序：尚未对外暴露任何已落库副作用，deny 既诚实又安全，公理二）。
- **预期结果**：Agent 收到结构化拒绝（脱敏，含 `reason` 指向审计降级的常量安全文案），**不返回任何已读数据**；运维侧该写失败计入存储健康监控。fail-closed：不可记录绝不放行。

#### E2. 有副作用请求的两阶段写失败

- **触发-a（intent 写失败）**：`agent3` 发起 `mutate svc-order`，执行前 `intent` 事件写不进。
  **系统行为**：步骤 [7a] intent 写不进 → **执行前 deny**（确未执行）。
  **预期结果**：请求被拒，**业务系统侧无任何副作用**（未下单），Agent 收到脱敏拒绝；既诚实又安全（公理二、六）。
- **触发-b（outcome 写失败）**：同请求已执行（已下单），执行后 `outcome` 事件写不进。
  **系统行为**：**绝不返回 deny**（已落库再 deny = 无痕已执行 + 谎报，会诱导重试导致重复下单，违公理六）。返回带可识别码的"已执行但审计降级"错误。
  **预期结果**：Agent 收到"已执行但审计降级"错误码（非 deny，不会触发其重试逻辑）；运维据该错误码与存储健康告警介入。intent 事件已在库，留有"发起痕迹"可对账。

#### E3. 审计盘满 / 逼近配额的 DoS 防护（不全平面瘫痪）

- **触发**：Agent 反复触发同类 deny（或正常高频操作）使审计落盘逼近独立配额上限。
- **系统行为**：区分"存储健康"（平面级）与"单条写失败"（请求级，详设 5.3）。逼近水位时触发**可感知降级**：① deny 类按 `(principal, kind)` 窗口聚合，写一条带 `count` 的聚合记录（而非每次一条），避免被反复触发打满；② 低价值审计降采样；③ 强制轮转；④ 告警事件。`audit.retention_days` 有界默认 + 强制轮转上限，到期整文件删除。
- **预期结果**：**不出现** "刷满磁盘 → 所有请求 fail-closed 全局瘫痪"的 Agent 可触发拒绝服务环。降级/聚合/轮转**本身写可见审计或告警**（口子自动愈合对运维可见）。高价值事件（`deny`/`policy_change`/`credential_event`）仍逐事件 fsync 不被降采样丢弃。诚实平衡：在 fail-closed 与可用性间，存储健康是平面状态、单条失败是请求事件，二者降级路径不同。

#### E4. 审计内容泄露探测（永不含凭据值与真实地址）

- **触发**：审计事件原本可能携带的字段里出现真实地址/凭据（如 mutate 的请求负载含 Cookie、connection_event 的 transport 配置含 host）。
- **系统行为**：审计"不含真实地址/凭据"的**真保证来自上游**——匿名化（真实地址不进 Agent 回路）+ 凭据零接触（凭据不进响应路径，公理四）；在此之上事件正文**写入前过同一 Sanitizer**（详设 5.3、6.4 作用面含"审计事件正文"）：系统级 ScrubSet 以**尽力而为黑名单**擦真实地址/IP/连接串/凭据值（不可枚举句柄，不承诺覆盖全部编码变体），`response_digest` 只存 sha256 摘要（**结构上不含内容**），`connection_event` 只记 resource/tier 名/transport 种类（不含真实地址/凭据，详设 5.3 kind 表、模块 06-daemon §3.5）。`intent_digest` 是**脱敏后**摘要（详设 5.3）。
- **预期结果**：`postern audit` 与 OTel 导出的任何字段中**均无** token/会话值/真实地址/账号密码——这是**结构性**结果（凭据/真实地址根本不进入审计可写字段），非仅靠输出端擦除；即便构造诱导回显的请求，审计与响应两条出口都经同一脱敏（公理四凭据零接触、技设 6.4 拒绝响应也脱敏）。诚实边界：高敏感字段优先靠结构化白名单输出而非全文黑名单（详设 6.4）。

#### E5. Scope 外探测经审计读不出存在性

- **触发**：分析者（人）想用 `postern audit` 反查"系统里有哪些资源"——但这是控制面、人有权看全；威胁在于 **Agent 侧**能否经任何路径推断 Scope 外资源存在。
- **系统行为**：Agent 对 audit/denials 端点**物理不可达**（`control.sock=0600`，Agent uid 不可读）；Agent 唯一拿到的是自身请求的 `DenyResponse`，其 `your_grants` 只含自身授权世界，`denied.objects` 经匿名化脱敏，**不泄露其他资源/角色/他人权限是否存在**（技设 6.4 安全边界；verify 项 4 主动校验此点）。
- **预期结果**：Agent 无法把审计或拒绝响应当拓扑/权限探测工具；verify 项 4 PASS 即证明 Scope 外访问被拒**且**不泄露存在性。

#### E6. verify 任一项未被拒（报告防线缺口）

- **触发**：跑 `postern verify` 时某项（如项 1 伪装写）**未**被拒——意味着某资源 `engine_enforced=false`（如旧版 redis 无 ACL）且归类被绕过，或 tier 声明超出账号真实权限（详设 6.3 子集校验缺失）。
- **系统行为**：该项返回 **FAIL**，verify 整体失败，并**指出具体防线缺口**（哪一项、本应卡在哪一步、为何漏放）。
- **预期结果**：运维明确看到"防线 X 有洞"（如"`db-main/ro` 实际有写权限，伪装写未被引擎拒——tier 声明 ⊄ 账号真实权限"），据此修复（收紧账号权限 / 升级 redis 到 6+ 获 ACL 兜底 / 补归类规则）。verify **不假装通过**——把真实强度如实暴露（公理三系统强制不靠口头、公理六只说事实）。这正是"主动验证而非假设"的价值：缺口在演练中暴露，而非在真实越权时。

#### E7. denials 聚合被滥用为放行依据（只呈现、不自动放行）

- **触发**：某主体反复撞同一堵墙，`denials` 榜首醒目——是否该自动放行以"减少噪音"？
- **系统行为**：网关**只呈现聚合模式 + 机械生成标准命令模板**（如 `postern elevate ...`），**绝不自动放行**（公理一默认拒绝、技设 6.5 人是立法者）。任何放行都须人经控制面显式写入并留 `policy_change` 审计。
- **预期结果**：反复 deny 不会"因为次数多"被系统自动转为 allow；调整方向是建议、放行是人的决定。fail-closed：不确定即维持拒绝。

#### E8. daemon 重启后审计连续性与 verify 复跑

- **触发**：`posternd` 重启（重启后从 `policy.db` 重建快照恢复策略状态、TTL 在求值时刻按墙钟二次校验继续计时——详设 6.2；挂起审批跨重启恒 deny——详设 6.10）。
- **系统行为**：审计是 append-only 文件载体（按 UTC 日轮转），重启**不丢历史**；`postern audit` 仍可查重启前事件（含上一 `policy_rev`）。重启后 `postern verify` 可复跑，对**当前**策略状态重新逐条验证。
- **预期结果**：审计跨重启连续可查；verify 反映重启后生效的策略（而非陈旧假设）；重启自身落 `lifecycle` 审计（启动/就绪/保险箱解锁阶段，详设 5.3 kind 表）。

#### E9. 放大后（信任域放大）的审计防篡改

- **触发**：信任域从单机放大到多节点 / 引入外部审计后端（技设公理五"边界可放大"）。
- **系统行为**：本地 JSONL 仍 append-only、`0600`、Agent 不可读、内容脱敏（不变量不退化）；放大档由更强实现承载防篡改——如审计写入替换为带防篡改的问责级实现、OTel 旁路导出到外部**不可变**后端、或外部 append-only 副本（详设 8.9：导出不可用不影响本地审计权威；技设 12.2 隔离点替换表"审计写入→带防篡改的问责级实现"、11.4 外部导出）。本地权威与外部副本**双写但不互为依赖**（本地先成功，导出旁路镜像、失败不回压）。
- **预期结果**：放大后审计的防篡改强度提升（外部不可变副本），但**本地权威性与 fail-closed 语义不退化**——外部副本不可用时本地照常记录、决策照常进行（公理二、三）。

#### E10. 据 denials 分析并发调整策略的乐观锁冲突

- **触发**：两名运维同时看到同一条 `denials` 榜首（如 `agent1 × db-main × rbac × mutate`），各自基于**同一次读取的 `version`** 对同一授权格发起调整（一人 `elevate`、一人改 `constraint`），两次写命中同一行。
- **系统行为**：控制面写端点强制携带期望 `version`（唯一来源是先前读取值），第二次写的 `version` 已陈旧 → 返回 **`409 Conflict`** 且**该冲突本身写 `policy_change` 审计**（记录冲突，详设 6.5 乐观锁端到端贯通）；不静默后写覆盖先写。审计/denials 查询是读动作，并发查询彼此无冲突（无锁读 `Arc<PolicySnapshot>`，详设 6.2）。
- **预期结果**：并发调整不互相覆盖——后写者收到 `409`，须重读最新 `version`（及最新 denials）再决策；不会出现"一次调整被另一次悄无声息抹掉"。系统协调写（sweeper 幂等回收过期 temp_grant）不走乐观锁、不与人写冲突。fail-closed：版本不匹配即拒绝写入，不猜测意图。

---

> **小结**：本场景把"事后可查（audit 分页）+ 持续收紧（denials 聚合 + 进化闭环）+ 主动验证（verify 逐条 PASS）"钉成验收基准。所有异常一律 fail-closed：审计不可记则不放行（E1）、有副作用动词两阶段且已执行绝不谎报 deny（E2）、盘满走配额降级而非全平面瘫痪（E3）、审计与响应双出口脱敏且永不含凭据/真实地址（E4）、Scope 外不泄露存在性（E5）、verify 任一项未被拒即报缺口绝不假装通过（E6）、反复 deny 绝不自动放行（E7）、跨重启审计连续且 verify 复跑反映当前策略（E8）、放大后本地权威不退化（E9）、并发调整版本不匹配即 `409` 不静默覆盖（E10）。七公理为最高准绳。
