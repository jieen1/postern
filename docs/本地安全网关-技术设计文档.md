# 本地安全网关 — 系统技术设计文档

> AI Agent 与基础设施之间的本地安全网关。本文档完整描述系统的技术设计:定位与公理、形态与进程、领域模型、架构分层、请求求值、权限管理、身份与凭证、连接管理、匿名化与脱敏、可扩展性、存储配置、安全模型,以及端到端流程。文档只描述"系统是什么样、如何运作",不涉及实现排布、工作量或阶段划分。

---

## 第一部分 · 定位与设计公理

### 1.1 一句话定义

本地安全网关是 AI Agent 与一切基础设施(服务器、数据库、容器、API、集群等)之间的**唯一通道**。Agent 一侧只能看到被授予的"能力";凭据、IP、实例标识、拓扑、协议细节全部存在于另一侧,对 Agent 不可见。所有访问由系统强制鉴权,默认拒绝,权限可动态调整,全程审计。

类比:**它是 Agent 的 IAM 加防火墙,运行在用户自己掌控的信任域内。**

### 1.2 要解决的根本问题

让 Agent 安全地操作真实基础设施,核心矛盾有四:

1. **凭据暴露** — Agent 的上下文会读取不可信内容(日志、网页、数据库内容),任何进入上下文的密钥都可能被提示注入窃取。凭据必须永不进入 Agent 可达范围。
2. **权限失控** — Agent 可能误用高权限操作,或被注入诱导执行破坏性动作。权限必须分级、精确、由系统强制,而非依赖 Agent 自觉或文档约定。
3. **信息泄露** — 服务器 IP、实例 ID、内网拓扑本身即敏感资产,也是攻击踏板。这些信息不应进入 Agent 视野。
4. **人不在环** — Agent 操作高频且常在无人值守时进行。安全机制不能依赖"人实时审批"才成立。

### 1.3 设计公理

以下公理贯穿全部设计。任何具体决策若与之冲突,以公理为准。

**公理一 · 默认拒绝(Deny by default)**
未被显式授予的操作一律拒绝。授权是白名单,不存在隐式许可。

**公理二 · 失败即关闭(Fail-closed)**
任何不确定状态——配置缺失、引擎异常、协议解析失败、凭证过期或无法验证、审批超时、连接不可建立、依赖不可用——一律解析为拒绝,绝不解析为放行。安全性在最坏假设下不退化。

**公理三 · 系统强制,而非约定(Enforcement, not convention)**
权限边界由系统在执行路径上强制。文档说明、对 Agent 的告知、命名约定都不构成边界。一个能力如果只靠 Agent 配合才安全,则视为该边界不存在。

**公理四 · 凭据零接触(Zero credential exposure)**
凭据、真实地址、拓扑标识永不离开网关进程,永不出现在任何返回给 Agent 的数据中——包括正常响应、错误信息、日志回传、拒绝响应。

**公理五 · 单一信任域,边界可放大(Single trust domain, expandable boundary)**
系统始终运行在用户掌控的信任域内(本机或组织内网),敏感数据永不交予第三方。核心模型按"信任域可能包含多用户、跨进程、跨网络"抽象,通过接口隔离可变实现;放大信任域时替换实现,不改核心。不提供公共云托管形态——将用户的密钥与生产访问能力托管于第三方云,与本产品的安全定位根本矛盾。

**公理六 · 网关说真话,且只说事实(Truthful, fact-only output)**
网关返回给 Agent 的一切信息,要么是策略与系统的客观事实,要么是资源所有者亲笔预写的内容。网关绝不代为生成建议、推测或话术。引导 Agent 行为的智能来自 Agent 对事实的推理,不来自网关编造的文案。

**公理七 · 入口对称,语义一致(Symmetric entry, uniform semantics)**
同一类操作无论经由哪种接入形态(MCP、HTTP、CLI、SDK)进入,都归一化为同一请求模型、走同一条求值与执行管线,得到完全一致的鉴权、匿名化、脱敏与审计。接入形态是外壳,安全语义在内核,外壳的差异绝不导致安全语义的差异。

### 1.4 非目标

明确不在本系统设计范围内:

- 公共云 SaaS 形态与第三方密钥托管(违背公理五)。
- 多组织多租户运营平台(单一信任域内可含多用户,但不是面向多租户运营的平台)。
- 替代基础设施自身的访问控制(网关是收敛与强制层;数据库账号权限、云 IAM 等底层控制仍是纵深防御的一部分,网关与之叠加而非取代)。
- Agent 自身能力的实现(网关授权 Agent 访问,不负责其推理与编排)。

## 第二部分 · 形态与进程模型

### 2.1 单核心,多前端,多入口

系统由一个核心 daemon 与若干客户端构成。核心是唯一承载逻辑的主体;客户端本身不含任何安全逻辑。核心对外有两个平面,各自挂载多种接入形态:

```
   ┌─ 数据面(Agent 与程序) ─┐        ┌─ 控制面(人类管理) ─┐
   MCP 外壳   HTTP 外壳   SDK         RPC ── CLI / 管理 SPA / 桌面壳
      │         │         │            │
      └────┬────┴─────────┘            └──────┬──────────────┘
           ▼                                  ▼
   ┌──────────────────────────────────────────────────────────┐
   │                    网关核心 daemon                         │
   │              (唯一逻辑主体,自给自足,headless)           │
   │                                                            │
   │   数据面请求内核 · 策略引擎 · 连接管理 · 传输 · 适配器     │
   │   身份与凭证 · 机密保险箱 · 匿名化映射 · 审计与观测        │
   └──────────────────────────────────────────────────────────┘
                                  │
                       任意 Transport(长/非长连接)
                                  ▼
                     Resources(服务器/DB/容器/API/...)
```

核心 daemon 是 headless 服务,不依赖任何图形环境,自身即可作为内网自部署服务端独立运行。

**数据面接入形态**(供 Agent 与程序发起操作,见第五部分):

- **MCP 外壳** — 面向支持 MCP 的 Agent,能力以语义化工具呈现。Agent 直接使用。
- **HTTP 外壳(REST/JSON)** — 面向脚本、程序化与复杂编排的通用入口。任何语言均可接入,承载 CLI/MCP 表达不便的复杂用法。
- **SDK** — HTTP 之上的语言便利封装,非独立协议。

**控制面接入形态**(供人类管理策略、配置资源、查看审计,经 RPC):

- **CLI** — 独立瘦客户端,经控制面 RPC 操作 daemon。
- **管理前端(SPA)** — 一套 Web 前端,经控制面 RPC 与 daemon 通信;可被桌面外壳的 webview 加载,也可由 daemon 直接经 HTTP 提供给浏览器。
- **桌面外壳(可选)** — 在有桌面环境时,将 daemon 与管理前端封装为原生桌面应用,额外提供系统托盘、原生通知、操作系统密钥链桥接、自动更新。外壳不含安全逻辑,仅是 daemon 的宿主与 SPA 的容器。

无论数据面还是控制面,所有外壳都共享其平面背后的同一内核(公理七)。

### 2.2 核心 daemon 的硬约束

为保证单核心可同时服务桌面与自部署形态,并保证信任域可放大,daemon 必须满足:

**约束一 · UI 无关** — daemon 不依赖任何特定前端技术,不假设前端存在或在前台。关闭所有前端,daemon 照常运行(连接保活、Agent 服务、策略执行不中断)。

**约束二 · 传输无关(双向)** — 对前端的控制面接口是与传输方式解耦的 RPC 抽象(本地绑定到进程间通信,放大时增加网络绑定);对资源的连接也是与协议、与长短连接解耦的抽象(见第八部分)。核心逻辑既不感知前端来自本地还是网络,也不感知资源经由何种隧道、以长连接还是非长连接抵达。

**约束三 · 自给自足** — daemon 不依赖任何外部托管服务即可提供全部功能。所有状态、凭据、审计、策略均落本地,无云依赖、无外部数据库强制要求。

### 2.3 控制面与数据面隔离

两平面物理隔离,是系统自我保护的基石:

- **数据面** — Agent 与程序发起操作的通道(MCP / HTTP)。Agent 只能触达数据面。每个数据面请求都经过完整求值管线。
- **控制面** — 人类管理的通道(RPC)。默认绑定到进程间通信(非网络端口),Agent 物理上无法触达;即便 Agent 能发起任意本地网络请求,也到不了一个不监听网络端口的控制面。

隔离的推论:

- 凭据保险箱与匿名化映射仅经控制面/机密面内部可达,数据面对其无任何读写路径。策略状态与审计:数据面对二者**无任何写路径**;对策略状态的唯一读形态是经求值内核中介的**只读原子快照**(PolicySnapshot,非直接库访问),且面向 Agent 的输出只暴露该 Principal 自身的授权世界(your_grants / 拒绝响应受 Scope 约束,契约 DENY_RESPONSE_SCOPE_BOUNDED)。快照内含什么、各域可见哪个子集,见详细设计 8.11 的 PolicySnapshot 内容清单。
- `admin` 级能力(修改网关自身配置)只存在于控制面,且在模型层面不可授予任何 Agent Principal(见 3.3)。
- 凭证的签发、轮换、吊销只在控制面进行;数据面只能"出示已签发的凭证",不能签发或修改凭证。
- 信任域放大时,控制面增加网络绑定并启用相应的控制面认证(见第七部分),但两平面的隔离关系不变。

### 2.4 进程生命周期

daemon 独立于任何前端的生命周期。桌面外壳关闭窗口时,daemon 退至后台继续运行(连接与 Agent 服务不中断);只有显式停止 daemon 才终止服务。自部署形态下,daemon 作为系统服务常驻。daemon 自身的启动需完成:打开权威策略状态存储(角色、绑定、凭证元数据、生效中的临时授权及其剩余 TTL、当前模式等随之恢复并继续计时)、解锁机密保险箱、就绪后方开放数据面。

## 第三部分 · 核心领域模型

系统的全部行为建立在五个概念之上:四个名词(参与者与对象)与一个动词集(权限原子)。

### 3.1 Principal(主体)

发起请求者的身份。每个 Agent 实例、每个程序化调用方、每个人类用户(信任域放大后)都是一个独立 Principal。

- Principal 是授权与审计的基本单位:授权按 Principal 计算,审计按 Principal 归属。
- Principal 经认证确立身份(认证方式可替换且多样,见第七部分),核心逻辑只接收"已认证的 Principal",不关心认证过程。
- 不存在全局匿名或共享身份。即便当前只有单一用户的单一 Agent,它也是一个具名 Principal,而非隐式的全局主体。这保证信任域放大时,新增用户、程序与 Agent 只是新增 Principal,授权与审计模型无需改动。
- Principal 与其凭证分离:Principal 是身份本身,凭证是证明该身份的手段。一个 Principal 可持有多份凭证(便于轮换),凭证可被吊销而 Principal 仍存在(见第七部分)。
- 凭证可附带**可选的可信域约束**:某份凭证只在请求来自约定来源(IP、IP 段、网络/域)时有效。约束绑定在凭证而非 Principal 上,故同一 Principal 的不同凭证可有不同可信域(例如给 CI 的凭证锁定在 CI 出口 IP,给本地的凭证锁定在 localhost)。该约束可选,不设则该凭证不受来源限制(见第七部分)。

### 3.2 Resource(资源)

被纳入网关管理的目标:一台服务器、一个数据库、一个容器运行时、一个 HTTP 服务、一个集群等。

- 每个 Resource 声明其**接入方式**(经哪个 Transport 到达)、**能力面**(由哪个 Adapter 解析、暴露哪些 Capability)与**凭据等级集合**(每个等级承载一组动词、对应一份资源凭据,见 10.5)。
- **同一物理目标可对应多个 Resource**:因不同操作用不同 Adapter/Transport/凭据。例如一个业务系统:"**用**它"(经业务系统账号调其 HTTP 接口/页面)是一个 http 适配器资源、"**管**它"(重启/扩缩/部署)是一个 docker/command 适配器资源、"看它的日志"是 docker 适配器的只读 observe——三个资源各自独立授权,精确正交。详见详细设计 5.2bis、6.13。
- **资源对 Agent 匿名化**:Agent 只能看到资源代号(如 `db-main`、`app-prod`)。真实地址、IP、实例 ID、region、端口、域名等全部由网关在内部"代号 ↔ 真实地址映射表"中持有,绝不出现在数据面的任何请求或响应中。
- 匿名化双向:Agent 不能以真实地址发起请求,也不能从返回内容中提取真实地址(响应脱敏保证,见第九部分)。
- Resource 可带**标签**(如 `host:A`/`env:prod`/`kind:business`),供 Scope 按标签选择器成组授权(见 3.3、详细设计 5.2bis)。

### 3.3 Role(角色)与 Scope(作用域)

授权的组织方式,采用两个正交维度。

**Role(角色)** 表达**信任等级**——一组 Capability 的集合,描述"被信任做哪些种类的动作"。

- 角色可继承:高信任角色继承低信任角色的全部授予并增量添加。推荐内置阶梯(经继承):`observer`(observe+query,只读一切)⊂ `operator`(+mutate+execute,改数据/跑受控模板)⊂ `maintainer`(+manage,启停/部署)。动词含义由各 Adapter 解释(`observer` 给 DB 是 SELECT、给 redis 是 GET、给 HTTP 是 GET),故一个角色复用于所有资源类型。`destroy` 不进标准角色,按单格 + TTL 显式授予。详见详细设计 5.2bis。
- 角色与具体资源无关,只描述动作集合,因而可在不同资源、不同 Principal 间复用。差异化不靠"为每种资源类型造角色"(组合爆炸),而靠 Scope + 细则 + 凭据等级。
- 保留角色 `admin` 代表"修改网关自身配置的能力",在模型层面硬约束为**不可授予任何 Principal**——它只存在于控制面的人类操作中,不是可分配角色。这是公理三在角色层面的固化。

**Scope(作用域)** 表达**辖区**——该 Principal 的授权作用于哪些 Resource 的集合。

- 与 Role 正交:Role 决定"能做哪类动作",Scope 决定"对哪些资源生效"。
- 正交性使同一角色可授予多个 Principal 而各自辖区不同,无需为每个 Principal 定义专属角色。
- Scope 可**枚举资源代号**,也可用**标签选择器**(如 `env:prod AND kind:business`)成组选取——后者给出"某机器全部权限/某环境/某类资源"的分组授权,新增同类资源打对标签即自动纳入,无需改授权(见详细设计 5.2bis)。

**授权展开**:`Principal —绑定→ (Role × Scope)`,求值时展开为判定空间 `Resource × Capability × 细则`。某次请求被允许,当且仅当其 (Resource, Capability) 落在展开后的授权空间内,且满足该格子上的细则与条件。

### 3.4 Capability(能力)— 权限原子动词

权限的最小单位是动词,而非粗粒度读/写。动词之间正交,组合表达任意等级。

| 动词 | 含义 | 典型操作 |
|---|---|---|
| `observe` | 观察运行状态 | 读日志、看指标、查容器状态、看健康 |
| `query` | 读取数据 | 数据库 SELECT、HTTP GET、读取文件内容 |
| `mutate` | 修改数据 | 数据库 UPDATE/INSERT、HTTP POST/PUT |
| `execute` | 执行受控动作 | 运行命令模板、触发任务、调用脚本 |
| `manage` | 生命周期管理 | 重启/启停服务、扩缩容、变更配置 |
| `destroy` | 破坏性操作 | 数据库 DROP/DELETE、删除资源、销毁实例 |

(`admin` 不属于此集合——它是"权限的权限",见 3.3,不可授予。)

设计要点:

- 动词**正交**:`observe` 不蕴含 `query`,`mutate` 不蕴含 `destroy`。授予是精确的逐项授予,不存在"给了高的自动包含低的"的隐式扩张(角色继承是显式组合,不是隐式蕴含)。
- 每个 (Resource × Capability) 格子可附加**细则(constraints)**,将动词进一步约束到具体对象:数据库的表/schema/列白黑名单、容器名白名单、命令的参数化模板白名单、HTTP 的方法+路径白名单等。
- 细则是 Capability 的一部分,与动词共同构成一个完整授权格。无细则表示该动词对该资源全部对象生效,有细则则收窄到指定对象。

### 3.5 模型关系总览

```
Principal ──持有──▶ Credential(s)(可多份,可轮换/吊销,可附可选可信域;见第七部分)
    │
    ├──绑定──▶ Role(可继承,含 Capability 集合)
    │            ×
    └─────────▶ Scope(Resource 集合)
                 │
                 ▼
   展开为授权空间:Resource × Capability × 细则
                 │
  Resource ──声明──▶ Transport(如何到达,长/非长连接)
                 ├──▶ Adapter(如何解析能力面)
                 └──▶ Credential Tier 集合(每个等级承载一组动词→一份资源凭据,见 10.5)
                 │
                 ▼
         真实地址(仅存于网关内部映射表,对 Agent 不可见)
```

## 第四部分 · 系统架构分层

daemon 内部分为若干面,每面职责单一,面间通过明确接口交互。

```
┌────────────────────────────────────────────────────────────┐
│  数据面入口(MCP / HTTP / SDK)   控制面入口(RPC)          │
├──────────────────────────────────┬─────────────────────────┤
│        ① 数据面请求内核           │   ② 控制面 / 策略引擎    │
│  归一化→求值→执行→脱敏(统一管线) │  RBAC求值·细则·条件·     │
│                                   │  拒绝协议组装·热更新·    │
│                                   │  模式·临时授权(TTL)·   │
│                                   │  凭证签发/轮换/吊销      │
├──────────────────────────────────┴─────────────────────────┤
│  ③ 身份与凭证(Identity)                                    │
│  认证器族(本地身份/APIKey/Token/账密/mTLS/SSO)·            │
│  凭证生命周期(签发·过期·轮换·吊销·审计)                   │
├──────────────────────────────────────────────────────────────┤
│  ④ 连接管理(Connection Pool & Lifecycle)                   │
│  池化复用·健康检查·并发上限·空闲回收·优雅销毁              │
├──────────────────────────────────────────────────────────────┤
│  ⑤ 适配器层(Adapter)        ⑥ 传输层(Transport)         │
│  协议感知:解析动词与对象·     远端接入抽象为本地 socket:   │
│  执行·暴露能力面·细则语义     长连接/非长连接·建立·保活·   │
│  (postgres/mysql/docker/      重连·关闭                     │
│   http/redis/k8s/...)         (SSM/SSH/Tailscale/CF/直连)  │
├──────────────────────────────────────────────────────────────┤
│  ⑦ 机密面(Secrets)          ⑧ 观测面(Observability)     │
│  凭据来源(静态/动态/证书)·   审计流(append-only)·         │
│  代号↔真实地址映射            拒绝分析·告警·可选外部导出     │
├──────────────────────────────────────────────────────────────┤
│  存储:策略状态(权威,嵌入式库)· 审计(append-only 事件流)   │
│        · 机密(隔离持有);声明式文件为可选导入/导出表示          │
└────────────────────────────────────────────────────────────┘
```

> **领域边界的权威定义**:本图给出各面的概略职责;各领域的封闭边界(内涵、范围内/外、对外接口、必守不变量、概念归属裁决)以**详细设计文档·"领域范围与内涵"章**为准。本图与该章冲突时以该章为准。该章把下文中"② 控制面 / 策略引擎"进一步拆为两个独立领域(纯求值的策略引擎 vs 唯一变更入口的控制面),并裁决脱敏、连接重连、凭证生命周期、凭据等级等横跨多面的概念归属。

各面职责简述(详见后续各部分):

- **① 数据面请求内核** — 把任意外壳进来的请求归一化为统一请求模型,串起求值、执行、脱敏。所有数据面外壳共享它(公理七)。
- **② 控制面 / 策略引擎** — 两个独立领域(详细设计领域章拆分):**策略引擎**对每个请求求值产出 allow/deny/escalate 并组装拒绝响应(纯求值、零副作用、只读快照);**控制面**是策略状态的唯一变更入口(人类 RPC 与系统自动机),写入热生效。凭证的签发/轮换/吊销是控制面的**操作入口**,其语义规则归 ③ 身份与凭证。
- **③ 身份与凭证** — 确立 Principal 身份(多种认证方式)并定义凭证的完整生命周期语义(有效期、可信域、吊销的判定规则)。
- **④ 连接管理** — 在传输层之上治理通路的池化与生命周期。
- **⑤ 适配器层** — 协议感知:把协议操作映射为动词与对象,并执行。插件化。
- **⑥ 传输层** — 把远端接入抽象为本地 socket,涵盖长/非长连接。插件化。
- **⑦ 机密面** — 资源凭据与真实地址的唯一持有者;凭据经可替换的 Credential Provider 提供(静态保管或动态签发)。
- **⑧ 观测面** — 审计(本地 append-only,自给自足)、拒绝分析、告警;审计可经可选导出器对齐开放遥测规范向外部投影。

## 第五部分 · 请求求值

求值是系统的核心算法。每个数据面请求,无论经哪个外壳进入,都在归一化后完整走过下述管线;任一环节判定拒绝即短路,进入拒绝响应组装。

### 5.1 求值管线

```
任意外壳进入的请求(MCP / HTTP / SDK)
   │
   ▼
[0] 外壳归一化 ── 将外壳特定的请求形态转为统一请求模型
                 (Principal 凭证, 资源代号, 意图)
   │
   ▼
[1] 认证 ── 验证凭证有效性 → 校验来源可信域 → 确立 Principal 身份
            凭证无效/过期/已吊销/无法验证 → deny(fail-closed)
            凭证设有可信域且请求来源不在其内,或来源无法可靠判定
              → deny(fail-closed)
            来源取自网关可观测的连接来源(传输层对端地址 / 本地进程上下文),
              不采信请求自报字段(公理三)
   │
   ▼
[2] 语义归一化 ── 由对应 Adapter 解析意图,提取三元组:
                 (Resource, Capability 动词, 操作对象)
                 协议感知;解析失败/无法归类 → deny(公理二)
   │
   ▼
[3] RBAC 判定 ── 展开 Principal 的 (Role × Scope):
                该 (Resource, Capability) 格子是否存在?
                不存在 → deny(公理一)
   │
   ▼
[4] 细则校验 ── 操作对象是否满足该格子的细则?
              (表白名单 / 容器前缀 / 命令模板 / 路径规则)
              协议感知,非字符串匹配;不满足 → deny
   │
   ▼
[5] 条件层 ── 逐一求值授权格上附加的条件谓词(可扩展集合):
            限流额度、时间窗、当前 Mode、临时授权 TTL 有效性 等
            任一谓词不满足 → deny
   │
   ▼
[6] 动作分流 ── 该授权格标注的动作类型:
              allow    → 进入执行
              escalate → 审批开?挂起等待(超时→deny);审批关?→ deny
   │
   ▼ (allow)
[7] 获取连接 ── 按本次动词选凭据等级(tier,见 10.5),向连接管理层请求
              "到 (该资源, 该 tier) 的可用通路"(池中复用或新建;
              无承载该动词的 tier → deny;无法建立 → deny,fail-closed)
   │
   ▼
[8] 执行 ── Adapter 经该通路对资源执行操作
   │
   ▼
[9] 响应脱敏 ── 擦除返回内容中的真实地址/凭据/敏感字段(第九部分)
   │
   ▼
[10] 审计 ── 记录请求三元组、决策、决策依据、所用凭证标识、响应摘要
   │
   ▼
返回(经原外壳格式化)
```

### 5.2 关键判定原则

- **语义归一化必须协议感知(步骤 2)**:判定一个请求属于哪个动词、作用于哪些对象,必须由理解协议语义的适配器完成,而非对请求文本做模式匹配。对 SQL,这意味着语法树解析:必须识破在查询语法外壳下隐藏的写操作(例如以公共表表达式包裹的删除、以函数调用产生的副作用),正确归类为 `mutate` / `destroy`。无法可靠归类的请求一律拒绝。

- **入口对称(步骤 0 与全管线)**:步骤[0]之后,请求已与外壳无关;[1]~[10] 对 MCP、HTTP、SDK 进来的请求完全相同。安全语义由内核保证,不因入口而异(公理七)。

- **拒绝同样进入审计(步骤 10)**:被拒请求是最有分析价值的信号——揭示 Agent 真实意图与预设的偏差,以及潜在注入或失控。deny 记录附完整拒绝依据(在哪一步、因何拒绝)。

- **条件层是可扩展的谓词集合(步骤 5)**:条件层不是写死的几项检查,而是一组正交的条件谓词,授权格可按需附加其中若干。内置谓词包括限流额度、时间窗、当前 Mode、临时授权 TTL 有效性;凭证可信域(第七部分)在判定上也是同一类谓词(基于可观测来源的约束),只是发生在认证阶段。新增一种条件维度即新增一个谓词,不改求值骨架。所有谓词的求值都是 fail-closed:无法判定即视为不满足。

- **决策确定性**:相同请求 + 相同策略状态 → 相同决策。求值无随机性、无隐式上下文依赖,保证可审计、可解释、可复现。

### 5.3 决策三值与审批地位

决策结果有三值:`allow` / `deny` / `escalate`。

`escalate` 表示该操作被标注为需人工确认。其行为由审批开关决定:

- **审批关闭(默认)** — `escalate` 等价于 `deny`,附"此操作需授权,当前无人值守"的原因。请求不挂起,Agent 立即得到确定拒绝。
- **审批开启** — 请求挂起,推送确认给人;超时未批按 `on_timeout` 处理,而 `on_timeout` 恒为 `deny`(公理二,绝不超时放行)。

审批是"人在场模式"下的可选便利,不是权限模型的承重结构(见第六部分)。系统在"人永不响应"的极端假设下安全性不退化——这正是把 escalate 默认折叠为 deny 的意义。

## 第六部分 · 权限管理与人机协作

核心理念:

> **预设决定一切,拒绝引导一切,人只修订预设。**

### 6.1 预设为主:授权是配置,不是逐次批准

绝大多数请求由预设(角色 + 作用域 + 细则)即时、确定地裁决,毫秒级,完全不依赖人在场。系统的严格性来自预设的精确,而非人的实时盯防。

人对权限的介入方式是**修订规则**(持久、异步、不阻塞),而非**批准请求**(单次、同步、阻塞)。前者把人放在"立法者"位置(偶尔调整规则),后者把人放在"看门人"位置(逐次放行)。本系统选择前者。

### 6.2 动态控制

策略不是静态文件,而是系统管理的权威状态(见第十一部分)。一切权限调整——无论经 UI、CLI、RPC 由人发起,还是由 TTL 到期、凭证过期等由系统自动触发——都是对策略状态存储的一次**事务性写入**,写入即时对后续求值生效(这正是"热更新"的含义:不重启、不中断连接),且每次写入进审计。求值永远只面对单一、统一、权威的策略状态,不需要合并多个来源。

具体可动态调整的内容:

- **角色与绑定修改** — 改变某 Principal 的角色、调整角色定义、增删作用域,后续请求立即按新策略裁决。
- **凭证管理** — 签发、修改(含可信域约束)、轮换、吊销凭证,均经控制面写入状态存储,热生效(见第七部分)。
- **临时授权(TTL)** — 对某 Principal 临时提升某项能力,**默认携带有效期**,到期由系统自动写入回收。永久变更是显式例外。意义:安全口子自动愈合,杜绝"临时放开后忘记收回"。临时授权及其剩余 TTL 是状态存储的一部分,daemon 重启后恢复并继续计时。
- **模式(Mode)** — 预设的权限组合,可一键切换并可附 TTL:
  - 观察模式:仅 `observe` + `query`,Agent 自由排查,无写入。
  - 维护模式:在观察基础上放开受控的 `mutate` / `execute`,用于有人值守的变更。
  - 冻结模式:全部拒绝,一键拉闸,用于发现异常时立即切断 Agent 的一切操作。
  - 模式作用于全局或单个资源;附 TTL 的模式到期自动回退。
- **审批开关** — 全局开启/关闭。关闭时 escalate 即 deny(见 5.3)。

切断失控 Agent 有两种粒度,可叠加:**吊销其凭证**(精确到身份,见第七部分)或**冻结模式**(精确到操作)。

### 6.3 拒绝之后:三级自动分流

拒绝不是终点,而是网关与 Agent 之间的一次结构化对话。多数"越权"无需人介入:

```
请求被拒 + 结构化拒绝响应
   │
   ├─ ① 误用(本欲查询却发出修改)
   │     Agent 读 reason 理解边界 → 自我修正 → 重试           【人无感知】
   │
   ├─ ② 存在低权限替代路径
   │     Agent 依 your_grants 推理出可行替代方法 → 完成任务   【人无感知】
   │
   └─ ③ 确需更高权限
         Agent 向用户陈述:要做什么、为什么、需要何种权限,
         并附上现成的提升命令 → 人异步看到,一条命令授权
         (默认带 TTL)                                       【人粘贴一次】
```

第③级中,传统同步"审批流"被"Agent 主动申请 + 人异步授权"替代:人无需在场守候,Agent 把申请置于对话中,人方便时处理,处理方式是执行一条提升命令。

### 6.4 结构化拒绝响应

拒绝响应是给 Agent 消费的协议,必须可推理、可行动,且严格遵守公理六(只含事实或人亲笔预写内容):

```
{
  decision:     "deny"
  denied:       { resource, capability, object }   // 已匿名化、已脱敏
  reason:       "<为何被拒,引用策略事实>"           // 如:role=observer 不含 db-main:mutate
  your_grants:  { <该 Principal 在相关资源上的现有授权> }  // 事实:你还能做什么
  request_hint: "<申请该权限的标准命令>"             // 由策略机械生成
  operator_note:"<可选:资源所有者在配置中亲笔预写的指引>"  // 无则不出现
}
```

- `reason` — 让 Agent 理解边界所在,避免盲目重试。引用客观策略事实。
- `your_grants` — Agent 自我修正与寻找替代路径的依据。这是"动态建议"的真正来源:网关只陈述 Agent 拥有什么,由 Agent 自行推理可行路径,网关不编造建议文案。
- `request_hint` — 申请缺失权限的标准命令,从策略状态机械生成。
- `operator_note` — 仅当资源所有者为该 (资源, 动词) 预写指引时出现,原样转述。

**安全边界**:拒绝响应只描述该 Principal 自身的授权世界,绝不透露系统中存在哪些其他资源、角色或他人权限,不能成为拓扑/权限探测工具。`denied.object` 等字段在返回前同样经匿名化与脱敏。

### 6.5 拒绝分析:预设进化闭环

被拒记录持续喂养策略演进:系统对 deny 记录聚合,呈现"某 Principal 反复请求某类被拒操作"的模式,据此提示可能的调整方向。人据此修订预设,角色稳定在"立法者"——偶尔审视差距、调整规则,而非逐次放行。

此闭环与 Agent 能力正向共生:Agent 越能干,①②两级自动消化的比例越高,人被打扰越少。这与同步审批制相反——后者在 Agent 操作越多时令人被打扰越频繁。

## 第七部分 · 身份与凭证

本部分回答"网关如何确认谁在请求",并管理凭证的完整生命周期。Principal 是身份(第三部分),凭证是证明该身份的手段,二者分离。

### 7.1 认证器族:统一抽象下的多种方式

核心逻辑只接收"已认证的 Principal"。"如何认证"被隔离为一个可替换的**认证器(Authenticator)抽象**——求值管线步骤[1]调用它确立身份,不关心内部机制。数据面与控制面共用同一认证器族。已知方式:

| 方式 | 适用 | 特点 |
|---|---|---|
| **本地进程身份** | 本地形态的 Agent | 经 IPC/本地端口连接,身份由操作系统进程上下文确立;Agent 零配置、不持有任何凭证 |
| **API Key** | 脚本、CI、长期运行的程序 | 长期静态密钥;简单,依赖轮换与吊销兜底 |
| **Token(短期令牌)** | 会话化的 Agent 接入 | 有限有效期,过期即失效;缩小泄露暴露窗口 |
| **账号密码** | 放大后人类用户交互登录 | 通常换取短期 Token 后使用,密码不在每次请求中流转 |
| **mTLS(双向证书)** | 放大后机器/服务强身份 | 证书即身份;适合内网机器间 |
| **外部身份(SSO/OIDC)** | 放大后接入组织身份体系 | 人类 Principal 由外部 IdP 确立 |

设计原则:

- **本地默认零凭证**:本地形态下,Agent 经沙箱进程上下文连接 daemon,身份即确立,无需配置 API Key 或 Token。这是"执行命令免账号"的落点——Agent 既不持有资源账号(公理四),在本地也不必持有网关凭证。
- **方式可替换、可并存**:不同 Principal 可用不同方式(本地 Agent 用进程身份、CI 用 API Key、人类用户用 SSO),统一在认证器族下,核心逻辑无差别。
- **认证只确立身份,不赋予权限**:认证回答"你是谁",授权(第三、五部分)回答"你能做什么"。一个通过认证的 Principal 若无对应授权,照样被拒(公理一)。
- **可信域是认证的前置门**:凭证可附带可选的可信域约束(允许使用该凭证的来源 IP/IP 段/网络/域)。校验发生在认证阶段——验证凭证有效性之后、确立 Principal 之前(求值步骤[1]):请求来源不在凭证可信域内,凭证即视为不可用,认证失败。它属于身份层(回答"这份凭证此刻能不能用"),不属于权限层。来源以网关可观测的连接来源为准(网络连接取传输层对端地址,本地连接取进程上下文),绝不采信请求自报字段(公理三);无法可靠判定来源时按拒绝处理(公理二)。放大到内网 mTLS 时,可信域与证书身份叠加,构成"证书 + 来源网络"双重约束。

### 7.2 凭证生命周期

凭证有完整生命周期,全部在控制面进行(数据面只能出示凭证,不能签发或修改)。凭证元数据(绑定的 Principal、有效期、可信域、吊销状态)是策略状态的一部分,存于权威状态存储(见第十一部分),其变更与策略变更同机制——事务写入、热生效、进审计:

- **签发** — 凭证由控制面签发,绑定到某 Principal,可附带有效期、用途约束与**可选的可信域约束**(允许使用来源)。签发即记审计。
- **可信域设定与修改** — 凭证的可信域可在签发时设定,也可后续修改;修改热生效,后续请求按新可信域校验。不设可信域的凭证不受来源限制。
- **过期** — 凭证可带有效期;过期凭证在认证步骤[1]即被拒(等同未认证,fail-closed)。
- **轮换** — 支持**重叠期轮换**:新凭证签发后、旧凭证失效前存在并存窗口,使在用的 Agent/脚本无中断切换;窗口结束旧凭证自动失效。轮换不要求停机。新旧凭证可各自携带可信域。
- **吊销** — 凭证可即时吊销,**热生效**:吊销后该凭证的后续请求立即在步骤[1]被拒。吊销精确到单份凭证,不影响该 Principal 的其他有效凭证,也不删除 Principal 本身。
- **审计** — 凭证的签发、可信域修改、轮换、吊销均进审计,与策略变更同等对待。

### 7.3 凭证与资源凭据的区别

须区分两类不同的"凭据",二者无交集:

- **Principal 凭证(本部分)** — Agent/程序向**网关**证明自己身份的手段(API Key、Token、证书等)。它进入 daemon 用于认证,但不用于访问任何资源。
- **资源凭据(第十部分,机密面)** — 网关向**资源**证明访问权的密钥(数据库密码、SSH key、API token)。它只在传输层内部使用,Agent 永不持有、永不接触(公理四)。

Agent 即便持有自己的网关凭证,也无从获得任何资源凭据;网关凭证的作用仅是"让网关知道是谁",而非"让 Agent 登录资源"。

## 第八部分 · 连接与传输

本部分描述网关如何到达资源,以及如何治理通往资源的连接。分两层:**传输层**(单条通路的接入抽象)与**连接管理层**(通路的池化与生命周期治理)。上层(适配器)只向连接管理层请求"一个到某资源的可用通路",不感知通路背后的协议、隧道形态,也不感知它是新建还是复用。

### 8.1 传输层:把远端接入抽象为本地 socket

职责:把一种远端接入方式抽象为"在本地提供一个可用 socket"。传输是插件(第十一部分),已知形态:SSM 端口转发、SSH 隧道、Tailscale、Cloudflare Tunnel、直连等。

单条通路的统一契约(概念层面)。明确"传输只管单条通路,通路间生命周期决策归连接管理层"的分界(见 8.2、详细设计第八部分 8.5/8.7,决策者-执行者分离总则见详细设计 8.0):

- **建立** — 由**连接管理层**在建立流程中从机密面取出该资源的真实地址与所需资源凭据,以**不透明句柄**(不可读取明文、不可复制/序列化、生命周期不超出本次建立调用)一次性传入;传输插件据此建到远端、在本地暴露 socket。传输插件**不自取**机密、也不持久持有(凭据取用方向以此为准,与 10.6、详细设计 4.1/6.3 一致)。
- **通路内保活** — 维持**已建**通路的活性:心跳、对有时限通路的协议级续约(如 SSM 会话续期)。传输只做"单条通路的协议级保活",**不自行重建**——通路死亡时如实上报,是否/何时重建是连接管理层的决策(见 8.2)。
- **健康** — 如实报告当前通路状态(供连接管理层决策)。
- **关闭** — 按连接管理层的指令释放通路(决策在连接管理层,执行在传输)。

**长连接与非长连接**:传输可以是**持续通路**(长连接,如 SSH 隧道、Tailscale,建立后保活、可复用)或**按需通路**(非长连接,用毕即释放或每次操作短连接)。二者对上层均呈现一致的"本地 socket",上层不感知差异——这是"万物收敛于本地 socket,传输是插件"的设计落点。资源凭据与真实地址仅在传输插件的 open 调用生命周期内使用,不向上传递、不向 Agent 暴露。

### 8.2 连接管理层:池化与生命周期治理

传输层解决"一条通路怎么建",连接管理层解决"许多通路怎么管"。多个 Agent/程序可能并发访问同一资源;建立成本高的隧道(SSM/SSH)频繁建销既慢又耗;长连接若无人治理会泄漏或僵死。因此在传输层与适配器之间设连接管理层。

职责:

- **池化与复用** — 对长连接型传输,通路被池化,多个请求共享复用,避免每次重建。池以资源代号为键。
- **建立策略** — 按需建立(首次访问某资源时)或预热;支持每资源的最小/最大通路数。
- **健康检查与指数退避重连** — 定期探活,剔除僵死通路;失败通路触发重建。重连采用**指数退避**(重试间隔随连续失败递增,而非固定间隔):偶发断开能快速恢复,持续故障时退避拉长重试间隔,避免在对端已不可用时形成重连风暴、刷爆日志、加剧对端压力;对端恢复后回到正常频率。重建期间到来的请求等待新通路或被拒(fail-closed)。
- **并发上限与背压** — 每资源及全局的并发上限;超限时排队或拒绝,拒绝走 fail-closed,不静默放行。
- **空闲回收** — 超过空闲时限的通路被回收,释放底层隧道与资源凭据引用。
- **优雅销毁** — 通路关闭(空闲回收、健康剔除、资源下线、daemon 停止)时排空在途请求后再释放底层隧道。**紧急切断例外**:freeze/吊销触发的切断不走优雅排空(那会等待在途完成、延长切断),而是对相关在用通路强制 abort/cancel(取消底层查询、关闭隧道)——切断语义的诚实界定见详细设计 6.2 与 8.5。
- **长/非长连接的差异处理** — 长连接型进池复用;非长连接型用毕即销毁、不入池。这一差异对上层透明:适配器只请求"一个可用通路",由连接管理层依传输类型决定复用还是新建。
- **归池前会话净化** — 长连接型通路归池复用前必须重置会话态(如 PostgreSQL `DISCARD ALL`、重置 search_path、回滚未决事务),防止前一请求的会话副作用泄漏给复用者(详细设计 6.3 归池前会话净化)。

决策与执行分离:连接管理层是"通路间生命周期"的**决策者**(是否重建、指数退避节奏、健康剔除、空闲回收、优雅/紧急销毁的发起),传输层是"单条通路"的**执行者**(建立、保活、按指令关闭)。"重连"只发生在连接管理层(8.1 不含自行重连)。

与匿名化、凭据的关系:池中通路以资源代号(及凭据等级,见 10.5)为键;真实地址与资源凭据仍只在传输插件 open 调用内,连接管理层只搬运不透明句柄、不读取明文,适配器更不经手地址或凭据。

### 8.3 上层无感知原则

适配器执行操作时(求值步骤[7]~[8]),向连接管理层请求一个到目标资源代号的通路,得到一个可用 socket 即执行。它不知道:该通路是长连接还是非长连接、是池中复用还是新建、底层是 SSM 还是 SSH、真实地址是什么。这层无感知使得新增传输方式、调整池化策略都不触动适配器与求值逻辑。

## 第九部分 · 匿名化与脱敏

匿名化(Agent 看不到资源真实身份)与脱敏(Agent 拿不到返回内容中的敏感信息)共同实现公理四,一在请求侧、一在响应侧,缺一不可。

### 9.1 资源匿名化(请求侧)

- Agent 在数据面只能以资源代号发起请求,代号无任何真实地址含义。
- 代号到真实地址的解析只发生在传输层内部,由网关持有的映射表完成。映射表存于机密面,数据面无读取路径。
- Agent 无法枚举资源:只能看到自身 Scope 内被授予的资源代号,无法得知信任域内是否存在其他资源。资源列表类请求同样受 Scope 约束。

### 9.2 响应脱敏(响应侧)

执行结果在返回 Agent 前经脱敏,擦除敏感信息(返回内容尤其日志、错误、数据库内容可能无意携带敏感数据):

- **真实地址类** — 内网 IP、主机名、实例 ID、内部域名、连接串。泄露即破坏匿名化,且是攻击踏板。
- **凭据类** — 日志或配置回显中出现的密钥、令牌、密码。
- **业务敏感字段** — 资源所有者在细则中声明的敏感列/字段(如个人身份信息),按规则擦除或掩码。

脱敏规则分两类:**系统级**(地址与凭据类,默认始终生效,内置识别)与**声明级**(业务字段,由资源细则配置)。脱敏作用于一切返回 Agent 的内容,包括正常响应、错误信息与拒绝响应中的对象描述。

### 9.3 双向闭合

请求侧 Agent 无法指定真实地址,响应侧脱敏尽力擦除回显中的真实地址。两侧合拢构成纵深防御。

**诚实的强弱分层**(避免把脱敏当成绝对保证):

- **真正的硬保证来自请求侧的匿名化与凭据零接触**:Agent 只能以资源代号发起请求,代号↔真实地址映射只存于机密面、数据面无读取路径;资源凭据永不进入 Agent 可达范围。因此 Agent 在结构上"无从指定、无从持有"真实地址与凭据——这一层不依赖对输出内容的识别,是确定性的。
- **响应侧的系统级脱敏(ScrubSet)本质是黑名单**(匹配并擦除已知的真实地址/凭据及其常见编码形态),对未预见的编码变体(base64/hex、IP 整数化、分行/分块拼接、零宽字符插入等)是尽力而为、非穷尽。它是纵深防御的一层,**不是**"Agent 绝无可能从返回内容拼出真实地址"的绝对承诺。
- 对真实地址尤其敏感的场景,应优先采用"结构化字段白名单输出"(只放行已知安全字段)而非"全文黑名单擦除"。

结论:Agent 无法**指定**真实地址(匿名化,硬保证)、无法**持有**资源凭据(零接触,硬保证);无法从返回内容**读出**真实地址是脱敏尽力保证的目标,以请求侧两道硬保证为底座,而非孤立依赖输出擦除。响应侧脱敏的流式模型见详细设计 6.4,错误路径不绕过脱敏见详细设计 7.2 红线 3。

## 第十部分 · 可扩展性:传输、适配器与凭据来源

系统的可替换/可扩展点均为接口隔离的插件。新增连接方式、资源类型或更换凭据后端,是替换或新增实现,不触及核心。

### 10.1 Transport 插件

职责见 8.1。统一契约:建立、保活、健康、关闭,并声明自身是长连接型还是非长连接型(供连接管理层决定是否池化)。已知形态:SSM、SSH、Tailscale、Cloudflare Tunnel、直连。资源凭据与真实地址仅在插件内部使用。

### 10.2 Adapter 插件

职责:理解某种资源协议,把协议操作映射为 Capability 动词与对象,并执行被允许的操作。统一契约(概念层面):

- **能力声明** — 声明该协议支持哪些 Capability(如数据库适配器支持 query/mutate/destroy,日志适配器支持 observe)。
- **能力面自动发现** — 连上资源后,适配器可主动探测该资源的具体对象面并回报:数据库适配器列出 schema/表/列,容器适配器列出容器,HTTP 适配器从接口描述发现可用路径。其作用是免去逐项手工配置,把"接入一个资源"从手写清单变为"连上即得到候选对象,人在其上圈选授权"。**关键边界:发现 ≠ 授权。** 自动发现只产出"该资源有哪些对象"的事实,不授予任何访问;发现的对象在被显式纳入某授权格的细则之前,一律按默认拒绝处理(公理一)。发现降低配置摩擦,不改变授权必须显式的原则。
- **语义归一化** — 将一个具体协议请求解析为 (Capability 动词, 操作对象),供求值步骤[2]。此处承载协议感知:数据库适配器须做语法树级解析以正确归类动词、提取对象(表/列);命令适配器须把请求匹配到参数化模板;HTTP 适配器须解析方法与路径。
- **执行** — 经连接管理层提供的通路,对资源执行已被允许的操作,返回结果。
- **细则语义** — 定义该协议支持哪些细则类型(表白名单、容器前缀、命令模板、路径规则等),供求值步骤[4]。

已知形态:关系数据库(postgres/mysql,SQL 语法树解析)、容器运行时(日志与状态只读观察)、HTTP 服务(方法+路径;含**应用层使用面**——经业务系统账号代 Agent 操作其接口/页面,见详细设计 6.13)、键值存储(redis,命令归类 + 只读 ACL 兜底)、命令/部署(command,`execute` 仅接受参数化模板,承载机器操作与系统部署)、集群编排等。各类资源支持的动词与细则 kind 见详细设计 5.2bis 适配器能力矩阵。

### 10.3 命令模板:execute 类能力的约束形态

对 `execute` 类能力,适配器不接受自由命令文本,只接受**参数化模板**的实例化。模板由资源所有者预定义(命令骨架 + 受约束参数,如枚举值、受限字符串),Agent 只能选择模板并填入合法参数。这把"执行能力"从"任意命令执行"收窄为"在预定义安全操作集合内选择",是 execute / manage 类高危动词得以安全开放的前提。模板白名单作为细则附加在授权格上。

### 10.4 远端探针(可选插件)

部分能力无法仅靠"传输 + 协议适配"在远端获取(例如容器运行时日志,需在资源本机访问其本地接口)。对这类能力,提供可选的**远端探针**:部署在资源侧的轻量只读组件,经传输层被网关调用。

- 探针**可选**,不是接入资源的前提。能仅靠传输与适配器满足的能力不部署探针。
- 探针以插件形态提供,经传输通道由网关按需部署。
- 探针只读、能力面最小化(只暴露其存在理由所必需的观察接口),不引入额外远端攻击面。

### 10.5 资源凭据等级(Credential Tier)

一个资源可声明**多个资源凭据账号**,每个账号(称一个"凭据等级")声明它承载哪一组 Capability 动词。这把"网关与底层访问控制叠加而非取代"(1.4 非目标)落到实处:权限既由网关在求值期强制,又由底层引擎账号在执行期兜底强制。

- **声明**:资源接入时声明若干 tier(如 `readonly` 承载 `observe`/`query`,`readwrite` 承载 `mutate`),每个 tier 指向机密面中的一份资源凭据(以引用形式,非明文)。
- **选择**:求值放行后,系统按本次请求的动词选用承载该动词的 tier 建立连接(求值步骤[7]获取连接时按 (资源代号, 凭据等级) 取/建通路)。无任何 tier 承载该动词 → deny(配置缺失即拒绝,公理二)。
- **引擎兜底**:即使适配器把一条伪装写操作误归类为只读动词,它也只会经只读等级账号执行,被底层引擎拒绝。该兜底是**需被验证的前提**——"tier 声明的动词集 ⊆ 底层账号的真实权限"必须在接入时主动校验(连上后探测账号实际权限并比对),不自动成立(详细设计 6.3 的 tier 子集校验)。
- **隔离粒度**:不同 tier 的连接不共享(只读连接绝不被升格执行写),账号隔离在连接粒度成立。
- **适用边界**:对具备引擎级账号分级的资源(关系数据库)tier 是第二道强制;对不具备者(HTTP API、容器运行时等),归类 + 细则仍是唯一防线,该差异在适配器能力声明中显式表达。

Tier 是 Resource 声明的一部分,与第三部分五个核心概念正交叠加;模型关系(3.5)中 Resource 除声明 Transport/Adapter 外,还声明其凭据等级集合。

### 10.6 凭据来源(Credential Provider)可替换

传输层连接资源时需要资源凭据,但凭据的**来源**被抽象为一个可替换的 **Credential Provider** 接口——连接建立流程向该接口请求"当前可用于访问某资源某等级的凭据",不关心凭据是被静态保管的还是运行时动态签发的。已知来源形态:

- **静态来源** — 从凭据保险箱(操作系统密钥链 / 本地加密存储)取一个长期保管的密钥(如自建数据库的密码)。这类凭据需要保管与轮换。
- **动态来源** — 运行时向签发方换取一个短期凭证,用完即弃,不长期存储:云身份签发的临时凭证、数据库的临时认证令牌、面向 API 的 OAuth 令牌等。这类来源把"保管静态密钥"消解为"运行时凭身份换短期凭证",从根上减少了需保管的长期密钥。
- **证书来源** — 短期证书,自动续签(信任域放大到内网时与机器身份结合)。
- **会话来源(Authenticated Session Source)** — 面向"网关以某账号登录业务系统、维持会话、按账号代 Agent 操作"的场景(如 Agent 经网关使用业务系统的 HTTP 接口/页面)。它与动态来源的区别是**有状态且需维持**:登录一次得到会话/令牌后**复用并刷新**,而非每次请求重登;登录可能需要**二次验证(2FA)**。其完整生命周期(接入期引导含 2FA、运行期物化与刷新、过期 fail-closed)见详细设计 6.13。资源凭据可按 tier 映射为不同的业务系统账号(只读账号 / 普通账号 / 管理账号),引擎兜底即落在业务系统自身的权限体系上(与数据库 SELECT-only 账号、redis 只读 ACL 同构)。

设计意图与边界:

- 静态与动态是同一接口的不同实现,可按资源混用——能动态签发的资源走动态来源,只认固定密钥的资源(如自建且不支持动态认证的数据库)走静态来源。迁移某资源到动态来源,只改该资源的来源配置,传输层与求值逻辑不变。
- **不强制无密钥**:本系统默认运行在本地信任域,网关自身往往拿不到"环境赋予的身份"(那是云内工作负载才有的待遇),因而本地形态下根身份通常仍是一份需保管的根凭证;动态来源是为支持动态认证的资源与放大后的环境预留的能力,不是强制路径。保留静态来源的兼容是有意为之。

凭据保险箱本身(静态来源的实现)同样是接口隔离的可替换后端:系统提供自有实现(OS 密钥链 / 本地加密存储),其后端可替换为对接外部凭据系统的实现,核心取用逻辑只面向接口,不感知凭据实际存于何处。

## 第十一部分 · 存储:策略状态、审计与机密

### 11.1 三类数据

系统的数据分为三类,各有其权威载体。关键纠正:**策略不是"静态文件配置",而是系统管理的权威状态**——它是高频、动态、程序化读写的对象(临时授权、凭证签发吊销、模式切换、TTL 自动回收),其家必须是一个有事务、可并发、可查询、可审计的状态存储,而非文本文件。

- **策略状态(权威源)** — 角色定义、Principal 绑定、资源声明、细则、模式、脱敏规则、审批设置、拒绝指引、凭证元数据(绑定、有效期、可信域、吊销状态)、生效中的临时授权及其 TTL、当前模式。**全部**统一存于状态存储(嵌入式库),经控制面以事务方式读写。在求值看来,"observer 角色含哪些动词"与"此刻有一条 15 分钟的临时 mutate 授权"是同一种东西——**当前生效的策略**,只是后者带过期时间;不存在"静态配置"与"动态状态"的人为劈分,求值永远面对单一、统一、权威的策略状态。
- **审计(不可变事件流)** — 每个请求(allow 与 deny)、每次策略变更、每次凭证操作、每次连接事件,作为 append-only 的事件流落地。审计是观测/问责载体,与策略状态分离(策略是"当前是什么",审计是"发生过什么")。
- **机密(隔离持有)** — 资源凭据与代号↔真实地址映射,存于机密面的可替换后端(见 11.3、10.6),与策略状态、审计均隔离。

策略状态、审计、机密三者载体不同、访问路径不同,共同满足 daemon 自给自足约束(单机、无外部服务依赖)。

### 11.2 声明式表示:可选的导入/导出,不是权威源

文件在系统中依然有价值,但其角色是策略状态的一种**可导出、可导入的文本表示**,而非策略的存储本身。系统"运行在状态存储上",文件是进出该状态的一种格式(类比"期望状态 apply"模型):

- **导出(投影)** — 把当前策略状态导出为声明式文件,用于版本管理、评审、团队分享、跨环境复制。文件让"人作为立法者"以文本审视和讨论策略——这是文件的真实价值。瞬时状态(如带 TTL 的临时授权)可不导出,它本就该随时间消失。
- **导入 / apply(期望状态)** — 把一份声明式文件作为期望状态导入,系统据此协调更新状态存储。用于初始化、批量定义、可重现部署。
- **不变量** — 导出的文件中**永不出现明文凭据与真实地址**:资源凭据、真实地址只以"指向机密面的引用"形式存在;Principal 凭证不以明文导出(只导出其元数据如绑定、可信域、有效期)。这保证文件可安全地版本管理与分享。

声明式表示描述的对象(概念结构,非具体语法):**resources**(代号、传输声明含指向机密面的引用、适配器声明、凭据等级声明、细则、脱敏规则)、**roles**(Capability 集合、继承)、**principals**(角色与作用域绑定;凭证以元数据形式,含可信域,不含明文)、**modes**(仅声明各辖区的当前模式默认值)、**approval**(开关、超时、超时行为恒为拒绝)、**deny_notes**(资源所有者预写的拒绝指引)。

**模式(Mode)是固定集合,不可自定义**:`normal / observe / maintain / freeze` 四个模式及其权限覆盖规则(如观察模式仅放行 `observe`+`query`)是系统内置语义,固化在内核中(求值步骤[5]的 mode 谓词引用),声明式表示与控制面只能**选择/切换**模式与设置其 TTL,不能定义新模式或改写既有模式的覆盖规则。

求值与运行时只读权威状态存储,从不直接读文件;文件只在显式 import/export 时与状态存储交互。这是"声明式表示是接口、状态存储是权威"的根本区别。

### 11.3 机密面

机密面是两类机密的唯一持有者,经可替换后端实现:

- **资源凭据** — 访问资源所需的密钥/密码/令牌,经 Credential Provider(10.6)提供,可以是静态保管的或运行时动态签发的。在建立资源连接时由网关取用并注入,生命周期完全在机密面与传输层内部:取出/签发 → 注入连接 → 用毕(动态凭证用毕即弃)。不进入策略引擎、不进入数据面、不进入审计内容、不进入任何返回 Agent 的数据。审计只记录"哪个凭据/来源被使用"的标识,不记录凭据值。
- **代号↔真实地址映射表** — 匿名化的实现核心,机密面是其唯一权威持有者。代号→真实地址的**解析操作**由机密面提供、连接建立流程调用、传输插件在 open 内消费;数据面无任何读取路径(与 9.1"解析只发生在传输层内部"一致——传输是消费点,机密面是持有点)。控制面对映射可写入/更新,但**默认回读为掩码形态**(如 `10.0.*.*`)或 `vault://` 引用;明文回读需经显式标注的管理端点(人是信任域所有者,可开启,但不作默认)。导出(11.2)永不含明文真实地址。

机密面与 Principal 凭证(第七部分)是不同的东西:前者是网关访问资源用的资源凭据,后者是 Principal 向网关证明身份用的凭证。

### 11.4 审计的外部导出(可选)

审计默认完全自给自足:append-only 落本地,网关自带查询即可满足"我的 Agent 做了什么"的自我观测,无需任何外部依赖。

在此之上,审计写出预留一个**可选的导出器接口**,使审计事件可在落本地的同时,以**规范化格式**导出到用户已有的外部系统。要点:

- **是格式对齐,不是平台绑定** — 导出器输出符合开放遥测规范(如 OpenTelemetry 语义)的事件,而不对接任何特定平台。凡是能消费该规范的后端(自建监控、第三方可观测平台等)都可接收,接哪个、接不接,完全由用户决定,与网关无关。
- **默认关闭,可选启用** — 本地自用场景无需导出;有需要时(尤其信任域放大到团队、希望把 Agent 行为审计汇入既有监控)启用对应导出器。
- **导出器是可替换插件** — 规范化格式导出是其中一种实现;若用户的外部系统需要别的格式,新增一个导出器实现即可,不改审计核心。
- **导出不削弱本地权威** — 外部导出是审计的副本投影,本地 append-only 始终是审计的权威载体;导出通道不可用不影响审计本身。

## 第十二部分 · 认证与信任域放大

### 12.1 信任域的三档与产品边界

信任域可放大,产品覆盖前两档,放弃第三档:

```
单机信任域      →  个人桌面,daemon + 桌面外壳
组织内网信任域  →  自部署 daemon,团队共享(人与 Agent 均为 Principal)
公共云信任域    →  不做(与安全定位矛盾,公理五)
```

放大改变的是"谁在信任域内"(从一人到一组织),不改变"敏感数据不出信任域"这一根本。

### 12.2 放大时的隔离点替换

放大信任域时,以下隔离点发生实现替换,核心模型与求值逻辑不变:

| 隔离点 | 本地实现 | 内网实现 |
|---|---|---|
| 前端 RPC 传输(控制面) | 进程间通信(socket/pipe) | 增加 mTLS over TCP 绑定 |
| 数据面绑定 | Unix Domain Socket(data.sock) | 可选增加 TLS/mTLS over TCP 监听;凭证 IP/网络可信域据传输层对端地址(`ConnOrigin::Tcp`)校验 |
| 认证器(Principal) | 本地进程身份 / API Key | 增加 mTLS / SSO / 账密换 Token |
| 凭证可信域 | 可选,约束本地来源 | 与 mTLS 证书叠加,构成证书+来源网络双重约束 |
| 凭证生命周期 | 本地签发与吊销 | 接入组织身份体系,集中签发/吊销 |
| 状态存储 | 嵌入式库(单文件) | 可替换为更强并发存储 |
| 审计写入 | append-only(自我观测) | 可替换为带防篡改的问责级实现 |
| 审计导出 | 默认不导出 | 启用导出器,汇入团队既有监控(对齐开放遥测规范) |
| 凭据来源 | 静态保险箱为主(本地根凭证) | 更多资源转动态签发 / 证书,减少静态密钥 |
| Principal 来源 | 用户自身的 Agent / 程序 | 多用户及各自的 Agent |

这些隔离点在设计上即被显式抽象,放大是替换实现而非重构核心。

### 12.3 审计定位随信任域而变

- **单一信任域内** — 操作发起者与审计所有者是同一主体,审计是自我观测工具,无问责对抗,故不需要防篡改。
- **放大后** — 当信任域含多用户,"操作者 ≠ 审计所有者"重新成立,审计的问责属性回归,防篡改成为有意义的需求,通过替换审计写入实现(12.2)引入。

审计写入接口因此从一开始就被抽象,使两种语义可切换。

## 第十三部分 · 安全模型

系统自身是信任的集中点,故须明确"看守者由谁看守"。

### 13.1 威胁模型(单一信任域)

主要威胁主体是 **Agent 本身**——非因恶意,而因它会读取不可信内容、可能被提示注入、可能自主探索越界。次要威胁是能触达本机的其他本地程序。信任域之外的网络攻击在本地形态下不在主面(控制面不监听网络);放大到内网后,网络面经 mTLS 等收口。

明确**不在威胁模型内**:用户篡改自己的审计(单一信任域下操作者即审计所有者,无此动机,见 12.3)。

### 13.2 威胁与防线

| 威胁 | 防线 |
|---|---|
| 提示注入诱导 Agent 窃取资源凭据 | 凭据零接触(公理四):资源凭据不在 Agent 可达范围,注入无物可窃 |
| 提示注入诱导 Agent 执行破坏性操作 | 默认拒绝 + RBAC + 协议感知细则:未授权操作执行前被拒,与 Agent 是否被说服无关 |
| Agent 自主探索触达控制面/配置/审计 | 控制面/数据面隔离:控制面不监听网络,数据面无读写策略与审计的路径 |
| Agent 提权(获取 admin) | admin 不可授予(模型层硬约束) |
| 通过返回内容泄露拓扑/凭据 | 响应脱敏(系统级默认生效) |
| 通过资源代号或拒绝响应探测拓扑 | 匿名化 + 拒绝响应只述自身授权世界 |
| 伪装写操作绕过只读授权 | 协议感知归一化:SQL 语法树解析识破伪装,正确归类动词 |
| 自由命令执行 | execute 仅接受参数化模板,无自由命令通道 |
| 临时授权遗留口子 | TTL 默认且自动回收 |
| 凭证泄露后被长期利用 | 凭证过期 + 轮换 + 即时吊销 |
| 凭证被从非授信位置盗用 | 凭证可信域:仅约定来源可使用该凭证,来源不符即认证失败(基于可观测来源,不采信自报) |
| 失控 Agent 持续操作 | 吊销其凭证(按身份)或冻结模式(按操作),均热生效 |
| 连接耗尽 / 僵死通路 | 连接管理层:并发上限、健康剔除、空闲回收(超限 fail-closed) |
| 任一环节异常被当作放行 | fail-closed(公理二):一切不确定解析为拒绝 |
| 误判系统是否真严格 | 红队自检(13.4):主动验证而非假设 |

### 13.3 网关自我保护

- 控制面与数据面物理隔离,Agent 触达不到控制面。
- 策略、审计、资源凭据、映射表仅控制面/机密面内部可达,数据面无路径。
- 凭证的签发/轮换/吊销只在控制面;数据面只能出示凭证。
- admin 能力在模型层不可授予 Principal。
- daemon 运行于 Agent 沙箱之外的信任侧(本地形态对应"daemon 与凭据在用户主环境、Agent 在隔离环境"的部署模式;隔离由部署环境提供,网关设计假设并要求该隔离存在)。

### 13.4 红队自检

系统提供主动验证手段:模拟一组越权请求(伪装的破坏性操作、模板外命令、对脱敏字段的读取、对 Scope 外资源的访问、提权尝试、使用已吊销/过期凭证),逐条确认全部被拒。这把"我相信系统严格"变为"我验证过系统严格",是公理三在运维层面的延伸,覆盖求值管线每道防线。

## 第十四部分 · 端到端流程串讲

以三个代表性场景串联全部设计。

### 14.1 场景:脚本经 HTTP 排查问题(入口对称 + 正常路径)

```
1. 用户的运维脚本(Principal=ci-bot,以 API Key 认证,角色 observer,
   Scope 含 app-prod / db-main)经 HTTP 外壳发起:读 app-prod 最近错误日志
2. 外壳归一化 → 统一请求模型 → 认证:验证 API Key,未过期未吊销 ✓ → Principal 确立
3. 容器适配器语义归一化:动词=observe,对象=app-prod 日志(容器前缀细则内)
4. RBAC:observer 含 app-prod:observe ✓;细则:前缀匹配 ✓;条件层 ✓ → allow
5. 向连接管理层请求 app-prod 通路:命中池中已有长连接(SSM 隧道),复用
6. 适配器读日志 → 响应脱敏(擦除日志中出现的内网 IP)
7. 审计:ci-bot / observe / app-prod / allow / 所用凭证标识 / 摘要
8. 经 HTTP 外壳返回脱敏日志
   —— 同一请求若来自 MCP 外壳的 Agent,自第2步起完全相同(公理七)
```

### 14.2 场景:Agent 尝试越权(拒绝与分流)

```
1. 本地 Agent(经进程身份认证,零凭证;角色 observer)推断需修正一条数据,
   经 MCP 外壳发起 UPDATE db-main 某表
2. SQL 适配器语义归一化:语法树解析 → 动词=mutate,对象=该表
3. RBAC:observer 不含 db-main:mutate → deny
4. 组装结构化拒绝:
     reason:       "role=observer 不含 db-main:mutate"
     your_grants:  { db-main: [query], app-prod: [observe] }
     request_hint: "<提升 db-main:mutate 的命令,默认带 TTL>"
     operator_note:(若所有者预写了指引则原样转述)
5. 审计记录该 deny 及完整依据;计入拒绝分析
6. Agent 依 your_grants 判断:无写权限,可
     - 自我修正:改以 query 复核现状(①/②级),或
     - 向用户陈述需修改的理由并附 request_hint(③级)
7. 用户在场认可 → 执行一条提升命令(带 TTL)→ 热生效
   → Agent 重试 → mutate 在授权内 → 通过 → 操作 → TTL 到期自动回收
8. 全程:人最多粘贴一条命令;同步审批未参与;边界由系统强制保持
```

### 14.3 场景:发现失控,即时切断

```
1. 拒绝分析/告警显示某 Principal 短时间大量越权请求
2. 用户两种粒度处置(均热生效):
     - 吊销该 Principal 的凭证 → 其后续请求在认证步骤即被拒(精确到身份)
     - 或切换冻结模式 → 该资源/全局一切操作被拒(精确到操作)
3. 在用的连接由连接管理层在优雅销毁时排空并释放
4. 处置动作(吊销/模式切换)进审计
```

## 附录 A · 设计决策速查

| 决策点 | 选择 | 依据 |
|---|---|---|
| 权限组织 | RBAC(角色×作用域)+ Capability 动词原子 + 可扩展条件谓词层 | 角色解决授权管理,动词解决执行精确,谓词集解决动态(可信域亦为其一) |
| 审批定位 | 默认关闭的可选机制;escalate 默认折叠为 deny | 人不在环是常态,安全不依赖实时审批(公理二) |
| 拒绝响应 | 结构化、只含事实或人预写;动态建议由 Agent 推理 | 公理六;避免系统编造误导性建议 |
| 数据面入口 | 协议无关内核 + 多外壳(MCP/HTTP/SDK);HTTP 为一等程序化入口 | 入口对称、语义一致(公理七);复杂用法走 HTTP |
| 进程形态 | 单核心 daemon + 多前端(共享 SPA + 可选桌面壳 + CLI) | 价值压在 daemon,UI 是壳;同覆盖桌面与自部署 |
| 控制面/数据面 | 物理隔离,控制面默认不监听网络 | Agent 触达不到管理面与配置(网关自保) |
| 身份认证 | 认证器族(本地身份/APIKey/Token/账密/mTLS/SSO),统一抽象 | 本地零凭证、放大可强认证;认证只确立身份不赋权 |
| 凭证生命周期 | 签发/过期/轮换(重叠期)/吊销(热生效)/可选可信域/审计 | 凭证泄露可控,失控可即时切断,异地盗用可防 |
| 凭证可信域 | 凭证可选绑定允许使用的来源(IP/网络/域),来源不符即认证失败 | 限制凭证只在授信位置可用;基于可观测来源 |
| 连接管理 | 传输层(本地 socket,长/非长连接)+ 连接管理层(池化/健康/指数退避重连/上限/回收) | 复用降成本、自愈不自残;上层无感知 |
| 资源接入 | 能力面自动发现(发现≠授权)+ 声明式配置(凭据以引用形式) | 降低手配摩擦;授权仍须显式;配置可安全分享 |
| 凭据来源 | Credential Provider 可替换:静态保险箱 / 动态签发 / 证书;本地留兼容 | 能动态的资源走动态、消解静态密钥;本地不强推无密钥 |
| 存储 | 策略=系统管理的权威状态(库);声明式文件=可选导入/导出表示;审计=不可变事件流;机密=隔离持有 | 策略高频动态需事务存储;文件是接口非权威;三类分离 |
| 审计防篡改 | 单一信任域内不做,写入接口预留 | 操作者即审计所有者,无问责对抗(12.3) |
| 审计外部导出 | 默认本地自给;可选导出器,对齐开放遥测规范而非绑定平台 | 接哪个平台由用户定;格式对齐零摩擦,本地仍权威 |
| 信任域 | 本机→内网可放大;不做公共云 | 数据不出信任域(公理五) |
| 扩展点 | Transport / Adapter / 凭据来源 / 导出器 插件;探针可选 | 新增连接方式/资源类型/凭据来源/导出目标不触核心 |
| execute 安全 | 仅参数化模板,无自由命令 | 把执行收窄为安全操作集合内的选择 |
| 凭据/地址 | 仅存机密面,配置只存引用,响应脱敏 | 凭据零接触 + 匿名化(公理四) |

## 附录 B · 公理与设计对应

- 公理一(默认拒绝)→ RBAC 白名单求值、未授权即拒。
- 公理二(失败即关闭)→ 求值各步异常即拒、凭证过期/无法验证/来源不在可信域即拒、连接不可建即拒、审批超时即拒、escalate 默认折叠为 deny。
- 公理三(系统强制)→ 协议感知校验、admin 不可授予、命令模板、凭证可信域基于可观测来源、红队自检。
- 公理四(凭据零接触)→ 机密面隔离、匿名化、响应脱敏、声明式表示只存引用、Principal 凭证与资源凭据分离。
- 公理五(单一信任域可放大)→ 隔离点显式抽象、不做公共云、数据不出信任域。
- 公理六(只说事实)→ 结构化拒绝只含事实与人预写内容、动态建议交由 Agent 推理。
- 公理七(入口对称)→ 数据面协议无关内核 + 多外壳、求值管线统一、跨入口安全语义一致。

## 附录 C · 关键接口隔离点

设计中显式抽象、可替换的隔离点(放大或扩展时替换实现,不改核心):

| 隔离点 | 作用 | 可替换为 |
|---|---|---|
| 数据面外壳 | 接入形态 | MCP / HTTP / SDK / 未来形态 |
| 控制面 RPC 传输 | 前端连接 daemon | 本地 IPC / 网络 mTLS |
| Authenticator | 确立 Principal 身份 | 本地身份 / APIKey / Token / 账密 / mTLS / SSO |
| Transport | 到达资源 | SSM / SSH / Tailscale / CF / 直连 / 未来形态 |
| Adapter | 理解资源协议 | postgres / mysql / docker / http / redis / k8s / 未来形态 |
| 连接管理策略 | 通路池化、健康、指数退避重连与回收 | 不同池化/退避/回收策略 |
| 凭据来源(Credential Provider) | 提供资源凭据 | 静态保险箱(自研本地加密/OS 密钥链/外部凭据系统)/ 动态签发 / 证书 |
| 状态存储 | 权威策略状态(角色/绑定/凭证元数据/临时授权/模式等) | 嵌入式库 / 更强并发存储 |
| 审计写入 | 审计落地 | append-only 自我观测 / 防篡改问责级 |
| 审计导出器 | 审计向外部投影 | 对齐开放遥测规范(OTel)/ 其他格式;默认不导出 |
| 远端探针 | 本机才能取的能力 | 可选,按能力类型 |
