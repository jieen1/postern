# postern — 详细设计文档

> 本文档是《本地安全网关-技术设计文档》的实现层设计：技术选型、代码分层、核心模块与接口、数据库与存储、关键机制、工程规范。技术设计文档回答"系统是什么样"，本文档回答"代码与数据长什么样"。两者冲突时以技术设计文档的公理为准。本文档不含任何阶段划分或工作量描述。

## 第一部分 · 已定技术决策

| # | 决策点 | 结论 | 关键理由 |
|---|--------|------|----------|
| D1 | 策略状态存储 | SQLite（rusqlite，bundled，WAL） | 策略是关系负载（矩阵、TTL、聚合）；安全产品存储层零新颖性风险 |
| D2 | 审计载体 | 按日轮转 JSONL 事件流 | 载体物理 append-only、事件流语义原生；本系统量级下扫描查询足够快（见 5.3） |
| D3 | 代码分层 | 中等粒度 cargo workspace（一面一 crate） | 依赖图即架构图，"数据面无路径触达机密/策略"成为编译期事实 |
| D4 | 控制面 RPC | HTTP/JSON over Unix Domain Socket（axum） | SPA/CLI 同一 API；与数据面共享一套 HTTP 栈；放大时同一 router 加挂 mTLS TCP listener |
| D5 | SQL 语义归一化 | sqlparser-rs 统一解析 + **资源凭据按能力分级**（引擎账号兜底强制） | 解析器负责归类与细则，权限强制兜底下沉到数据库引擎账号——与"网关与底层访问控制叠加而非取代"（技术设计文档·非目标）一致 |
| D6 | 机密保险箱 | 统一加密文件载体（XChaCha20-Poly1305 加密 payload；随机 data-key 经解锁来源包裹）+ 解锁材料来源插件化 | 存储路径全形态唯一；headless/桌面/严格场景差异收敛在解锁来源接口；各来源真实强度不同（argon2id KDF 仅作用于 passphrase 来源，见 5.4） |
| D7 | 命名 | 全面统一 **postern**：CLI = `postern`，daemon = `posternd`，crate 前缀 = `postern-` | 单一品牌；全文术语一致 |
| D8 | 数据库层统一规范 | 8 个统一基础字段、审计字段自动化、仅逻辑删除、禁裸 SQL（豁免须申请）、强制后端分页、统一雪花 IdGen——每条固化为 Stele 契约并带反例自检 | 数据库层约束必须系统强制而非约定（公理三）；反例保证契约真实生效（见 5.1、7.5） |

自定默认项：tokio（异步运行时）、axum（两平面共用 HTTP 栈）、rmcp（MCP 外壳，官方 Rust SDK）、thiserror（库层错误）、tracing（结构化日志）、clap（CLI）、TOML（声明式导入/导出格式）、编译期 trait 对象注册（插件机制）、Cargo edition 2021。

**D5 的设计后果**（贯穿全文）：

- "资源凭据等级（Credential Tier）"成为一等领域概念：一个 SQL 资源可声明多个账号（如 `readonly` / `readwrite`），每个账号声明其承载的 Capability 集合；求值放行时，**策略引擎**依快照中的 tier 声明按本次动词选定承载该动词的凭据等级（产出 `Decision::Allow{tier}`，无匹配 tier → deny），**连接管理层**据此 `(资源, tier)` 取/建连接（连接管理只按已选 tier 建连，不做动词→tier 的选择，属主裁决见第八部分 8.3/8.5 与速查表）。即使解析器把一条伪装写操作误归类为 `query`，它也只会走 SELECT-only 账号，被数据库引擎拒绝——强制兜底在引擎层。
- SQL 解析（sqlparser-rs）的职责收敛为：动词归类（决定查哪个授权格、走哪个账号）、对象提取（表/列，供细则与审计）、语句形态白名单（无法可靠归类的语句类型一律拒绝，公理二）。
- 对**不具备引擎级账号分级**的资源类型（HTTP API、容器运行时等），归类 + 细则仍是唯一防线；该差异在适配器能力声明中显式表达。
- SQL 类资源接入时强烈建议（不强制）声明至少 read / write 两个凭据等级；`postern verify` 红队自检包含"伪装写操作经只读等级账号被引擎拒绝"的验证项。

## 第三部分 · 代码分层

> 本部分把领域映射到 crate（载体）。各**领域的语义边界、范围内/外、对外接口与必守不变量**的权威定义见**第八部分 · 领域范围与内涵**——领域是语义边界，crate 是载体，一个 crate 可承载多个领域（如 `postern-daemon`）。读代码分层前若对"某职责归哪个领域"有疑问，以第八部分为准。

### 3.1 workspace 布局

```
postern/
├── Cargo.toml                  # [package] postern-contracts + [workspace]
├── tests/contract/             # Stele 生成的契约测试（受保护，路径不可动）
├── contract/                   # Stele 契约源（受保护）
├── docs/
└── crates/
    ├── postern-core/           # 领域模型 + 求值管线 + 全部插件 trait（零 IO）
    ├── postern-store/          # 策略状态存储（SQLite）+ 审计事件流（JSONL）
    ├── postern-secrets/        # 机密面：保险箱、代号↔真实地址映射、Credential Provider
    ├── postern-transports/     # Transport 插件：ssh / ssm / direct
    ├── postern-adapters/       # Adapter 插件：postgres / docker-logs / http
    ├── postern-daemon/         # 二进制 posternd：外壳、控制面、连接管理、组装
    └── postern-cli/            # 二进制 postern：控制面瘦客户端
```

根包命名为 `postern-contracts`，仅承载 Stele 契约测试；`tests/contract/` 与 `stele.config.json`（受保护）规定的生成路径不参与重命名。

### 3.2 依赖规则（编译期架构强制）

```
                 postern-core        ← 不依赖工作区内任何 crate
                /   |    |   \
       postern-store |  postern-transports  postern-adapters
                postern-secrets
                     \   |   /
                  postern-daemon     ← 唯一的组装点
                      |
                 postern-cli         ← 仅依赖 core（共享类型）+ HTTP 客户端
```

**禁止边**（违反即架构破坏，CI 以 `cargo tree` 校验）：

| 禁止 | 对应设计约束 |
|------|--------------|
| `adapters` → `secrets` / `transports` / `store` | 适配器只拿"一个可用通路"，不经手凭据与真实地址（技术设计文档 8.3） |
| `transports` → `store` | 传输层不触策略与审计 |
| `cli` → `store` / `secrets` | 客户端不含安全逻辑，一切经控制面 API |
| `core` → 任何 IO crate | 求值管线可被契约测试纯内存驱动 |

插件以模块 + cargo feature 存在于面 crate 内（如 `postern-transports` 的 `ssh`、`ssm` feature），不再细分 crate。

### 3.3 二进制

| 二进制 | crate | 职责 |
|--------|-------|------|
| `posternd` | postern-daemon | 核心 daemon：两平面监听、求值、连接、审计 |
| `postern` | postern-cli | 控制面 CLI：策略/凭证/资源/审计管理，经 UDS 调 daemon |
| `postern mcp-stdio` | postern-cli 子命令 | 零逻辑 stdio↔UDS 桥，供仅支持 stdio 的 MCP 宿主 |

核心类型：

```rust
pub enum Capability { Observe, Query, Mutate, Execute, Manage, Destroy }
// 注意:不存在 Admin 变体——"admin 不可授予"在类型层面成立(公理三)

pub struct ResourceCode(String);     // 资源代号,如 "db-main";真实地址类型只存在于 secrets crate
pub struct PrincipalId(SnowflakeId);  // 雪花 ID,全工作区唯一来源是 core::id::IdGen
pub struct CredentialTier(String);   // 资源凭据等级名,如 "readonly"

/// 外壳归一化产物(步骤[0]输出)——自此请求与外壳无关(公理七)
pub struct NormalizedRequest {
    pub presented: PresentedCredential,   // Agent 出示的网关凭证(或本地进程上下文)
    pub origin: ConnOrigin,               // 网关可观测的连接来源,绝不采信自报字段
    pub resource: ResourceCode,
    pub intent: Intent,                   // 协议原始意图(SQL 文本/容器日志请求/模板调用...)
}

pub enum ConnOrigin {
    UnixPeer { uid: u32, gid: u32 },      // SO_PEERCRED:仅取 uid/gid 作信任域门
    Tcp { remote: SocketAddr },
}
// 注意:不取 pid/exe 路径作身份依据——PID 可复用、/proc 读取存在 TOCTOU 伪冒面,
// 均为可伪造特征。本地进程的 Principal 区分见 4.1 Authenticator·local_process 可信度边界。

/// 适配器语义归一化产物(步骤[2]输出)
pub struct ClassifiedIntent {
    pub capability: Capability,
    pub objects: Vec<ObjectRef>,          // 表/列、容器名、路径、模板 id...
}

pub enum Decision {
    Allow { grant: MatchedGrant, tier: CredentialTier },
    Deny(DenyResponse),
    Escalate { fallback: DenyResponse },  // 审批关闭时即取 fallback(恒 deny)
```

插件 trait（全部定义在 core，实现在各面 crate——接口隔离点与附录 C 一一对应）：

```rust
/// 认证器族(步骤[1])。实现:local_process / api_key / token;放大后增 mTLS / SSO
pub trait Authenticator: Send + Sync {
    fn kind(&self) -> &'static str;
    fn authenticate(&self, presented: &PresentedCredential, origin: &ConnOrigin,
                    creds: &CredentialView, now: Timestamp) -> Result<PrincipalId, AuthError>;
    // 凭证无效/过期/吊销/可信域不符/来源无法判定 → Err → deny(公理二)
    // now 显式传入(对齐 Evaluator):凭证 expires_at/revoked_at/可信域时效在求值时刻
    // 按墙钟二次校验,过期即刻失效——不依赖后台 sweeper 时序(见 6.2)
}

/// 适配器(步骤[2][4][8])。实现:postgres / docker_logs / http
#[async_trait]
pub trait Adapter: Send + Sync {
    fn protocol(&self) -> &'static str;
    fn capabilities(&self) -> &'static [Capability];
    /// 引擎级强制可用性:SQL 类为 true(凭据分级兜底),HTTP/容器类为 false(归类+细则是唯一防线)
    fn engine_enforced(&self) -> bool;
    fn classify(&self, intent: &Intent) -> Result<ClassifiedIntent, ClassifyError>; // Err → deny
    fn check_constraint(&self, spec: &ConstraintSpec, ci: &ClassifiedIntent)
        -> Result<bool, ConstraintError>;                                            // Err → deny
    async fn execute(&self, ch: &mut Channel, intent: &Intent) -> Result<RawResponse, ExecError>;
    async fn discover(&self, ch: &mut Channel) -> Result<CapabilitySurface, DiscoverError>; // 发现≠授权
}

/// 传输(步骤[7b]取连接的底层)。实现:ssh / ssm / direct
#[async_trait]
pub trait Transport: Send + Sync {
    fn kind(&self) -> &'static str;
    fn persistent(&self) -> bool;        // 长连接型→入池;非长连接型→用毕即销
    async fn open(&self, target: ResolvedTarget, cred: ResourceCredential)
        -> Result<Channel, TransportError>;
    // ResolvedTarget / ResourceCredential 由 daemon 从机密面取出注入,
    // 二者不实现 Clone/Serialize,Debug 输出恒为 REDACTED,生命周期不出本调用
}

/// 资源凭据来源(技术设计文档 10.6)。实现:静态保险箱;接口预留动态签发/证书
#[async_trait]
pub trait CredentialProvider: Send + Sync {
    async fn credential_for(&self, res: &ResourceCode, tier: &CredentialTier)
        -> Result<ResourceCredential, CredentialError>;
}

/// 条件谓词(步骤[5],可扩展集合)。内置:rate_limit / time_window / mode / ttl
pub trait ConditionPredicate: Send + Sync {
    fn kind(&self) -> &'static str;
    fn eval(&self, ctx: &EvalContext, spec: &serde_json::Value) -> Result<bool, PredicateError>;
    // Err 或无法判定 → 视为不满足 → deny(公理二)
}

/// 审计写入(隔离点:append-only 自我观测 / 放大后防篡改实现)
pub trait AuditSink: Send + Sync {
    fn record(&self, event: AuditEvent) -> Result<, AuditError>;
}

/// 响应脱敏(步骤[9])。小响应整体脱敏;流式大输出走滑动重叠窗口(见 6.4)
pub trait Sanitizer: Send + Sync {
    /// 完整 payload 一次脱敏(小响应、错误串、拒绝响应)
    fn scrub(&self, payload: RawResponse, declared: &[MaskRule]) -> SanitizedResponse;
    /// 流式脱敏:逐 chunk 处理,内部保留上块尾部 N 字节参与下块匹配,消除边界分块逃逸;
    /// observe 大流由有界缓冲与背压约束(见 6.4 流式脱敏模型)
    fn scrub_stream(&self, declared: &[MaskRule]) -> Box<dyn StreamScrubber>;
}

/// 策略读取视图:求值面对的"单一权威策略状态"
pub trait PolicyView: Send + Sync {
    fn snapshot(&self) -> Arc<PolicySnapshot>;   // 见 6.2 热生效机制
}
```

**`local_process` 认证的可信度边界**（SO_PEERCRED 的真实强度）：本地进程身份由 `SO_PEERCRED` 确立，但内核只能可靠提供对端的 uid/gid，无法区分同 uid 下的多个 Agent 实例。这与"每个 Agent 实例是独立具名 Principal"（3.1）存在张力，须如实界定：

- **只取 uid/gid 作信任域门**：放弃 pid、exe 路径这类可伪造特征作为身份——PID 复用与 `/proc` 读取的 TOCTOU 使其不可作为认证依据。
- **同 uid 的 Principal 区分**，二选一并在审计语义中如实标注采用了哪种：
  - **承认"一个 uid = 一个 Principal"**：同 uid 进程在网关眼中是同一本地 Principal，审计语义据实说明，不假装能区分；或
  - **本地握手令牌**：控制面拉起 Agent 时为每个本地实例注入一个轻量本地握手令牌（属身份凭证、**非资源凭据**，不触碰机密面），Agent 出示该令牌据以区分实例。
- **部署隔离假设**：`local_process` 的可信度依赖部署侧隔离（见 5.5/13：Agent 运行于独立于 daemon 的 uid）；该假设是 `local_process` 成立的前提，文档据实声明，不夸大其单独的认证强度。

求值器（步骤 [1]~[6] 的纯逻辑，零 IO，可被 Stele 契约测试直接驱动）：

```rust
pub struct Evaluator { /* 持有 Authenticator 注册表、ConditionPredicate 注册表 */ }

/// 步骤[4]细则校验的结果,由 kernel 先行调用 Adapter::check_constraint 得出后传入
pub struct ConstraintCheck { pub passed: bool /* false → evaluate 据此 deny */ }

impl Evaluator {
    /// 输入:归一化请求 + 适配器归类结果 + 细则校验结果 + 策略快照 + 墙钟;
    /// 输出:三值决策 + 完整求值轨迹
    pub fn evaluate(&self, req: &NormalizedRequest, ci: &ClassifiedIntent,
                    constraint: &ConstraintCheck,
                    policy: &PolicySnapshot, now: Timestamp) -> (Decision, EvalTrace);
}
```

`EvalTrace` 记录"在哪一步、因何判定"，供审计与拒绝响应组装；`now` 显式传入保证决策确定性（相同输入→相同决策，技术设计文档 5.2）。

**细则校验的归属（CONS-8）**：步骤[4]细则校验由 `Adapter::check_constraint` 执行（它属适配器，需 IO 上下文），而 `Evaluator` 保持 core 零 IO、不持有 Adapter。二者协调方式：**kernel 在调用 `evaluate` 前先跑 `check_constraint`，把 `ConstraintCheck` 作为入参传入** `evaluate`。如此 `evaluate` 仍是纯逻辑（`ConstraintCheck` 是已物化的布尔事实），契约测试可纯内存构造 `ConstraintCheck` 驱动求值，"core 零 IO、契约测试纯内存驱动"成立。`check_constraint` 返回 `Err`/`false` 时 kernel 直接以 deny 短路，或传入 `passed=false` 的 `ConstraintCheck` 令 `evaluate` 据此 deny（二者等价，失败语义见 6.1 步骤[4]）。

### 4.2 postern-store

```
src/
├── base/          # 统一基础仓储:唯一写路径(自动填充审计字段+乐观锁+逻辑删除)、
│                  # 默认作用域构建器(delete_flag=0)、分页执行(LIMIT/OFFSET)
├── policy/        # PolicyRepo:全部策略状态的事务读写(rusqlite,同步,一律经 base)
├── snapshot/      # PolicySnapshot 构建:一次事务内全量加载并展开授权空间
├── audit/         # JsonlAuditSink:按日轮转 JSONL 写入 + 分页扫描查询
└── migrate/       # schema 版本与迁移(user_version)
```

- rusqlite 为同步 API：写路径由控制面单线程串行化（写互斥锁），读路径只在快照重建时发生；数据面求值不触库（读内存快照）。

postgres 适配器的语义归一化（D5 落点）：

```
SQL 文本 → sqlparser(PostgreSqlDialect) → Vec<Statement>
  ├─ 解析失败 / 多语句 / 未知节点      → ClassifyError → deny
  ├─ 语句形态白名单归类(显式枚举,按语句树内最高危写节点定档):
  │    遍历整棵语句树(含 CTE、子查询、INTO 目标),取出现过的最高危写节点
  │    任意位置出现 Delete/Drop/Truncate → Destroy
  │    任意位置出现 Insert/Update/Merge   → Mutate
  │    纯只读 Query(无任何写节点/无 INTO) → Query
  │    Show / Explain(不带 ANALYZE)       → Observe
  │    其余一切语句类型(Set/DO/COPY/CALL/...) → 不可靠归类或改变会话语义 → deny
  └─ 对象提取:遍历 AST 收集 schema.table 与列 → 细则校验与审计
```

归类按**语句树内出现的最高危写节点**定档，而非按顶层语句类型——危险动词的可见性在整棵树范围内成立，CTE 或子查询里的 `DELETE`/`DROP`/`TRUNCATE` 一律提升为 Destroy，绝不因被只读外壳（如 `WITH x AS (DELETE ...) SELECT ...`）包裹而降级为 Mutate 或 Query。

`SET` 改变会话语义（search_path、role、超时等），不属于只读观测，**一律拒绝、不设白名单逃生口**——这与伪代码的 `Set → deny` 一致，且避免"放行的 SET 该归哪个动词"这一悬空问题（无归属动词则 RBAC 步骤[3]无格可判，违 fail-closed）。会话语义的调整（如设定只读会话）应由适配器在建立连接时统一施加，而非接受 Agent 的 `SET` 请求。`EXPLAIN` 仅允许不带 `ANALYZE` 的形态归为 Observe——`EXPLAIN ANALYZE` 会真实执行被解释的语句，按其内部最高危写节点定档（含写节点即非 Observe）。

白名单之外宁可误拒（公理二）；归类正确性的**强制兜底**是凭据分级——`Query` 动词恒走只读账号，引擎层拒绝一切写入。

## 第五部分 · 数据库与存储设计

### 5.1 数据库层统一规范（全部固化为 Stele 契约）

适用范围：policy.db 全部表。审计 JSONL 是事件流而非数据库表，不适用本节字段规范（其事件 `id` 同样取自统一 IdGen）。每条规则都对应 Stele 契约（括号内为契约 id），且各带一条反例自检契约（见 ⑦）。

**① 统一基础字段**（`DB_BASE_FIELDS_REQUIRED`）— 每张表必须声明以下 8 个字段：

| 字段 | 类型 | 语义 |
|---|---|---|
| `id` | INTEGER PRIMARY KEY | 雪花 ID，唯一来源 `postern-core::id::IdGen`（见 ⑥） |
| `version` | INTEGER NOT NULL DEFAULT 0 | 乐观锁版本号，每次 UPDATE 自增 |
| `created_at` | TEXT NOT NULL | UTC ISO-8601，写入时自动填充 |
| `created_by` | TEXT NOT NULL | 创建者标识，自动填充 |
| `updated_at` | TEXT NOT NULL | 最后更新时间，自动维护 |
| `updated_by` | TEXT NOT NULL | 最后更新者，自动维护 |
| `delete_flag` | INTEGER NOT NULL DEFAULT 0 | 逻辑删除标记（0=正常，1=已删除） |
| `enable_flag` | INTEGER NOT NULL DEFAULT 1 | 业务启停标记（1=启用，0=停用） |

**② 审计字段自动化**（`DB_WRITE_PATH_CENTRALIZED`）— `version/created_at/created_by/updated_at/updated_by` 由统一基础仓储（`postern-store::base` 模块）自动填充，调用方**无法**传入这些字段（仓储 API 不暴露相应参数）。INSERT/UPDATE 语句只允许出现在 base 模块——写路径唯一是自动化成立的前提。乐观锁：`UPDATE ... SET version = version + 1 ... WHERE id = ? AND version = ?`，影响行数为 0 → 版本冲突错误（不静默重试）。`created_by/updated_by` 取值：控制面写入 = 已认证操作者标识；系统自动写入（sweeper 回收、import 协调）= `system`。

**③ 逻辑删除，无物理删除接口**（`DB_LOGICAL_DELETE_ONLY` + `DB_DEFAULT_SCOPE_EXCLUDES_DELETED`）— 删除一律 `UPDATE ... SET delete_flag = 1`（连带 version 自增与 updated_* 维护）；整个工作区不存在针对 policy.db 的 `DELETE` 语句，仓储不提供物理删除 API。默认查询作用域：base 模块的作用域构建器自动追加 `delete_flag = 0`，业务查询默认看不到已删数据；`enable_flag` 不在默认过滤内（启停是业务语义，由调用方显式使用）。import/apply 协调期望状态时对"多余"记录同样置 delete_flag，不物理删。

**④ 禁止裸 SQL，例外需申请**（`DB_NO_RAW_SQL_OUTSIDE_STORE`）— SQL 字符串与 rusqlite 依赖只允许存在于 postern-store；其他 crate 一律经仓储 API。例外的唯一豁免来源是 `contract/sql-exceptions.json`（**Stele 受保护路径**，初始为空数组），条目须写明 `{crate, file, reason, approved_by}`——修改该文件必然经过人工审批的契约变更流程，"明确目的 + 申请后才允许"由系统强制而非约定。

**⑤ 统一分页**（`DB_PAGINATION_MANDATORY`）— 分页工具定义于 postern-core，是全部集合查询的唯一形态：

```rust
pub struct PageQuery { pub page_no: u32, pub page_size: u32 }   // page_no 从 1 起
pub struct Page<T> { pub items: Vec<T>, pub page_no: u32, pub page_size: u32, pub total: u64 }

impl PageQuery {
    pub const DEFAULT_SIZE: u32 = 20;
    pub const MAX_SIZE: u32 = 200;       // 全局上限:超出钳制到上限,不报错
    pub fn clamp(self) -> Self { /* page_no>=1, page_size ∈ [1, MAX_SIZE] */ }
}
```

规则：store 集合查询函数必须接收 `PageQuery`，SQL 必须 `LIMIT ? OFFSET ?`；控制面全部集合端点必须携带 `page_no/page_size` 参数（缺省取默认值）并返回 `Page<T>` 信封；审计 JSONL 扫描查询同样按分页窗口截断。禁止无界查询、禁止"全量取回内存再切片"、禁止把分页交给前端。

**⑥ 统一雪花 ID 生成器**（`DB_UNIFIED_ID_GENERATOR`）— `postern-core::id::IdGen`：41 bit 毫秒时间戳（纪元 2026-01-01T00:00:00Z）+ 10 bit 节点号（config，默认 0）+ 12 bit 序列号；时钟回拨时拒绝生成（fail-closed，绝不产出可能重复的 id）。全部表主键与审计事件 id 都取自它；工作区依赖封禁 uuid/ulid/nanoid 等替代 id 库。JSON 序列化时雪花 id 一律为**字符串**（避免 JS 53 bit 精度丢失）。

**⑦ 契约与反例（teeth）** — 上述每条规则对应两条 Stele 契约：**规则本体**（扫描真实代码，违规计数必须为 0）+ **反例自检**（同一扫描器跑在内嵌的已知违规夹具上，必须检出 ≥1 条；检不出 = 扫描器失效 = 契约测试立即失败）。反例常驻于契约测试内部，保证规则永远"有牙齿"而非空转通过。契约总数为 **24 条（12 规则 × 本体 + 反例）** —— 7 条数据库层规则（本节 ①~⑥ 对应的 `DB_*`）与 5 条安全规则（见 7.3）各带本体与反例自检；12 规则（7 数据库层 + 5 安全核心）均经受限通道提案至 `contract/proposals/agent-additions.stele`，接线实现规范见 7.6。

### 5.2 policy.db schema（SQLite，WAL，权威策略状态）

约定：**每张表都含 5.1-① 的 8 个基础字段**（下文只列业务字段，不再重复）；时间一律**固定宽度** UTC ISO-8601 文本（精度钉死）；JSON 列由对应插件定义并校验语义；`PRAGMA foreign_keys=ON`；schema 版本走 `PRAGMA user_version`；启停语义由 `enable_flag` 承担（授予性表；限制性表禁用 enable_flag），不再有 disabled_at 列。

基础字段的完整形态以 `principals` 为例展示一次：

```sql
CREATE TABLE principals (
  -- 统一基础字段(全表一致,契约 DB_BASE_FIELDS_REQUIRED)
  id          INTEGER PRIMARY KEY,              -- 雪花 ID(IdGen)
  version     INTEGER NOT NULL DEFAULT 0,       -- 乐观锁
  created_at  TEXT    NOT NULL,
  created_by  TEXT    NOT NULL,
  updated_at  TEXT    NOT NULL,
  updated_by  TEXT    NOT NULL,
  delete_flag INTEGER NOT NULL DEFAULT 0,
  enable_flag INTEGER NOT NULL DEFAULT 1,
  -- 业务字段
  name        TEXT NOT NULL,                    -- 具名主体(无全局匿名身份,3.1)
  kind        TEXT NOT NULL CHECK (kind IN ('agent','program','human'))
);
CREATE UNIQUE INDEX uq_principals_name ON principals(name) WHERE delete_flag = 0;
```

各表业务字段与约束：

| 表 | 业务字段（基础 8 字段省略） | 约束/索引要点 |
|---|---|---|
| `principals` | name, kind | name 归一化后未删除集内唯一（归一化策略） |
| `credentials` | principal_id, kind(local_process/api_key/token), secret_hash(argon2id，明文永不落库), match_spec, trust_domain, expires_at, revoked_at | revoked_at 是领域语义（吊销≠删除）；expires_at/revoked_at 在求值时刻按墙钟二次校验（见 6.2）；index(principal_id)；enable_flag 取舍 |
| `roles` | name, description | `CHECK (lower(trim(name)) <> 'admin')`（模型层硬约束，防大小写/空白绕过）；name 归一化后未删除集内唯一 |
| `role_inherits` | role_id, parent_role_id | (role_id,parent_role_id) 未删除集内唯一；应用层校验无环 |
| `role_capabilities` | role_id, capability(6 动词 CHECK), action(allow/escalate) | (role_id,capability) 未删除集内唯一 |
| `resources` | codename, adapter, transport, transport_config(JSON，仅非敏感；敏感项 vault:// 引用) | codename 归一化后未删除集内唯一（归一化策略）；**本库不存任何真实地址** |
| `resource_labels` | resource_id, key, value | (resource_id,key) 未删除集内唯一；标签如 `host:A`/`env:prod`/`kind:business`，供 Scope 按选择器展开（见 §授权矩阵与标签 Scope） |
| `resource_credential_tiers` | resource_id, tier, capabilities(JSON 动词数组), secret_ref(vault://), auth_flow(JSON，仅非敏感的认证流程配置；敏感项 vault:// 引用) | (resource_id,tier) 未删除集内唯一；无匹配 tier → deny；`auth_flow` 承载会话来源配置（认证端点/流程类型/注入方式/刷新策略/是否需 2FA，见 6.13） |
| `bindings` | principal_id, role_id | index(principal_id)；(principal_id,role_id) partial unique（`WHERE delete_flag=0`） |
| `binding_scope` | binding_id, kind(`resource`/`selector`), resource_id(kind=resource 时), selector(JSON，kind=selector 时) | 枚举式（指定资源代号）或选择器式（按标签匹配，如 `{all:[{key:'env',value:'prod'}]}`）二选一；选择器在快照构建时展开为具体资源集（见 §授权矩阵与标签 Scope） |
| `grant_constraints` | resource_id, capability, kind, spec(JSON) | `kind` ∈ {`table_allow`,`column_mask`,`container_prefix`,`http_route`(method+path),`command_template`,`script_template`,`path_whitelist`,`key_prefix`,`command_class`,`vhost_allow`,`queue_prefix`,`route_allow`,`mask_fields`}，**由对应 Adapter 声明支持哪些**；index(resource_id,capability)；`CHECK (enable_flag=1)`（限制性表禁 enable_flag，见 5.2） |
| `grant_conditions` | resource_id(NULL=全局), capability(NULL=全动词), predicate, spec(JSON) | `CHECK (enable_flag=1)`；NULL 列（全局/全动词）无唯一性是有意设计（见下） |
| `temp_grants` | principal_id, resource_id, capability, granted_at, expires_at, ended_at, end_reason | index(principal_id,expires_at)；终态；enable_flag 取舍 |
| `mode_state` | scope_resource_id(NULL=全局), mode(normal/observe/maintain/freeze CHECK), expires_at | 唯一索引作用于 `COALESCE(scope_resource_id, 0)`；`CHECK (enable_flag=1)`；无 set_at（由基础字段 updated_at 承担） |
| `deny_notes` | resource_id, capability, note(人亲笔预写，公理六) | (resource_id,capability) 未删除集内唯一；`CHECK (enable_flag=1)` |
| `settings` | key, value（审批开关/audit.fsync/保留期等） | key 未删除集内唯一；`CHECK (enable_flag=1)` |

唯一性约束统一采用 **partial unique index**（`WHERE delete_flag = 0`）：逻辑删除后同名记录可重建，历史行保留可追溯。

**全局辖区的唯一性兜底（fail-closed）**：`mode_state` 的 `scope_resource_id IS NULL` 表示全局辖区；SQLite 中 `NULL` 不参与唯一索引，朴素的 `WHERE delete_flag=0` partial unique index **对全局行不生效**，致全局模式可多行并存——freeze 可被另一行 normal 旁路。修正：

- 唯一索引改作用于 `COALESCE(scope_resource_id, 0)`（雪花 id 恒正，`0` 作全局哨兵），使全局辖区也唯一。
- 6.2 快照构建固化 **fail-closed 兜底**：万一同辖区仍出现多行生效模式，取**最严格者**（`freeze > maintain > observe > normal`）并写告警审计，绝不取最宽松者。
- `grant_conditions` 的 NULL 列（`resource_id IS NULL` 全局、`capability IS NULL` 全动词）**无唯一性约束是有意设计**——同一全局条件允许多行并存（多行取交集），故注明而非"修复"。

求值时的"当前生效授权" = `bindings×role_capabilities(含继承展开)` ∪ `temp_grants(expires_at >= now 且 ended_at 为空)`，再与 `mode_state` 求交、过 `grant_constraints`/`grant_conditions`——求值是纯内存查表。快照加载条件按表的语义分两类（见 6.2 与下文"限制性表禁用 enable_flag"）：

- **授予性表**（principals/roles/bindings/role_capabilities/resources/resource_credential_tiers 等）：加载 `delete_flag = 0 AND enable_flag = 1`——停用即收回授权，与 fail-closed 一致。
- **限制性表**（grant_constraints / grant_conditions / mode_state / deny_notes）：加载条件**仅** `delete_flag = 0`，**不引入 enable_flag 过滤**（见下）。

**限制性表禁用 `enable_flag`（fail-closed）**：对限制性表，`enable_flag=0` 会使一行约束/条件/冻结被"停用"，等价于**扩权或解除冻结**——这是 fail-open，违公理一、公理二。因此：

- grant_constraints / grant_conditions / mode_state / deny_notes 四表**禁止使用 `enable_flag`**：建表加 `CHECK (enable_flag = 1)`，基础仓储拒绝对这些表写入非 1 的 `enable_flag`。
- 解除一条细则/条件/冻结，只能经**显式删除**（`delete_flag=1`）或**显式 `mode set`**，二者各写专门的审计事件（`policy_change`/`mode_change`），留下"是谁、何时、解除了哪条限制"的可见痕迹。停用一条限制绝不能是一次悄无声息的 flag 翻转。

**级联逻辑删除与引用链可见性（fail-closed）**：`PRAGMA foreign_keys=ON` 约束物理删除，但本库只逻辑删除——父行逻辑删除后子行仍 `delete_flag=0`，会成为**生效中的悬挂引用**（如 resource 逻辑删除后其 credential_tiers / constraints 仍被快照加载）。两道防护：

- **级联逻辑删除规则**（5.1-③ 增补）：父表逻辑删除时，在**同一事务**内级联把直接子行置 `delete_flag=1`，`updated_by` 标注级联来源（如 `cascade:resources#<id>`）。逐表列出级联图：`resources → {resource_credential_tiers, binding_scope, grant_constraints, grant_conditions, mode_state(scope_resource_id), deny_notes}`；`principals → {credentials, bindings, temp_grants}`；`roles → {role_inherits, role_capabilities, bindings}`；`bindings → binding_scope`。
- **快照构建兜底**（6.2 固化）：引用链上任一父行不可见（`delete_flag=1` 或不存在）⇒ 该行不可见——快照构建按引用链可见性过滤，即便级联遗漏也不会放行悬挂引用（fail-closed）。
- **不提供 undelete 接口**：逻辑删除是终态，重建 = 新增同名新行（partial unique 允许）；不暴露 undelete，避免"复活"绕过审计与级联语义。

**乐观锁的端到端语义**：`version` 只在 base 仓储内部自读自写比对会**恒成立**，等于没有并发保护。修正三点：

- **控制面读端点统一返回 `version`**；更新/删除端点**必须携带期望 `version`**，不匹配返回 `409 Conflict` 并写审计（见 6.5）。
- **区分两类写**：①"用户意图写"——必带期望 version，参与乐观锁；②"系统协调写"——sweeper 按谓词幂等更新（如"把所有 `expires_at < now` 的行回收"），不参与乐观锁，文档写明理由（系统写是幂等谓词驱动，无"读后写"竞态）。
- **期望 version 唯一来源是调用方读取值**（5.1-② 增补）：base 仓储**不得自读自比**——期望 version 由控制面调用方从其先前读取的 `Page<T>`/单条响应中带入。

**temp_grants 终态字段**：到期回收若复用 `revoked_at` 会混淆"自然过期"与"人工吊销"。`temp_grants` 用独立终态字段：

- `ended_at TEXT` + `end_reason TEXT CHECK (end_reason IN ('expired','revoked'))`——sweeper 写 `expired`，人工写 `revoked`，二者语义分明、审计可区分。
- 因 6.2 已保证"过期对求值即刻不可见"（求值时刻墙钟校验），sweeper 的回收只为留痕与打扫，不影响正确性。
- **temp_grants 与 credentials 的 `enable_flag` 取舍**：二者均**禁用 `enable_flag`**（`CHECK (enable_flag=1)`），其生效与否由领域字段判定——temp_grants 看 `expires_at`/`ended_at`，credentials 看 `expires_at`/`revoked_at`；判定优先级与各自审计类型（`policy_change`/`credential_event`）写明，避免 enable_flag 与领域终态字段双轨混淆。

**时间列精度钉死（字典序可比）**：时间列若混合精度（有的带毫秒、有的不带），文本字典序比较会错序，直接影响 TTL/sweeper 的 `< now` 判定。钉死契约：

- 固定格式 `YYYY-MM-DDTHH:MM:SS.sssZ`——**恒 UTC、恒 `Z` 后缀、恒 3 位毫秒、长度恒 24**。
- 唯一生成点是 base 仓储（与审计 `ts` 共用同一格式化函数，全工作区一致）。
- 时间列加 `CHECK (length(col) = 24)` 作格式兜底。
- "时间文本单一格式"纳入数据库层契约扫描的设想（与 5.1 契约族同源）。

**settings 表语义**：

- 语义钉死：**行不存在或已逻辑删除 = 取内建默认**；本表禁用 `enable_flag`（`CHECK (enable_flag=1)`）。
- **已知 key 注册表**（类型 / 取值域 / 内建默认）：`approval.enabled`（bool，默认 `false`——审批默认关闭）、`approval.on_timeout`（enum，固定 `deny`，不可配 allow）、`audit.fsync`（enum `per_event|relaxed`，默认 `per_event` 逐事件）、`audit.retention_days`（int，有界安全默认，见 5.3防护）。**安全攸关 key 的内建默认必须取最严格值**。
- daemon 启动时校验 settings：出现未知 key 即**拒绝加载**（fail-closed），不静默忽略。

**mode_state 切换语义**：

- 删除冗余 `set_at`——切换时刻由基础字段 `updated_at` 承担。
- **辖区无生效行 = `normal`**（无需为 normal 显式存行）。
- 切换 = 同辖区行的**事务内 UPSERT**（原地 `UPDATE`）；`delete_flag` 仅用于撤销某辖区级覆盖（回落到上层默认）。模式历史查 `mode_change` 审计，不在本表留多行。

**模式集合与覆盖规则（CONS-18，系统内置）**：模式集合**固定**为 `normal / observe / maintain / freeze`，是系统内置概念，不可由策略新增模式。

- 各模式对动词的覆盖规则（如 `observe` 只放行只读动词、`freeze` 拒绝一切、`maintain` 限定维护类）是**系统内置**的固定语义，**只有 `default`（缺省模式）可由声明指定**（见 6.6 `[modes].default`）。
- 覆盖规则固化于 **core 内的常量表**；求值步骤 [5] 的 `mode` 谓词引用该常量表判定，不从策略库读取"模式定义"——模式语义不漂移、不可被数据扩展。

**入库归一化**：`roles.name` 的 `CHECK (name <> 'admin')` 可被 `Admin`、` admin `（含空白）绕过。修正：

- `roles.name` 用 `CHECK (lower(trim(name)) <> 'admin')`；name 入库前由 base 仓储统一**归一化**（`trim` + 明示的大小写策略）；唯一索引作用于归一化值。
- `principals.name`、`resources.codename` 一并写明归一化策略（统一 `trim`、大小写策略明示），唯一索引作用于归一化值。

**约束合并语义**：

- `bindings` 加 `(principal_id, role_id)` partial unique（`WHERE delete_flag=0`），防同一绑定重复。
- `grant_constraints` **同格（同 resource+capability）多行合并**：推荐**同 `kind` 多行取交集**（全部须满足，fail-closed）；不同 `kind` 之间恒 `AND`。若某 kind 选"白名单并集"语义（多行取并集），须在该 kind 的 spec 文档显式写明理由与其扩权特性（默认不取并集，因并集是放宽方向）。

### 5.2bis 授权模型落地：角色阶梯、标签 Scope、资源建模、适配器细则矩阵

本节把"如何为异构资源类型灵活分配权限"落到 schema 与约定上。授权是一个网格 `(信任等级 Role) × (辖区 Scope) × (动词 Capability) × (对象细则 constraint) × (条件 condition) × (引擎账号 tier)` 上的点；下面逐轴定清。

**① 标准角色阶梯（资源类型无关，靠继承）**：角色只描述动词集、与具体资源类型无关，动词的含义由各 Adapter 解释（`observer` 给 DB 是 SELECT、给 redis 是 GET、给 docker 是看日志、给 HTTP 是 GET）。推荐内置阶梯（经 `role_inherits` 显式继承）：

| 角色 | 动词集（含继承） | 含义（对任何资源类型成立） |
|---|---|---|
| `observer` | observe, query | 只读一切：看日志/状态、读数据/配置/页面 |
| `operator` | observer + mutate, execute | 改数据、提交表单、跑受控脚本/命令模板（不碰生命周期） |
| `maintainer` | operator + manage | 启停/部署/重启/扩缩容 |
| `admin` | —（保留） | **不可授予任何 Principal**（`roles.name` CHECK + Capability 无 Admin 变体双重硬约束） |

`destroy` 不进任何标准角色，一律按 `(资源 × destroy)` 单格 + 默认 TTL 显式授予。差异化不靠"为每种资源造角色"（组合爆炸），而靠 Scope（哪些资源）+ constraint（哪些对象）+ tier（哪个引擎账号）。

**② 标签 Scope（grouping/层级感的来源）**：`binding_scope` 支持两种 `kind`——
- `resource`：枚举具体资源代号（精确、少量资源时直观）。
- `selector`：按 `resource_labels` 标签匹配，`spec` 形如 `{all:[{key:'env',value:'prod'},{key:'kind',value:'business'}]}`（`all`=与，可扩展 `any`=或）。

选择器在**快照构建时**展开为当时匹配的具体资源集（展开结果进快照，求值仍是纯内存查表）。新增一台同类资源只要打对标签即自动纳入对应授权——"某机器全部权限""某环境""某类资源"都成了一句标签选择，与"角色资源无关"原则正交。**fail-closed**：标签语法无法解析或选择器展开为空集 → 该绑定不授予任何资源（空集，不报错但不放行）。

**③ 同一物理服务 → 多资源（用 vs 管）**：一个资源 = (代号 + Adapter + Transport + tier)。同一物理服务的不同操作用不同 Adapter/Transport/凭据，故拆成多个资源：
- **用 app**（调接口、看页面、提交表单）：`svc-x`（http 适配器 + 业务系统账号 tier，见 6.13 会话来源），动词 observe/query/mutate/destroy 作用于应用数据，细则 `http_route` 收窄到具体接口/页面。
- **管 app**（重启/扩缩/部署）：`docker-x` 或 `deploy-x`（docker/command 适配器 + 基础设施凭据），动词 manage/destroy。
- **看 app 日志**：`docker-x` 的 observe（容器日志），或远端探针（6.12）。

一个 Agent 可能"能用 `svc-x`（query）但不能重启它（`docker-x:manage` 未授）"——精确正交。

**④ 适配器能力与细则矩阵（每类资源支持的动词与细则 kind，由 Adapter 声明）**：

| 资源类型 | adapter | engine_enforced | 支持动词 | 支持的细则 kind | 引擎兜底（tier→账号） |
|---|---|---|---|---|---|
| 业务系统(应用层) | http | 取决于业务系统 RBAC（声明） | observe/query/mutate/destroy | `http_route`(method+path)、`mask_fields` | 业务系统账号（只读/普通/管理，见 6.13） |
| 关系数据库 | postgres/mysql | true | observe/query/mutate/destroy | `table_allow`、`column_mask` | SELECT-only / 读写账号 |
| redis | redis | redis 6+ ACL=true；旧版=false（如实声明） | observe/query/mutate/manage/destroy | `key_prefix`、`command_class` | 只读 ACL / admin ACL；旧版无 ACL 则归类是唯一防线 |
| 消息队列(rabbitmq) | rabbitmq | true（vhost/user 权限：configure/write/read） | observe/query/mutate/manage/destroy | `vhost_allow`、`queue_prefix`、`route_allow` | 只读监控账号 / 普通账号 / 管理账号 |
| docker | docker | true（账号即 socket 权限） | observe/manage/destroy | `container_prefix` | 只读日志 API / 全 docker API（**裸 docker.sock 即 root，observe 须走只读端点**） |
| 系统部署 | command/deploy | true（OS 账号权限） | observe/query/execute/manage/destroy | `script_template`、`path_whitelist`、`command_template` | 部署账号（受限 sudo / 普通） |
| 机器操作 | command(shell) | true（OS 账号权限） | observe/execute | `command_template`（参数化，无自由命令） | 受限 OS 账号 |

`execute` 类（部署、机器操作）**只接受参数化模板的实例化，绝不接受自由命令文本**——"某些脚本"=模板白名单，"某些目录"=模板里被约束的参数（`path_whitelist`）。这是高危动词能安全开放的前提（技术设计 10.3）。`engine_enforced=false` 的资源（如旧版 redis）须在能力声明与文档如实标注"归类+细则是唯一防线"。

### 5.3 审计事件流（JSONL，按日轮转）

**量级核算**（D2 依据）：单 Agent 高频操作约 10⁴ 事件/日、每事件 ≤1 KB → ≈10 MB/日；拒绝分析典型窗口 7 日 ≈70 MB，顺序扫描 + serde_json 解析在百毫秒级——SQLite 索引在此量级无实质收益，而 JSONL 换来载体物理 append-only、`tail -f` 实时观测、零 schema 迁移、导出器即"读文件"。

```
<data_dir>/audit/2026-06-11.jsonl      # 按 UTC 日轮转,文件只追加
```

事件信封（schema 版本化）：

```json
{"v":1,"id":"72683915651304704","ts":"2026-06-11T08:30:00.123Z","kind":"request",
 "entry":"mcp","origin":{"unix_peer":{"uid":1001,"gid":1001}},
 "principal":"claude-main","principal_id":"72683915651300001",
 "credential_id":"72683915651304705","resource":"db-main","resource_id":"72683915651302002",
 "capability":"query","objects":["public.orders"],"decision":"deny",
 "stage":"rbac","reason":"role=observer 不含 db-main:mutate",
 "policy_rev":42,"intent_digest":"sha256:...",
 "tier":null,"duration_ms":2,"response_digest":null}
```

**信封内一切 id 均为雪花 id 的字符串序列化**（/CONS-15）：`id`、`principal_id`、`resource_id`、`credential_id` 等都是雪花 id 字符串（如 `"72683915651304705"`），不是 ULID/UUID（早期示例中的 `01JX...` ULID 形态与雪花 ID 契约矛盾，已纠正）；`principal`/`resource` 等保留为便读名字字段。

新增信封字段：

- `principal_id` / `resource_id` / `credential_id`：对应实体的雪花 id 字符串（名字字段并存，便读 + 可对账）。
- `entry`：外壳类型（`mcp` / `http`），区分请求入口。
- `origin`：`ConnOrigin` 的序列化——属网关侧可观测事实（不返回 Agent，不违公理四）。
- `policy_rev`：快照重建时递增的策略修订号，写入 `policy_change` 与 `request` 事件，支撑"相同请求 + 相同策略 → 相同决策"的可复现对账。
- `intent_digest`：脱敏后意图摘要（classify 失败、无 capability 的 deny 也能聚合归类）。
- `decision` 取值集合显式含 `escalate_denied`（审批关闭/超时折叠为 deny 的子类，与普通 deny 可区分）。

| kind | 触发 | 附加字段 |
|------|------|----------|
| `request` | 每个数据面请求（allow 与 deny 都记，deny 含 stage+reason；有副作用动词另见 6.1 两阶段 intent/outcome） | 见上；allow 附 `tier`、`response_digest`（sha256，不含内容） |
| `policy_change` | 任何策略状态写入（人为或 sweeper 自动回收） | actor、`policy_rev`、**实体表名 + 行 id + 写后 version**（可与权威库逐行对账）、变更摘要（旧值→新值，凭据值除外） |
| `credential_event` | 签发/轮换/吊销/可信域修改 | credential_id、操作 |
| `connection_event` | 通路建立/健康剔除/回收 | resource、tier 名、transport |
| `mode_change` | 模式切换/到期回退 | 旧→新、scope、ttl |
| `lifecycle` | daemon 启动/就绪/停止/保险箱解锁 | 阶段 |

写入策略：`deny`、`policy_change`、`credential_event` 逐事件 fsync；`allow` 类默认逐事件 fsync，可经 `settings: audit.fsync=relaxed` 改为 1s 周期批量（自我观测场景的性能阀）。审计内容永不含凭据值与真实地址（写入前过同一 Sanitizer）。

**审计存储健康与拒绝服务防护**：朴素的"审计写失败 = 全局 deny"会构成 Agent 可触发的全局拒绝服务环——刷满磁盘即令所有请求 fail-closed 瘫痪。为在 fail-closed 与可用性间取得诚实平衡，审计存储按以下设计区分两类故障并加防护：

- **存储健康 vs 单条写入失败**：明确区分"审计存储健康"（容量、可写、轮转正常）与"单条写入失败"（瞬时 IO 错误）两类故障——前者是平面级状态，后者是请求级事件，二者降级路径不同。
- **独立配额与水位监控**：审计落盘有独立于其他数据的存储配额与水位监控。逼近上限时触发**可感知的降级**（如告警事件、强制轮转、对低价值审计降采样），而非让整个数据平面瞬间瘫痪。
- **deny 类审计去重/限流**：同 Principal、同类的 deny 事件按窗口聚合计数（写一条带 `count` 的聚合记录），避免可被 Agent 反复触发的 deny 把审计存储打满。
- **保留期有界默认**：`audit.retention_days` 取有界的安全默认值，并设强制轮转/回收上限（不再以"默认不删除"为缺省，避免无界增长成为 DoS 面）。到期文件整体删除。

去重、降级、轮转本身均写入可见的审计/告警记录，"口子自动愈合"对运维可见。

### 5.4 机密保险箱（vault 文件格式）

```
<data_dir>/vault.postern
┌──────────────────────────────────────────────┐
│ magic "PSTRN" + format_version u8            │
│ kdf-wrapped data-key 槽(单槽起步,预留多槽):    │  ← 明文头(无敏感信息)
│   slot[i] = { source, kdf_params?, salt?,    │
│              nonce_i, wrapped_data_key }      │
│ nonce (24B)                                  │
│ ciphertext = XChaCha20-Poly1305(payload)     │
└──────────────────────────────────────────────┘
明文头全段作为 ciphertext 的 AEAD AAD(防头部篡改/降级)
payload(JSON,仅存在于解锁后的进程内存,Zeroizing):
{
  "secrets": { "db-main/readonly": {"user":"ro","password":"..."},
               "db-main/readwrite": {...}, "app-prod/ssh": {"key_pem":"..."} },
  "targets": { "db-main": {"host":"10.0.3.17","port":5432},
               "app-prod": {"instance_id":"i-0abc...","region":"..."} }
}
```

- `secrets` 键即 `vault://` 引用路径；`targets` 即代号↔真实地址映射表（技术设计文档 11.3）。
- 每次写入整体重加密（条目量级为个位数到百级，成本可忽略）。

**解锁材料来源的真实强度（诚实表述）**：`payload` 由一个随机生成的 **data-key** 加密；data-key 被各解锁来源**包裹**进明文头的槽。`MasterKeySource` 实现（config.toml 选定其一）的强度各不相同，不可一概宣称"argon2id 保护"：

| 来源 | 主密钥获取方式 | 真实强度 | 适用形态 |
|---|---|---|---|
| `passphrase` | argon2id KDF 派生（**KDF 仅作用于 passphrase 来源**） | 取决于口令熵 + argon2id 参数 | 有人值守（启动交互输入） |
| `key_file` | 直接持有 32 字节主密钥 | **等于文件系统权限**（与无口令的 SSH 私钥同级），无 KDF 加固 | 无人值守 |
| `os_keychain` | 直接持有 32 字节主密钥 | 等于 OS 钥匙串保护强度 | 桌面外壳桥接 |
| `systemd_cred` | 直接持有 32 字节主密钥 | 等于 systemd 凭据保护（可走 TPM 封存） | 无人值守、服务化 |

- **key_file 不作为简单默认推荐**：其弱模式（强度仅靠文件系统权限）的威胁后果须显式声明——任何能读到 key_file 的主体即可解锁全部凭据。加固选项：`systemd_cred` 走 TPM 封存；key_file 与 vault 置于**不同 uid/挂载**且仅 daemon 专用账户独占可读。
- **passphrase 与无人值守的矛盾调和**：`passphrase` 仅适用于有人值守场景——常驻 daemon 重启需无人干预即可解锁，与启动交互输入口令互斥。无人值守常驻应选受保护的自动解锁来源（`systemd_cred`/TPM、或受 OS 钥匙串保护的 `os_keychain`），不要用 passphrase 配合明文缓存口令的变通。

**vault 写入协议与格式演进**：

- **原子写入**：写临时文件 → `fsync` → 原子 `rename` 覆盖；保留上一代 `.bak`（避免写中途崩溃损坏 vault）。
- **随机 data-key + 包裹槽**：主密钥不直接加密 payload，而是包裹随机 data-key；改 passphrase 或换 KDF 只需重包裹 data-key，无需整体重加密换密钥语义。单槽起步，结构预留多槽。
- **nonce 纪律**：每次写入随机重生成 nonce，**绝不复用**（XChaCha20-Poly1305 的 24B nonce 空间使随机生成安全）。
- **AAD 绑定**：明文头全段作为 AEAD 的 AAD，头部任何篡改（含 `format_version` 降级）使解密失败。
- **格式版本 fail-closed**：`format_version` 不被当前实现识别时**拒绝解锁**，绝不尝试按旧假设解析。
- **rekey / rotate-kdf**：作为控制面操作（重包裹 data-key / 换 KDF 参数），完成后写 `lifecycle`/`credential_event` 审计。

### 5.5 磁盘布局与文件权限

```
$XDG_CONFIG_HOME/postern/config.toml       # daemon 自身运行参数(监听/目录/解锁来源);策略不在这
$XDG_DATA_HOME/postern/
├── policy.db                              # 0600
├── vault.postern                          # 0600
└── audit/*.jsonl                          # 0600
$XDG_RUNTIME_DIR/postern/
├── control.sock                           # 0600(仅属主) ← Agent 沙箱用户不可连
└── data.sock                              # 0660 或专用组   ← Agent 可连的唯一入口
```

控制面/数据面隔离落地为**两个 socket 的属主与权限位差异**；策略/审计/机密文件属主即用户本人，Agent 进程（按部署要求运行在隔离侧）无读路径。

**硬前置条件 ① — `data_dir` 对 Agent uid 不可读**：`policy.db`、`vault.postern`、`audit/*.jsonl` 三者对 Agent 进程所属 uid **必须不可读**（属主为 daemon 运行账户，0600）。这是凭据零接触与策略单一权威的存储层前提，非"建议"。

**硬前置条件 ② — Agent 与 daemon 必须分属不同 uid**：`control.sock=0600` 只对**不同 uid**的进程设防；对**同 uid**进程不设防。单机桌面形态若 daemon 与 Agent 同 uid，Agent 即可直连 `control.sock` 获取全部 admin 能力，瓦解两平面隔离。因此：

- **Agent 必须运行于独立于 daemon 的 uid（或更强隔离）** 是**硬性部署前置条件**，不是可选加固。桌面形态落地：daemon 以专用服务账户运行，Agent 以登录用户运行（二者 uid 不同）。
- **启动自检 fail-closed**：`posternd` 启动时检查 `data.sock` 的可连 uid 集合；若该集合包含 daemon 自身 uid，则 **fail-closed 拒绝启动**（或在明知风险的形态下强制告警），杜绝"Agent 与 daemon 同 uid 还能跑"的静默不安全状态。
- **control.sock 叠加控制面认证**：即便同 uid 也成立的控制面认证——`SO_PEERCRED` uid 比对 + 控制面专用本地凭证，使"恰好同 uid"不等于"自动获得 admin"。
- **postern verify 项**：`postern verify` 增加"从模拟 Agent uid 连 `control.sock` 必须失败"的红队项。

## 第六部分 · 关键机制设计

### 6.1 求值管线 → 代码映射

| 管线步骤(技术设计文档 5.1) | 承载代码 | 失败语义 |
|---|---|---|
| [0] 外壳归一化 | daemon `shells/*` → `NormalizedRequest` | 形态不合法→经同一 Sanitizer 的外壳层 4xx(常量安全文案,见 6.1·错误路径不变量) |
| [1] 认证+可信域 | `Evaluator` ← `Authenticator` 注册表 | Err→deny |
| [2] 语义归一化 | `Adapter::classify` | Err→deny |
| [3] RBAC | `Evaluator` × `PolicySnapshot` 查表 | 格子不存在→deny |
| [4] 细则 | kernel 先跑 `Adapter::check_constraint`,结果入参传入 `evaluate`(见 4.1/CONS-8) | false/Err→deny |
| [5] 条件谓词 | `ConditionPredicate` 注册表逐一求值 | 任一 false/Err→deny |
| [6] 动作分流 | `Decision::{Allow,Escalate}`;审批关→Escalate 取 fallback | 超时恒 deny |
| [7a] 意图审计 | 对有副作用动词(mutate/execute/manage/destroy)先落 **intent** 审计事件 | intent 写不进→执行前 deny(此时确未执行) |
| [7b] 取连接 | `connpool` 按 `(resource, tier)` 取/建 | 不可建→deny |
| [8] 执行 | `Adapter::execute` over `Channel` | 错误经脱敏后返回;已执行的请求绝不返回 deny |
| [9] 脱敏 | `Sanitizer::scrub`(含错误与拒绝响应) | — |
| [10] 结果审计 | `AuditSink::record` 落 **outcome** 事件 | 见下"审计相对执行的时序不变量" |

**审计相对执行的时序（显式不变量）**：审计相对执行的先后顺序是安全不变量，不是实现细节。

- **只读动词（observe/query）** 无副作用，沿用执行后单次审计：`AuditSink::record` 写失败→该请求按 deny 返回（不可记录 = 不放行；尚未对外暴露任何已落库副作用）。
- **有副作用动词（mutate/execute/manage/destroy）** 采用**两阶段审计**：
  - 执行前先落 **intent** 审计事件（步骤 [7a]）。intent 写不进 → 执行前 deny —— 此时**确未执行**，deny 既诚实又安全（公理二、公理六）。
  - 执行后落 **outcome** 审计事件（步骤 [10]）。
  - **已执行的请求绝不返回 deny**：破坏性写一旦落库，再返回 deny 既造成"无审计痕迹的已执行操作"，又是谎报，会诱导 Agent 重试导致重复执行（违公理六）。outcome 审计写失败时返回明确的"已执行但审计降级"错误（带可识别错误码），而非 deny。
  - 由 intent 事件与 outcome 事件以同一请求 id 关联，可对账"发起即有痕、结果可追溯"。

### 6.2 策略热生效（快照机制）

```
控制面写入(事务) ──┐
sweeper 回收(事务) ─┼─→ COMMIT 成功 → 同一写锁内重建 PolicySnapshot(Arc 原子替换)
import/apply ──────┘
数据面求值:Arc<PolicySnapshot> 无锁读;TTL 在求值时刻按墙钟二次校验
```

- 快照在一次事务内全量加载并完成角色继承展开、授权空间物化——求值零库访问、微秒级。
- 正确性不依赖 sweeper 时序：过期判定以求值时刻 `expires_at < now` 为准（即刻失效）；sweeper 只负责把过期记录事务性回收并写审计（口子自动愈合的可见记录）。
- "单一权威状态"语义保持：快照是权威库的原子投影，重建与写入在同一临界区，不存在双源。

**时间到期的二次校验（显式不变量）**：快照重建是**事件驱动**的（控制面写入 / sweeper 回收 / import 触发），不随墙钟自动推进。因此一切"时间到期"语义都不能只靠快照新鲜度，必须在**求值时刻按墙钟二次校验**或由独立 sweeper 兜底：

- `temp_grants.expires_at`、`credentials.expires_at`/`revoked_at`、可信域时效、`mode_state.expires_at` 均在求值时刻以 `< now` 二次校验——`Evaluator::evaluate` 与 `Authenticator::authenticate` 都显式接收 `now`（见 4.1）。
- 这一不变量覆盖**安静系统**的盲区：在长时间无任何写入（快照不重建）的系统上，过期凭证若只靠快照过滤会长期有效；求值时刻墙钟二次校验使其**即刻失效**。
- sweeper 覆盖范围相应扩展到 `credentials.expires_at`，把过期凭证回收并写审计；但回收只是"打扫"，正确性由求值时刻校验保证，不依赖 sweeper 已经跑过。

**freeze/吊销与在飞请求的中断语义（显式界定）**：freeze/revoke 经快照原子替换即时生效，但须诚实区分"对谁生效"：

- **对尚未开始执行的后续请求**：即时生效——下一次 `evaluate` 读到新快照即拒绝。
- **对已进入执行阶段的在飞操作**：快照替换**不保证中止**正在底层执行的查询/隧道。优雅排空只是"不再接新"，不等于"砍断在飞"。
- **真正的中断手段**：freeze/吊销时，连接管理层对相关 `(resource[, principal])` 的在用连接**强制 abort/cancel**（取消底层查询、关闭隧道），而非仅优雅排空。
- **执行阶段的可打断性**：对长操作设置可被控制面信号打断的检查点或超时，使"已开始的危险操作"存在可控的中止路径。

### 6.3 凭据分级与连接管理

```
Allow{tier} → connpool.acquire(resource, tier)
  池键 = (ResourceCode, CredentialTier)
  ├─ persistent transport:池化复用;健康检查;指数退避重连(基数 1s,上限 60s,带抖动)
  ├─ 非长连接 transport:即建即用即弃,不入池
  ├─ 每资源/全局并发上限,超限排队(有界)或 deny(fail-closed)
  ├─ 空闲回收(默认 10min)与优雅销毁(排空在途请求)
  ├─ 归池前会话净化(见下不变量)
  └─ 建立时:vault 取 (target, credential[tier]) → Transport::open → 凭据引用即时释放
```

不同 tier 不共享连接（一条只读连接绝不会被升格执行写操作——账号隔离在连接粒度成立）。

**tier 声明权限 ⊆ 底层账号真实权限（接入时主动校验）**：D5 的"引擎层兜底"成立的前提是 tier 声明的动词集确实是底层账号真实权限的子集；这是**需被验证的前提**，不自动成立。资源接入时主动校验：连上账号后探测其实际权限（如 PostgreSQL 角色的 `has_table_privilege`/默认权限），与 tier 声明的 capabilities 比对——不符则拒绝接入或降级 tier。该校验纳入 `postern verify` 常规项（一个声明 `readonly` 的账号若实际拥有写权限，校验须报缺口）。

**归池前会话净化（不变量）**：连接归池复用前强制重置会话态，消除上一请求遗留的会话副作用：

- PostgreSQL 类：`DISCARD ALL`、重置 `search_path`、回滚未决事务、清除临时表与会话变量。
- 对存在会话副作用且无法可靠净化的请求形态，禁用连接复用（即建即用即弃）。
- 会话净化是不变量而非优化：净化失败的连接销毁而不归池（fail-closed）。

**Principal 作为隔离维度（评估）**：对会话态敏感的适配器，评估把 Principal 纳入池键隔离维度（池键扩展为 `(ResourceCode, CredentialTier, PrincipalId)`），避免跨 Principal 复用同一会话态连接带来的串味；权衡是连接数与复用率，按适配器声明取舍。

### 6.4 响应脱敏（Sanitizer）

- **系统级 ScrubSet**（恒生效）：覆盖保险箱 `targets` 中全部真实地址字符串/IP、`secrets` 中全部凭据值（及其常见编码形态）、私网 IP 段模式、连接串模式。**ScrubSet 由机密面**（postern-secrets）在保险箱解锁后构造、随保险箱写入更新、并以**不透明 match-and-erase 句柄**形式签发给内核（机密面是属主，见 8.8）；daemon/内核只持该句柄，只能"匹配并擦除"，不可枚举、不可序列化，句柄内容永不出现在输出路径。
- **声明级 MaskRule**：来自 `grant_constraints.kind='mask_fields'`，按列/字段名擦除或掩码。
- 作用面：正常响应、错误信息、拒绝响应的 `denied.objects`、审计事件正文——一切离开内核的字节。

**脱敏诚实度（系统级脱敏的能力边界）**：系统级 ScrubSet 本质是**黑名单**——匹配已知的真实地址/凭据串并擦除。它不能也不承诺识别全部编码变体（base64、hex、整数化 IP、分块拼接等）。因此真正的保证不来自输出端擦除，而来自更上游的两道机制：

- **匿名化**：请求侧 Agent 无从指定、也无从获知真实地址（只见资源代号），真实地址因此不进入 Agent 可控的输入/输出回路。
- **凭据零接触**：凭据从不经手 Agent，也从不进入响应路径。

系统级脱敏对真实地址/凭据是"尽力而为的黑名单"防漏，是匿名化与凭据零接触之上的一层兜底，而非绝对识别保证。高敏感场景应优先采用**结构化字段白名单输出**（只放行声明允许的字段，其余一律不出）而非全文黑名单擦除——白名单对未知编码变体天然封闭。

**流式脱敏模型**：`Sanitizer::scrub` 作用于完整 payload 的形态只适用于小响应；MCP 流式输出与容器日志是流式大输出，须按流式模型脱敏：

- **跨 chunk 滑动重叠窗口**：保留上一 chunk 尾部 N 字节参与下一 chunk 的匹配，消除"敏感串恰好跨 chunk 边界被切开从而逃逸匹配"的边界分块逃逸；N 取系统级 ScrubSet 中最长匹配模式的长度上界。
- **有界缓冲与背压**：observe 类大流定义有界缓冲与背压，下游消费慢时对上游施压而非无界缓冲（与连接管理层的有界排队一致，fail-closed）。

### 6.5 控制面 API（HTTP/JSON over control.sock）

| 方法/路径 | 语义 |
|---|---|
| `POST /v1/principals` · `GET /v1/principals` | 主体管理 |
| `POST /v1/credentials` · `POST /v1/credentials/{id}/revoke` · `.../rotate` | 凭证签发/吊销/重叠期轮换 |
| `PUT /v1/credentials/{id}/trust-domain` | 可信域设定/修改(热生效) |
| `POST/GET /v1/roles` · `/v1/bindings` | 角色与绑定 |
| `POST/GET /v1/resources` · `POST /v1/resources/{code}/discover` | 资源与能力面发现（控制面 `discover` 触达 `Adapter::discover`，是接入侧探测，区别于数据面 `postern_surface`，见 CONS-20）（发现≠授权） |
| `POST/GET/DELETE /v1/resources/{code}/constraints` | 细则（grant_constraints）增删查 |
| `POST/GET/DELETE /v1/resources/{code}/conditions` | 条件谓词（grant_conditions）增删查 |
| `POST/GET/DELETE /v1/resources/{code}/deny-notes` | 拒绝注记（deny_notes，公理六）增删查 |
| `GET/PUT /v1/settings/{key}` | 设置项读写（已知 key 注册表见 5.2；写入受版本/校验约束） |
| `POST /v1/grants/temp` · `DELETE /v1/grants/temp/{id}` | 临时授权(elevate/revoke) |
| `PUT /v1/mode` | 模式切换(全局/单资源,可带 ttl) |
| `GET /v1/grants/{principal}` | 授权视图(your_grants 同源) |
| `GET /v1/audit?since&principal&kind&decision&page_no&page_size` | 审计查询（分页，倒序） |
| `GET /v1/denials/summary?window=7d` | 拒绝聚合分析（告警阈值来源，见 6.9） |
| `GET /v1/approvals` · `POST /v1/approvals/{id}/approve\|deny` | 审批挂起队列查询与裁决（见 6.10） |
| `POST /v1/export` · `POST /v1/import` | 声明式导出/导入(见 6.6) |
| `POST /v1/verify` | 红队自检(见 6.7) |
| `GET /v1/health` · `POST /v1/shutdown` | 生命周期 |

所有写端点 = 一次事务 + 快照重建 + 审计事件；错误响应遵循统一 `{error: {code, message}}`。全部集合端点强制分页：`page_no/page_size` 缺省取默认值 20，`page_size` 钳制到上限 200，响应统一 `Page<T>` 信封，分页在 SQL/扫描层执行，禁止前端分页（契约 `DB_PAGINATION_MANDATORY`）。雪花 id 在 JSON 中一律字符串序列化。

**乐观锁的端到端贯通**：读端点（`GET` 单条与集合）在响应中**统一返回 `version`**；更新/删除端点**必须在请求体或 header 携带期望 `version`**——不匹配返回 `409 Conflict` 并写 `policy_change` 审计（记录冲突）。期望 version 的唯一来源是调用方先前读取值，base 仓储不自读自比（见 5.1-②）。系统协调写（sweeper 幂等回收）不走乐观锁。

### 6.6 声明式导入/导出（TOML）

```toml

[[resources.constraints]]

[[deny_notes]]
resource = "db-main"; capability = "mutate"
note = "写库请先在 staging 验证;确需生产写入找 @owner"
```

瞬时状态（temp_grants 及其 TTL）不导出；导入校验失败整体拒绝（无部分 apply）。

### 6.7 红队自检（postern verify）

控制面触发，daemon 以一个临时低权 Principal 自我发起一组**应被拒绝**的数据面请求，逐条确认 deny 且 deny 出现在审计中：

1. 伪装写操作（写 CTE 包裹）经只读授权 → 应在归类或引擎层被拒
2. CTE 包裹 DELETE（`WITH x AS (DELETE ...) SELECT ...`）经 mutate 授权（非 destroy）→ 应被拒：该语句按最高危写节点归为 Destroy，mutate 授权不足以放行
3. 模板外自由命令 → deny
4. Scope 外资源代号访问 → deny 且拒绝响应不泄露该资源是否存在
5. 已吊销/已过期凭证请求 → 认证层 deny
6. 可信域外来源使用受限凭证 → deny
7. 提权类操作（请求 admin 性质能力）→ 模型层不存在该授权
8. 响应脱敏探测（诱导回显连接串/IP 的请求）→ 返回内容经擦除
9. escalate 标注操作在审批关闭下 → 立即 deny

任一项未被拒绝 → verify 失败并指出防线缺口。

### 6.8 MCP 外壳工具面

工具集是**固定的动词工具**（不随授权动态增删，描述只含事实）：

| 工具 | 对应 |
|---|---|
| `postern_grants` | 返回该 Principal 的 your_grants 事实 |
| `postern_query` / `postern_observe` / `postern_mutate` / `postern_execute` / `postern_manage` / `postern_destroy` | 各动词入口，参数 `(resource, request...)` |
| `postern_surface` | 该 Principal Scope 内资源的**已授权能力面**——授权快照的投影，禁止触达 `Adapter::discover` |

每个工具调用归一化为同一 `NormalizedRequest` 走完整管线（公理七）；拒绝响应原样作为工具结果返回（Agent 据此分流）。

**`discover` 一词的两术语边界（CONS-20）**：避免数据面与控制面同名混淆——

- **数据面 `postern_surface`**：Agent 可见的工具，返回的是**授权快照的投影**（该 Principal Scope 内已授权的能力面），**不触达任何底层资源**、不调用 `Adapter::discover`。
- **控制面 `Adapter::discover` / `POST /v1/resources/{code}/discover`**：接入侧的能力面探测，真实连上资源探测其能力，供运维接入用，绝不暴露给数据面 Agent。
- 命名规范（7.1）固化这两个术语的边界，禁止互相借用。

### 6.9 审计导出器与告警

**审计导出器（`AuditExporter`）**：在本地权威 JSONL 写入之外，提供旁路导出到外部可观测系统的设计；导出是**旁路**，其失败绝不影响本地权威写入（本地 JSONL 始终是单一权威，公理三）。

```rust
/// 审计导出器:旁路镜像本地审计事件到外部系统;失败不回压、不阻断本地写入
pub trait AuditExporter: Send + Sync {
    fn export(&self, event: &AuditEvent);   // 无 Result 回到主路径:失败仅自记+丢弃/缓冲,不传播
}
```

- **与本地权威写入的关系**：`AuditSink::record`（本地 JSONL）先成功，再异步喂给 `AuditExporter`；导出器背压/失败由其内部缓冲与丢弃策略消化，不回到数据面决策路径。
- **OTel 字段映射**（导出为 OpenTelemetry log/span 时）：`ts`→timestamp；`kind`→event name；`decision`/`stage`/`reason`→attributes；`principal_id`/`resource_id`/`credential_id`→attributes（雪花 id 字符串）；`policy_rev`/`intent_digest`→attributes；凭据值与真实地址在 `AuditSink` 写入前已脱敏，导出器只搬运已脱敏事实。

**告警（最小设计）**：

- 数据来源是 `denials` 聚合（`GET /v1/denials/summary`）：deny 事件按 Principal/资源/阶段聚合，超过阈值产生**可被控制面查询的告警事件**。
- 通知为**可选插件**：默认仅控制面可查询告警事件，外发通知（如 webhook）作为可选插件挂接，不进默认数据平面路径。

### 6.10 审批挂起机制

审批开启（`settings: approval.enabled=true`）时，`escalate` 动作不立即折叠为 deny，而是进入**挂起队列**等待人工裁决；审批关闭时 `escalate ≡ deny`（恒 fallback，见 4.1 `Decision::Escalate`）。

- **挂起请求的形态**：挂起请求以内存队列承载，并持久化必要的恢复元数据；**daemon 重启时一切挂起请求恒 deny（fail-closed）**——不跨重启"复活"一个待批的危险操作。
- **超时与 sweeper**：每个挂起请求带超时定时器，超时即 deny（`approval.on_timeout` 固定 `deny`，不可配 allow）；sweeper 兜底回收超时仍未裁决的挂起项并写审计（与 TTL 回收同源，时序无关性见 6.2）。
- **控制面端点**：`GET /v1/approvals`（查询挂起队列）、`POST /v1/approvals/{id}/approve|deny`（裁决）。裁决写审计事件。
- **CLI**：`postern approvals list` / `postern approvals approve|deny <id>`。
- **推送通道**：默认仅控制面可查询；通知为**可选插件**（与 6.9 告警通知同款插件接口），不内置外发。
- **settings 修改路径**：审批开关（`approval.enabled`/`approval.on_timeout`）经 `GET/PUT /v1/settings/{key}` 管理，并可经 `import/apply` 的 `[approval]` 段声明（见 6.6）；`on_timeout=allow` 在导入校验与设置写入两处都拒绝。

### 6.11 SDK 设计边界

SDK 是**纯 HTTP 客户端封装**，供集成方以宿主语言调用数据面/控制面 API：

- **零安全逻辑**：SDK 不做任何认证、归类、脱敏、求值——一切安全决策在 daemon 侧。SDK 只负责构造请求、传输、解析响应。
- **不在 workspace 内**：SDK 作为独立仓库 / 语言包发布，不属于 `postern/` cargo workspace（避免安全核心与客户端封装耦合）。
- **原样透传结构化拒绝**：收到结构化拒绝响应（`DenyResponse`）必须**原样透传**给调用方，不得吞掉、改写或重试——拒绝是事实，由上层 Agent 据此分流（公理六）。

### 6.12 远端探针

部分资源（如远端主机的容器日志）的取数需要在远端侧有一个**只读探针**作为通路终点：

```rust
/// 远端探针:部署在资源侧的只读能力终点;只读、能力面声明、版本协商
pub trait Probe: Send + Sync {
    fn capabilities(&self) -> &'static [Capability];   // 只读能力面声明(恒不含写动词)
    fn protocol_version(&self) -> u32;                 // 与网关侧版本协商
    // 仅暴露只读取数;不承载任何写/管理动作
}
```

- **只读契约**：探针只暴露只读取数，能力面声明恒不含写动词；网关侧据声明与本地策略叠加，探针不做授权（授权在网关）。
- **与 Transport 的关系**：探针端口作为 `Transport` 通路的终点——网关经 `Transport::open` 建立到探针的通路，探针在远端代理只读取数。
- **部署与升级**：探针随资源侧部署，版本经 `protocol_version` 与网关协商；不兼容版本拒绝建立通路（fail-closed）。
- **docker_logs 两种取数形态及安全差异**：
  - **直连远端已暴露的只读 API**：网关经 `Transport` 直连远端容器运行时的只读端点。安全前提是远端已自行限定该端点只读且可达性受控；网关不为远端端点的越权暴露兜底。
  - **经探针取数**：探针作为只读终点，把"只读"约束下沉到远端进程边界——即便远端运行时本身能力更宽，探针只转发只读取数。安全性更高（最小暴露面），代价是需在远端部署并升级探针。

### 6.13 资源系统认证与会话凭据生命周期

本节回答"网关如何以某账号登录业务系统、维持其会话、并按该账号代 Agent 操作"，处理 token 维护与二次验证（2FA）。它是 `CredentialProvider` 的**会话来源**实现（技术设计 10.6），与数据库密码（静态）、redis ACL 账号（静态）、云 STS（动态）并列；区别是它**有状态**：登录一次→复用→刷新，绝不每次请求重登。

典型场景：业务系统暴露 80 端口、经 Transport 转发；Agent 经网关使用其 HTTP 接口/页面。网关持有该系统的账号（资源凭据，Agent 永不接触，公理四），按 tier 可持有多个账号（只读/普通/管理），引擎兜底落在业务系统自身权限体系上。

#### 两阶段：接入期引导（可交互，含 2FA） vs 运行期物化（无人值守）

与"机密保险箱解锁需人在环、数据面求值无人值守"同一原则——**2FA 只发生在接入期（控制面、人在场），绝不进入每请求的数据面路径**（公理二：人不在环是常态）。

**① 接入期引导（控制面，一次性，可交互）**

```
postern resource add <code> --auth session ...     // 由运维在控制面发起
   │
   ▼ 执行该资源声明的【认证流程描述符】(auth_flow):
     表单登录(POST 用户名/密码 → Set-Cookie + 提取 CSRF) /
     API token 直配 / OAuth(授权码或客户端凭据) / SSO 跳转
   │
   ├─ 若需 2FA：此刻人在场,完成 OTP/推送确认
   │
   ▼ 捕获【持久凭据】写入机密面 vault(加密;Agent 永不可见):
     · 可刷新的长效凭证(refresh token / 长效会话 cookie)+ 刷新方法 + 过期策略
     · 若系统只发短效会话且无刷新、且每次登录都需 2FA → 如实记录该约束
   │
   ▼ 接入期校验:用刚得到的会话试探一次只读请求,确认账号 tier 与声明的动词集自洽(同 SEC tier⊆账号权限校验)
```

**② 运行期物化（数据面，无人值守，每次建连时）**

```
连接管理层建连 → CredentialProvider.credential_for(resource, tier)
   │
   ▼ 查【live-session 缓存】(进程内,Zeroizing,键=(resource, tier)):
     ├─ 命中且未临近过期        → 直接复用缓存中的访问令牌/会话(★不重登)
     ├─ 缺失/临近过期(expiry−skew) → 触发刷新(见下),刷新成功后回填缓存
     └─ 刷新失败/硬过期(refresh token 失效、需重新 2FA)
          → fail-closed:该请求 deny,reason="资源会话已过期,需经控制面重新接入授权
            (可能涉及二次验证)",附 operator_note;绝不在数据面静默重试登录
   │
   ▼ 把活跃会话以【请求注入描述】交给 http 适配器(Cookie / Authorization: Bearer /
     自定义头 / 每请求回填 CSRF),由适配器贴到转发请求上
```

#### 刷新策略

- **单飞（single-flight）**：同一 `(resource, tier)` 任一时刻至多一个在途刷新；并发请求等待该刷新或复用现值，杜绝刷新风暴。
- **重叠窗口**：在 `expiry − skew` 主动刷新，旧令牌在新令牌就绪前仍有效，在用请求不中断（与凭证重叠期轮换同构）。
- **刷新令牌轮换**：若刷新返回新的 refresh token，事务性写回 vault（持久凭据更新，热生效）。
- **载体分层**：长效 refresh token 存 vault（加密、持久）；短效访问令牌只在 live-session 缓存（内存、Zeroizing），不落盘。

#### 2FA 的安置与诚实边界

- 2FA 一律是**接入期、人在场、控制面**的一次性成本，换取一个数据面可无人值守续期的长效凭证。这正是"安全不依赖实时审批"在认证侧的落点。
- **诚实限制**：若某系统每次登录都强制 2FA 且不提供任何可刷新的长效会话，则网关只能在单次会话有效期内运作；会话硬过期后 fail-closed 拒绝并提示重新接入。设计不假装能绕过这种系统——它把约束如实暴露给运维，而非静默降级。

#### 与各层的关系

- **机密面（8.8）**：持久凭据（refresh token / 长效会话）是 vault 中的资源凭据；live-session 缓存与刷新逻辑随会话来源实现，属机密面/连接建立流程，明文永不出该边界、不进审计内容。
- **传输层**：刷新/登录本身是对业务系统认证端点的一次 HTTP 调用，经 `Transport` 通路完成（探测式短连接或复用通路）。
- **审计（观测面）**：接入引导、每次刷新、刷新失败均记 `credential_event`，只记"哪个会话/账号/tier"，绝不记令牌值。
- **tier**：`(resource, tier)` 决定用哪个业务系统账号登录；不同 tier 的会话互不共享连接（连接粒度隔离，归池前会话净化照常）。

#### schema 落点

会话来源所需的非敏感配置（认证端点、流程类型、注入方式、过期/刷新策略、是否需 2FA）作为 `resource_credential_tiers.auth_flow`（JSON）声明；敏感项（账号、refresh token）一律 `vault://` 引用，永不入 policy.db、永不出现在导出文件（11.2 不变量）。

## 第七部分 · 工程规范

### 7.1 代码规范

- **格式**：rustfmt 默认配置，无自定义；CI `cargo fmt --check`。
- **Lint（workspace 级，fail-closed 写进门禁）**：`[workspace.lints]` 统一声明，CI 以 `-D warnings` 强制（warn 级在 CI 等同失败，不靠人盯）。完整 deny 清单：
  - `unsafe_code = "forbid"`（全 crate）——类型层安全保证（机密类型不可构造/复制、`Debug=REDACTED`）以内存安全为前提，一处 unsafe 即可绕过；已知唯一可能需要 unsafe 的点（`SO_PEERCRED`）有安全 API（`tokio UnixStream::peer_cred`），故全工作区 forbid。若未来确需 unsafe，须登记于受保护豁免文件（与 `contract/sql-exceptions.json` 同构：crate/file/reason/approved_by）。
  - `clippy`：`unwrap_used` / `expect_used` / `panic` / `indexing_slicing` / `arithmetic_side_effects`（关系到 IdGen 时间戳运算与 TTL 比较）/ `todo` / `unimplemented` / `unreachable` = deny（测试代码以 `cfg_attr` 豁免前三项）。
  - **吞错放行**（`.ok` / `.unwrap_or(true)` / `.unwrap_or_default` 在求值路径 `core::eval`、`daemon::kernel`）由契约 `EVAL_NO_ERROR_SWALLOWING`（+反例）强制，而非仅靠 lint——fail-closed 是被机器守护的不变量，不只是代码规范（见 7.3）。
- **错误模型**：每 crate 一个 thiserror 错误枚举；core 中维护"错误变体 → 拒绝阶段"的穷尽 match（新增错误变体不写映射无法编译）。anyhow 仅出现在两个二进制的 `main`。
- **panic 政策**：数据面外壳挂 CatchPanic 层——任何 panic 一律转为脱敏后的 deny 响应 + 一条 `kind=anomaly` 审计事件，绝不让 panic 成为唯一不留痕的失败路径（公理二）；release profile 显式声明 panic 策略（默认 `unwind` 以便外壳捕获，理由写入 Cargo 配置注释）。`indexing_slicing`/`arithmetic_side_effects` lint 收窄间接 panic 源。
- **机密类型纪律**：资源凭据/真实地址（`ResolvedTarget`/`ResourceCredential`）与 Agent 出示的网关凭证（`PresentedCredential`）只存在于 `Zeroizing` 容器；这些类型不 derive `Clone`/`Serialize`（`ResolvedTarget`/`ResourceCredential` 由契约 `SEC_SECRET_TYPE_DISCIPLINE` 强制）、`Debug` 恒输出 `REDACTED`、不实现 `Display`；tracing 字段在类型层即无法直接记录它们。
- **命名**：crate `postern-*`；模块按设计面命名（与第四部分目录一致）；领域词汇与技术设计文档严格对齐（Principal/Resource/Role/Scope/Capability/Tier——其中 Tier 在技术设计 10.5 定义，不引入同义词）。
- **注释与语言**：代码、标识符、代码内注释、日志与错误文案为英文；`docs/` 设计文档为中文。注释只写代码无法表达的约束与不变量。

### 7.2 安全编码红线

1. 真实地址与资源凭据的构造路径只在 `postern-secrets`（契约 `SEC_CONSTRUCTION_SITES` 强制 `ResolvedTarget`/`ResourceCredential` 只在该 crate 构造）；任何把它们传出 `Transport::open` 调用边界的代码不予合并。传输/连接层错误在跨 `postern-secrets`/`postern-transports` 调用边界上抛前，必须先在 crate 内部脱敏为不含真实地址的错误码（与机密类型 `Debug=REDACTED` 纪律一致），绝不让 `connection refused to 10.0.3.17` 一类原始错误串外泄。
2. 数据面 handler 的依赖注入集合中不得出现 PolicyRepo / vault 句柄（构造函数签名审查）；跨 crate 的禁止依赖边由契约 `ARCH_FORBIDDEN_EDGES` 强制（adapters↛secrets/transports/store、transports↛store、cli↛store/secrets、core↛任何 IO）。
3. 一切返回 Agent 的字节必须经过 `Sanitizer`——**不存在任何绕过 Sanitizer 的输出路径**：正常响应、错误信息、拒绝响应、外壳层 4xx、框架级错误、panic 响应体，一律经同一 `Sanitizer` 并套统一 `{error:{code,message}}` 信封；信封 `message` 只用常量化安全文案，绝不回显原始 intent 或底层错误串。
4. 不采信请求自报的来源字段；`ConnOrigin` 只能由 listener 层构造（契约 `SEC_CONSTRUCTION_SITES` 强制 `ConnOrigin` 只在 daemon shells 构造）。
5. 审计相对执行的时序是安全不变量（见 6.1）：只读动词审计写失败按 deny 返回（不可记录 = 不放行）；有副作用动词采用两阶段审计——执行前 intent 写失败则执行前 deny，已执行的请求绝不返回 deny，outcome 审计失败返回"已执行但审计降级"错误。

### 7.3 测试规范

- **单元测试**：与被测代码同文件 `#[cfg(test)]`；求值器测试以纯内存 `PolicySnapshot` 驱动（core 零 IO 的直接收益）。
- **集成测试**：各 crate `tests/`；adapters 的归类测试维护一份"伪装攻击语料"（写 CTE、多语句、注释混淆、DO 块等），全部断言 deny 或正确归类。
- **契约测试（Stele）** — 区分两类不变量，二者机制不同、不要混为一谈：

  **(a) 结构性安全核心规则（带 teeth、构成 24 条契约的 5 条安全规则）**：经受限通道提案至 `contract/proposals/agent-additions.stele`，由 build.rs 扫描器（扫真实代码 + 内嵌反例）接线，与 7 条数据库层规则同机制。这 5 条是：
  - `SEC_ADMIN_NOT_GRANTABLE` — Capability 枚举无 Admin 变体，roles 表带禁止 admin 名的 CHECK
  - `SEC_SECRET_TYPE_DISCIPLINE` — `ResolvedTarget`/`ResourceCredential` 不得 derive 或手写 impl Clone/Serialize
  - `SEC_CONSTRUCTION_SITES` — `ConnOrigin` 只在 daemon shells 构造、`ResolvedTarget`/`ResourceCredential` 只在 postern-secrets 构造
  - `ARCH_FORBIDDEN_EDGES` — 禁止依赖边（adapters↛secrets/transports/store、transports↛store、cli↛store/secrets、core↛任何 IO）
  - `EVAL_NO_ERROR_SWALLOWING` — 求值路径禁 `.ok()`/`.unwrap_or(true)`/`.unwrap_or_else(|_| true)`/`.unwrap_or_default()` 吞错放行

  **(b) 运行时求值面行为不变量（独立于上述 24 条，待求值器实现后以 `(assert)` 值契约由纯内存 Evaluator 驱动）**：`ESCALATE_FOLDS_TO_DENY`（审批关闭/超时下 escalate 恒 deny）、`UNCLASSIFIABLE_INTENT_DENIED`（classify Err 恒 deny）、`EXPIRED_OR_REVOKED_CREDENTIAL_DENIED`（过期/吊销凭证在步骤[1]被拒，求值时刻墙钟二次校验，见 6.2）、`TEMP_GRANT_EXPIRY_EFFECTIVE`（过期临时授权对求值即刻不可见）、`AUDIT_FAILURE_DENIES`（只读动词审计写失败不放行；有副作用动词遵循两阶段审计时序，见 6.1）、`NO_TIER_MATCH_DENIED`（放行动词无可承载凭据等级时 deny）、`DENY_RESPONSE_SCOPE_BOUNDED`（拒绝响应只含该 Principal 自身授权世界的事实）。这些是行为契约，需运行时数据驱动，不属于当前 24 条静态结构契约。
- **契约总数**：**24 条（12 规则 × 本体 + 反例）**——7 条数据库层规则（见 5.1）+ 5 条结构性安全核心规则（上列 (a)）。各规则经受限通道 `stele propose invariant` 提案、经人审写入并由 `stele lock`/`stele check` 守护；规则本体扫描真实代码、反例自检扫描内嵌违规夹具，二者同源（teeth 机制见 7.6）。上列 (b) 的运行时行为不变量不计入这 24 条。
- **红队自检**：`postern verify`（6.7）是运行时功能也是发布前必跑项。

### 7.4 版本管理与门禁规范

- Conventional Commits（`feat:`/`fix:`/`docs:`/`refactor:`/`test:`/`chore:`，英文祈使句），类型与语义化版本联动（fix→patch、feat→minor、breaking→major）。
- **MSRV 与工具链锁定**：声明最低支持 Rust 版本（MSRV），仓库根放置 `rust-toolchain.toml` 锁定开发/CI 工具链——安全产品的构建须在已知编译器版本上可复现（契约接线曾暴露过"换机后工具行为漂移"的真实风险）。
- **版本号**：`workspace.package` 统一 version，`posternd` 与 `postern` 二进制版本随之锁同。
- 主分支 `main`；`docs/` 设计文档纳入版本管理（设计变更走 commit 评审）。
- **CI 门禁确切命令序列**（功能分支合入前须全绿，任一步失败即拦截）：
  1. **meta-gate**：核对 `contract/*.stele` 的 `(assert ...)` 计数 ≤ `tests/contract/test_contract.rs` 中 `assert!(` 计数（防 Stele 后端把断言降级为空转；已由 build.rs 在构建期 fail-closed，CI 再核一次）。
  2. `cargo fmt --check`
  3. `cargo clippy --workspace --all-targets --all-features -- -D warnings`
  4. `cargo hack check --each-feature`（至少 `--no-default-features` 与 `--all-features` 两档，覆盖插件 feature 矩阵）
  5. `cargo test --workspace --all-features`（含 `cargo test --test test_contract`；契约测试在干净构建下执行，避免增量假绿）
  6. **架构禁止边校验**：契约 `ARCH_FORBIDDEN_EDGES` 已覆盖；如需冗余可加 `cargo tree` 断言脚本。
  7. `cargo deny check`（advisories / bans / sources / licenses）——这是常驻持有生产凭据的 daemon，供应链是威胁模型的直接组成；`deny.toml` 的 `[bans]` 列 uuid/ulid/nanoid 等 id 库**连同传递依赖**（弥补 `DB_UNIFIED_ID_GENERATOR` 文本扫描只覆盖直接依赖的盲区）；`deny.toml` 纳入受保护路径，使"新增依赖须说明理由"复用 `sql-exceptions.json` 同构的人工审批机制（申请记录即 `deny.toml` 的 allow 条目）。
  8. `stele check`（权威门，fail-closed）。

### 7.5 运行日志规范

运行日志（tracing）与审计事件流（5.3）是两套东西：审计是问责载体（append-only、进 5.5 权限布局），运行日志是运维诊断载体。二者都不得泄漏机密。

- **去向与权限**：日志文件落 `$XDG_STATE_HOME/postern/log/`，`0600`，按日/大小轮转；纳入 5.5 磁盘布局表。默认 stdout/journald 形态同样受"内容红线"约束。
- **内容红线**：`Intent` 原文（SQL 文本可含业务敏感数据）、`PresentedCredential`、vault payload、任何 `secret_ref` 解引用结果一律禁止入日志；上述类型已纳入机密类型纪律（`Zeroizing`/`Debug=REDACTED`/无 `Display`），在类型层即无法被 tracing 字段直接记录。
- **字段白名单与命名**：`request_id`（与审计事件 id 同源，便于日志与审计对账）、`principal`、`resource`（恒为代号）、`capability`、`stage`、`decision`、`duration_ms`。
- **级别约定**：逐请求细节只进审计；tracing 限生命周期/连接/异常事件；`debug` 级在 release 构建默认关闭。

### 7.6 数据库契约接线机制

本节规定数据库层契约的接线机制（而非任何一次接线的状态）。接线形态与 Stele Rust 后端的机制对齐——生成的运行时从 **fixture 文件**读取上下文（`tests/contract/mod.rs` 不参与生成测试的执行，是占位）：

```
契约源:   contract/main.stele + contract/proposals/agent-additions.stele(12 规则,纳入 git 跟踪)
生成:     stele generate → tests/contract/{_stele_runtime.rs, test_contract.rs}(受保护)
接线:     build.rs(受保护路径)——每次构建扫描真实工作区,把 12×2 个计数写入根目录
          .stele_fixture.json(SteleValue 外部标签 JSON:{"Map":{...}} / {"Int":n});
          生成测试经 stele_context 读取该 fixture 并 assert
测试目标: 根 Cargo.toml 显式声明 [[test]] name=test_contract(tests/ 子目录不被 cargo 自动发现)
门禁:     stele generate → cargo test --test test_contract → stele lock → stele check
```

12 条规则 = 7 条数据库层规则（5.1）+ 5 条安全核心规则（admin 不可授予、机密类型禁 derive、构造点限定、禁止依赖边、求值禁吞错；见 7.3），各带一条反例自检，共 24 条契约。

**受保护边界（契约机制的自我完整性）**：`build.rs`、根 `Cargo.toml`、`contract/sql-exceptions.json` 与 `contract/**/*.stele`、`tests/contract/**`、`stele.config.json` 一并纳入 Stele 受保护集并经 `stele lock` 哈希锁定。理由：扫描器对真实工作区的计数与对反例夹具的计数共用同一函数,只有当 build.rs 不可被静默改写时,"规则失效⇒反例也失效"才成立——若 build.rs 可写,Agent 可把真实计数那一路改成常量 0 而反例那一路照常返回 1,使规则恒过、反例恒过、整套契约被无声中和。把扫描器接线纳入保护,堵死这条元层面的绕过(公理三)。

**fixture 格式**：`.stele_fixture.json` 是 SteleValue 的外部标签 JSON——`{"Map":{...}}` 承载键值上下文、`{"Int":n}` 承载计数。每条规则在 fixture 中体现为两个计数键：真实工作区违规计数 `violation_count` 与反例夹具违规计数 `selftest_violation_count`。`.stele_fixture.json` 是构建产物（gitignore），其权威来源是受保护的 build.rs。

**扫描器双输入结构**：同一扫描器逻辑跑两份输入——真实工作区（产出 `violation_count`，必须为 0）与内嵌反例夹具（产出 `selftest_violation_count`，必须 ≥1）。逻辑只有一份，"规则失效但反例仍过"在结构上不可能成立。

build.rs 扫描规约（std-only——构建脚本不可用 dev-deps）：

- 文本级保守扫描：允许误报、禁止漏报（公理二）；注释与字符串里的 SQL 不豁免。误报的出路是改写代码或走豁免申请，不是放松扫描器。
- 路径不存在 → 计数 0：相应代码尚不存在时规则空转通过（诚实），teeth 契约始终真实断言。
- 豁免唯一来源 `contract/sql-exceptions.json`（受保护，不存在视同 `[]`）；其创建与每次修改都必经人工契约流程——"申请"的系统强制形态。
- `cargo:rerun-if-changed` 注册 crates/ 目录与全部已扫描文件，violation 变更触发重扫描。

**豁免文件流程**：豁免条目格式 `{crate, file, reason, approved_by}`，唯一来源 `contract/sql-exceptions.json`（Stele 受保护路径，初始为空数组）。修改该文件必然经人工审批的契约变更流程——明确目的 + 申请后才允许，由系统强制而非约定（见 5.1-④）。

**验收不变量**：生成的 `test_contract.rs` 必须含 `assert!`——若生成结果中不存在 `assert!`（例如生成器把不变量降级为丢弃布尔结果的 `let _ = <比较>;`），视为生成器失效，**拒绝采用该生成结果**。这是接线的硬验收项：assert 契约一旦退化为空转，所有 teeth 形同虚设，故以"生成产物含 assert"作为可机械核验的接线正确性门。

## 第八部分 · 领域范围与内涵

> 本章是全部领域边界争议的**最终裁决依据**。每个领域给出：一句话内涵、范围内（封闭列举）、范围外（显式排除并指向属主域）、对外接口、必须守住的不变量。与本章冲突的散落行文以本章为准；本章与七公理冲突时以公理为准。技术设计文档第四部分架构图的概略职责，其边界定义同样以本章为准。

### 8.0 边界裁决总则

1. **单一属主**：每个概念、每份数据、每项职责在系统中恰有一个属主域（owner）。其他域只能经属主域的对外接口消费，不得自建副本语义。
2. **面 ≠ crate**：领域是语义边界，crate 是载体。一个 crate 可承载多个领域（`postern-daemon` 是组装点，承载数据面内核、控制面、连接管理、脱敏执行与策略生命周期自动机），但一个领域的权威定义不可分属两处。
3. **跨域数据以接口类型表达边界**：类型的构造权与可见性即边界（`ResolvedTarget` 只能在机密面构造，即为公理四的编译期表达；契约 `SEC_CONSTRUCTION_SITES` 强制之）。
4. **决策者与执行者分离要显式标注**：凡一项动作由 A 域决策、B 域执行（如连接回收：连接管理决策、传输执行），本章逐项写明，不留默认。

**争议概念归属速查表**

| 概念 | 属主域 | 其他域的角色 |
|---|---|---|
| 脱敏执行（调用点） | 数据面请求内核 | 观测面声明事实要求；外壳只接收脱敏产物 |
| ScrubSet（系统级擦除集） | 机密面（构造、更新、持有） | 内核持不透明句柄，仅可匹配擦除 |
| 通路内保活（心跳/续约） | 传输 | — |
| 重建决策 / 退避 / 回收 | 连接管理 | 传输按指令执行关闭 |
| 网关凭证语义（过期/吊销/可信域规则） | 身份与凭证 | 存储持久化元数据；控制面提供变更入口 |
| 凭证签发/轮换/吊销操作 | 控制面 | 身份域定义其合法性规则 |
| CredentialTier 声明 | 策略状态（存储承载，资源声明的一部分） | — |

| Intent 负载格式 | 适配器（唯一解释者） | 外壳只装箱搬运 |

### 8.1 领域核心模型（`postern-core::domain / request / decision`）

**内涵**：系统全部安全语义共享的词汇表与类型系统——Principal、Resource（代号）、Role、Scope、Capability、授权格、细则、条件、CredentialTier、NormalizedRequest、Decision 等纯数据定义及其代数关系。

**范围内**：上述类型与其关系的定义；不变量的类型化表达（Capability 枚举无 Admin 变体）；授权空间展开的纯语义；错误枚举与"错误→拒绝阶段"穷尽映射；统一 IdGen 与 PageQuery；全部插件 trait 的**定义**（实现归各域）。

**范围外**：任何 IO 与副作用（→各实现域）；持久化形态（→存储层）；`ResolvedTarget`/`ResourceCredential` 的构造（→机密面，本域仅不透明声明）；认证机制（→身份与凭证）；协议解释（→适配器）；决策的执行（→数据面内核）。

**对外接口**：domain/request/decision 公开类型；插件 trait（Authenticator、Adapter、Transport、CredentialProvider、ConditionPredicate、AuditSink、Sanitizer、PolicyView）；IdGen、PageQuery。

**必守不变量**：零 IO、不依赖工作区任何 crate（契约 `ARCH_FORBIDDEN_EDGES`）；admin 在类型上不可表达（契约 `SEC_ADMIN_NOT_GRANTABLE`）；机密类型在本域不可构造；领域词汇与技术设计文档严格对齐。

### 8.2 数据面请求内核（`postern-daemon::kernel`）

**内涵**：把任意外壳进入的请求按统一管线 [0]→[10] 串到底的编排者——安全语义的执行主轴，公理七的承重体。

**范围内**：管线编排与逐步短路；将 Decision 翻译为执行或拒绝；deny 与 allow 路径同等走出口处理；**出口统一脱敏的调用职责**（一切离开内核的字节——正常响应、错误、拒绝响应、审计事件正文——必经 Sanitizer）；审计事件的提交与"审计相对执行的时序不变量"（只读单次审计；有副作用动词两阶段审计，见 6.1）。

**范围外**：决策逻辑（→策略引擎）；协议解释与执行（→适配器）；通路获取与治理（→连接管理）；ScrubSet 的内容（→机密面，内核持句柄）；事件落地（→存储层载体）；外壳形态（→外壳层）；策略状态变更（→控制面，内核对策略只读）。

**对外接口**：对外壳暴露唯一入口"提交 NormalizedRequest → 返回脱敏后的结果或结构化拒绝"；依赖注入仅含 PolicyView（只读快照）、Adapter/Authenticator/Predicate 注册表、连接管理句柄、Sanitizer、AuditSink。

**必守不变量**：依赖集合中不存在 PolicyRepo 与 vault 句柄（构造函数签名保证 + 契约 `ARCH_FORBIDDEN_EDGES`）；任何字节离开内核前必经 Sanitizer（红线 7.2-3）；审计不可记=不放行；步骤[0]之后请求与外壳无关；内核绝不对策略状态产生写入。

### 8.3 策略引擎（`postern-core::eval`）

**内涵**：对单个已归一化请求，基于单一权威策略快照，产出确定性三值决策（allow/deny/escalate）与完整求值轨迹的**纯函数域**。

**范围内**：求值步骤 [1][3][5][6] 的纯逻辑编排（调用认证器与条件谓词注册表；消费 kernel 先行跑出的适配器归类与细则校验结果，见 CONS-8 的接口裁决）；RBAC 展开查表；tier 选择（allow 时依快照中的 tier 声明完成动词→凭据等级映射，无匹配→deny）；escalate 折叠语义；DenyResponse 的**事实组装**（reason/your_grants/request_hint/operator_note，全部取自快照）；EvalTrace。

**范围外**：策略状态的任何变更（→控制面）；快照的构建与重建时机（→存储层 snapshot 模块）；凭证语义规则的定义（→身份与凭证，引擎只消费认证结论）；拒绝响应的最终脱敏（→内核出口，"已脱敏"是出口保证而非本域构造保证）；连接与执行（→连接管理/适配器）；审计落地（→观测面/存储）。

**对外接口**：`Evaluator::evaluate(req, ci, constraint_check_result, policy, now) -> (Decision, EvalTrace)`；消费 `PolicyView::snapshot`。

**必守不变量**：纯函数、零 IO、确定性（now 显式传入，相同输入→相同决策）；一切 Err 解析为 deny（公理二，契约 `EVAL_NO_ERROR_SWALLOWING`）；DenyResponse 只含该 Principal 自身授权世界的事实或人亲笔预写内容（公理六、`DENY_RESPONSE_SCOPE_BOUNDED`）；不持有可变状态。

### 8.4 身份与凭证（Identity，认证器族）

**内涵**：回答"谁在请求"——把出示物与可观测连接来源解析为已认证 Principal，并定义网关凭证（Principal 凭证）的全部语义规则。

**范围内**：Authenticator 族（本地进程身份/APIKey/Token/账密/mTLS/SSO）及其注册；凭证元数据 schema 的定义权（绑定、有效期、可信域、吊销状态）；有效性/过期/吊销/可信域的**判定规则**（求值时刻墙钟二次校验）；Principal 与凭证分离模型；凭证操作的合法性规则（如重叠期轮换语义）。

**本地进程身份的可信度边界**：`ConnOrigin` 仅取 `SO_PEERCRED` 的 uid/gid 作信任域门，放弃 pid/exe 路径这类可伪造特征。同 uid 下区分 Principal 取以下设计之一（部署形态权衡，需选型锁定；推荐后者以守住"每个 Agent 实例独立具名 Principal"的模型承诺）：①承认本地"一个 uid = 一个 Principal"并在审计语义中如实说明；②为每个本地 Agent 实例注入轻量本地握手令牌（控制面拉起 Agent 时下发，非资源凭据）。

**范围外**：凭证元数据的持久化（→存储层 credentials 表）；签发/轮换/吊销/可信域修改的**操作入口**（→控制面）；资源凭据（→机密面——与 Principal 凭证零交集，技术设计 7.3）；授权判定（→策略引擎：认证只确立身份不赋权）；ConnOrigin 的采集（→外壳层 listener，本域只消费）。

**对外接口**：`Authenticator::authenticate(presented, origin, cred_view, now)`，产出 PrincipalId 或 AuthError。

**必守不变量**：fail-closed（无效/过期/吊销/可信域不符/来源无法判定→Err→deny）；绝不采信请求自报来源；认证不赋权；凭证明文哈希（secret_hash）可进快照、凭证明文永不落库永不进快照；数据面只能出示凭证，凭证状态变更无数据面路径。

### 8.5 连接管理（`postern-daemon::connpool`）

**内涵**：治理"许多通路"的域——以 (ResourceCode, CredentialTier) 为键，对通路的获取、复用、健康、上限与回收做全部**通路间生命周期决策**。

**范围内**：池化与复用；建立策略（按需/预热/最小最大数）；**重建决策与指数退避**（传输上报死亡后是否、何时重建）；健康剔除；并发上限与背压（超限排队有界或 deny）；空闲回收与优雅销毁的**发起**，以及 freeze/吊销触发的在用连接强制 abort/cancel；tier 连接隔离（不同 tier 永不共享连接）；建立流程中向机密面请求 (ResolvedTarget, ResourceCredential) 不透明句柄并一次性传入 `Transport::open`；**归池前会话净化**。

**范围外**：单条通路如何建立与通路内保活（→传输）；句柄内容的读取（→机密面，本域仅搬运、不可读、不持有超出单次建立调用）；用通路做什么（→适配器）；动词→tier 的选择（→策略引擎，本域只接收 Allow{tier}）；关闭的物理执行（→传输，经 Channel 关闭接口，本域决策）。

**对外接口**：`acquire(resource, tier) -> Channel`（或通路租约）；对内核与适配器只暴露"一个可用通路"。

**必守不变量**：无法建立→deny（fail-closed）；超限绝不静默放行；不同 tier 不共享连接（账号隔离在连接粒度成立）；本域代码不可读取凭据/地址明文（契约 `ARCH_FORBIDDEN_EDGES`/`SEC_CONSTRUCTION_SITES`）；优雅销毁前排空在途请求，紧急切断则强制中断。

### 8.6 适配器（`postern-adapters`）

**内涵**：某种资源协议的**唯一解释者**——把协议意图翻译为 (Capability 动词, 操作对象)，并在已获通路上执行被允许的操作。

**范围内**：`classify`（协议感知语义归一化，SQL 为语法树级、按语句内最高危写节点定档，见 4.4）；细则类型的语义定义与 `check_constraint`；`execute`；能力声明与 `engine_enforced` 声明；Intent 负载格式的定义（含 MCP 动词工具的参数 schema）；`discover`（能力面探测的执行——仅接受控制面触发）；伪装攻击识别。

**范围外**：授权判定（→策略引擎，适配器产出归类、不产出决策）；通路来源（→连接管理，只见 Channel）；凭据与真实地址（→机密面；契约禁止 adapters→secrets/transports/store 依赖）；脱敏（→内核出口）；发现结果的授权化（→人，经控制面圈选；发现≠授权）；触发探测的权限（→控制面）。

**对外接口**：`Adapter` trait（protocol/capabilities/engine_enforced/classify/check_constraint/execute/discover）。

**必守不变量**：无法可靠归类→Err→deny（白名单归类，宁可误拒）；只见 Channel，不可达地址与凭据；discover 只产出事实、不产生授权；execute 只执行已被求值放行的意图；`engine_enforced=false` 的协议须在文档与能力声明中如实标注"归类+细则是唯一防线"。

### 8.7 传输（`postern-transports`）

**内涵**：把一种远端接入方式抽象为"本地可用通路"的**单通路域**——一条通路从建立到关闭的协议级机制。

**范围内**：通路建立（消费一次性注入的 ResolvedTarget/ResourceCredential）；**通路内保活**（心跳、租约续期、SSM 会话续约等协议级机制）；通路状态的如实上报（健康接口）；按连接管理的指令执行关闭与释放；`persistent` 声明。

**范围外**：重建决策与退避节奏（→连接管理——通路死亡只上报，**绝不自行重建**）；池化（→连接管理）；凭据与地址的保管（→机密面，本域仅在 open 调用生命周期内消费）；协议语义（→适配器）；何时回收（→连接管理决策）。

**对外接口**：`Transport` trait（kind/persistent/open）+ Channel 上的健康与关闭语义。

**必守不变量**：凭据与真实地址不向上传递、不落日志、生命周期不出 open 调用；通路断开如实报告，无静默重建；同一 Channel 对上层呈现一致的"本地 socket"抽象，长/非长连接差异不外溢。

### 8.8 机密面（`postern-secrets`）

**内涵**：两类机密（资源凭据、代号↔真实地址映射）的**唯一权威持有者**，以及一切机密派生视图的**唯一签发者**。

**范围内**：保险箱载体（格式、加解密、Zeroizing 内存持有、原子写入与轮换）；MasterKeySource 族（解锁来源）；代号↔真实地址映射及其解析操作（代号→ResolvedTarget）；CredentialProvider 实现（静态保险箱；动态签发/证书为同接口预留）；`ResolvedTarget`/`ResourceCredential` 类型的唯一构造权；**ScrubSet 的构造、更新与持有**（由 targets/secrets 派生的单向匹配视图，随保险箱写入更新）。

**范围外**：Principal 凭证（→身份与凭证——名称相近、概念无交集）；tier 的声明与选择（→策略状态/策略引擎，本域只按 (res,tier) 解析）；脱敏的执行调用（→内核，本域只签发 ScrubSet 句柄）；保险箱解锁的时机编排（→daemon 启动序列）；机密的录入入口（→控制面，本域提供写接口）。

**对外接口**：`CredentialProvider::credential_for(res, tier)`；`resolve(code) -> ResolvedTarget`；ScrubSet 不透明句柄（仅 match-and-erase 能力）；`vault://` 引用语义；面向控制面的录入/更新接口（回读默认仅掩码或引用形态）。

**必守不变量**：明文只存在于本域内存与 `Transport::open` 调用内；机密类型不可 Clone/Serialize、Debug 恒 REDACTED（契约 `SEC_SECRET_TYPE_DISCIPLINE`）；ScrubSet 只可匹配擦除，不可枚举、不可序列化；数据面无任何取用路径；导出永不含明文。

### 8.9 观测面（Observability）

**内涵**：把"发生过什么"固化为不可变事实并供分析的域——事件的语义、记录纪律、聚合与可选外投。

**范围内**：审计事件 schema 与 kind 体系及版本化；记录纪律的定义（allow 与 deny 都记、deny 必含 stage+reason、两阶段审计时序、fsync 策略、保留期与 DoS 防护语义）；拒绝聚合分析（denials 的统计与模式呈现逻辑）；导出器接口与"副本投影"语义（`AuditExporter`，对齐 OTel，默认关闭，见 6.9）；告警事实的产生（条件求值）。

**范围外**：载体与轮转、扫描的物理执行（→存储层 JsonlAuditSink）；"审计写失败→deny"的处置（→数据面内核）；事件正文脱敏的执行（→内核统一出口）；给人看的查询端点与建议呈现（→控制面；面向人的机械推导建议不受公理六约束——公理六只约束数据面返回 Agent 的内容）；告警的通知投递（→控制面查询暴露，可选经桌面外壳通知）。

**对外接口**：`AuditSink` trait；`AuditExporter` trait；事件类型定义；聚合查询逻辑。

**必守不变量**：append-only，无修改与物理删除语义（保留期到期的整文件删除除外）；事件永不含凭据值与真实地址；导出不可用不影响本地审计权威；deny 事件可完整还原"在哪一步、因何拒绝"。

### 8.10 控制面（`postern-daemon::control` + sweeper）

**内涵**：策略状态的**唯一变更入口域**——对"系统应当如何"的一切写操作与人类管理查询都经此进入，变更发起主体包括人类操作者（经 RPC）与系统自动机（sweeper 回收、import 协调，actor=system）。

**范围内**：控制面 API 全部端点（principals/credentials/roles/bindings/resources/grants/mode/constraints/conditions/deny-notes/settings/approvals/audit 查询/denials/export/import/verify/生命周期）；凭证签发/轮换/吊销/可信域修改的操作入口；`Adapter::discover` 的触发权；声明式导入导出的协调；红队自检的触发；系统自动机的事务写入；"写入 = 一次事务 + 快照重建 + 审计事件"三联动的发起。

**范围外**：写入数据的语义合法性规则（→各属主域：凭证规则归身份域、模式语义归领域模型）；持久化机制（→存储层）；求值（→策略引擎）；机密明文的存放（→机密面，控制面只写 `vault://` 引用）；面向 Agent 的任何服务（→数据面；两平面物理隔离）。

**对外接口**：HTTP/JSON over `control.sock`（`0600`，仅属主，叠加同 uid 也成立的控制面认证）；放大时同一 router 加挂 mTLS TCP。

**必守不变量**：默认不监听网络，Agent 物理不可达；admin 性质操作只存在于此且不可授予任何 Principal；每次写入必产审计；`on_timeout` 恒 deny 且导入校验拒绝其他取值；系统自动机与人类写入走同一事务路径、同等审计；正确性不依赖 sweeper 时序（过期判定在求值时刻）。

### 8.11 存储层（`postern-store`）

**内涵**：策略状态与审计事件两类数据的**载体域**——以"事务"与"append-only"两种纪律把权威状态与事件流可靠落地（机密的载体由机密面自持，不在本域）。

**范围内**：policy.db schema 与迁移；统一基础仓储（8 基础字段自动化、乐观锁、逻辑删除、默认作用域、分页执行——契约 D8 全部落点）；PolicyRepo（事务读写，仅控制面可达）；**PolicySnapshot 构建**（一次事务内全量加载、继承展开、授权空间物化——权威库的原子投影，**内容清单**见下）；JSONL 审计载体（按日轮转、分页扫描）。

**PolicySnapshot 内容清单**（数据面与策略状态之间唯一的桥，其内容边界即 2.3 隔离声明的实际含义）：**含**——展开后的授权空间（绑定×角色继承 ∪ 未过期临时授权）、凭证元数据（含 secret_hash，明文哈希）、tier 声明、grant_constraints/grant_conditions、各辖区 mode、deny_notes、approval 设置；**不含**——任何 vault 内容（资源凭据、真实地址映射）。每项仅供其消费域读取，面向 Agent 的出口只暴露该 Principal 自身授权世界。

**范围外**：存什么的语义（→领域模型/身份域/观测面定义 schema 语义，本域承载）；写入的发起与合法性（→控制面）；快照的消费（→策略引擎）；事件 schema 定义（→观测面）；机密（→机密面）；id 生成（→core IdGen）。

**对外接口**：PolicyRepo；`PolicyView::snapshot -> Arc<PolicySnapshot>`；JsonlAuditSink（impl AuditSink）+ scan 分页查询。

**必守不变量**：写路径唯一（INSERT/UPDATE 只在 base 模块，契约 `DB_WRITE_PATH_CENTRALIZED`）；只有逻辑删除（`DB_LOGICAL_DELETE_ONLY`）；SQL 与 rusqlite 不出本 crate（`DB_NO_RAW_SQL_OUTSIDE_STORE`，豁免仅经受保护登记）；快照重建与写入同临界区（无双源）；本库不存任何真实地址与凭据明文（secret_hash 除外，其为单向哈希）。

### 8.12 外壳层（`postern-daemon::shells` + `postern-cli` + SPA/桌面壳）

**内涵**：接入形态的**翻译与事实采集层**——把外部协议形态（MCP/HTTP/SDK、控制面 RPC 前端）忠实翻译为内核的统一请求/响应，自身不做任何安全决策。

**范围内**：各协议端点的形态解析与语法层 4xx（不合法形态不进管线——这是协议语法判断，非安全决策）；NormalizedRequest 的构造（装箱：出示物 + ConnOrigin 采集 + 资源代号 + 原始 Intent 负载）；ConnOrigin 的构造（仅 listener 层，`SO_PEERCRED`/对端地址，契约 `SEC_CONSTRUCTION_SITES`）；响应的外壳格式化（输入恒为已脱敏产物）；MCP 固定动词工具面（工具描述只含事实、不随授权增删；`postern_surface` 为纯快照事实查询，不触发 `Adapter::discover`）；stdio↔UDS 桥；控制面瘦客户端（CLI/SPA/桌面壳——零状态、零安全逻辑，桌面壳额外提供托盘/通知/密钥链桥接/自动更新等宿主便利）。

**范围外**：Intent 负载格式的定义与解释（→适配器，外壳只搬运、不增不减）；任何策略判断（→策略引擎）；认证判定（→身份域，外壳只采集出示物与来源）；脱敏（→内核）；来源校验（→身份域可信域规则）；策略/审计/机密的任何触达（→分别归控制面/观测面/机密面，外壳无路径）。

**对外接口**：对外为 MCP / HTTP / SDK 端点（data.sock）与控制面前端（control.sock）；对内仅内核单一入口与控制面 API。

**必守不变量**：不做安全决策，但承担两项安全关键事实采集义务——ConnOrigin 只能由 listener 构造（绝不采信自报）、原始意图完整传递；任何外壳差异不得引起安全语义差异（公理七）；外壳间差异只允许是表达便利差异，不允许是能力面差异；客户端崩溃或缺席不影响 daemon 任何安全行为。

**SPA / 桌面外壳的边界裁决（CONS-17）**：控制面 `control.sock` 是 UDS-only、不监听网络（13/5.5）；"daemon 经 HTTP 向浏览器提供 SPA"与此存在张力。裁决方向（二选一，按形态取舍）：

- **桌面外壳 webview 经 UDS 代理**（默认、与"控制面不监听网络"一致）：SPA 静态资源由桌面外壳进程自带并在 webview 内加载，对控制面 API 的调用经外壳进程代理到 `control.sock`——浏览器/网络侧从不直接触达控制面，UDS-only 边界保持。
- **daemon 可选 localhost 绑定 + 控制面认证**（须显式开启）：若选 daemon 直接对 `127.0.0.1` 提供 SPA 与控制面 API，则该 TCP listener **必须叠加控制面认证**（与 5.5 的 control.sock 控制面认证同款：本地凭证 + 来源校验），并明确这是对"控制面不监听网络"默认边界的有意放宽，仅限 localhost、默认关闭。
- **os_keychain 桥接**：桌面外壳形态下，`MasterKeySource = os_keychain` 由外壳进程向 OS 钥匙串取主密钥并经受保护通道交给 daemon（外壳进程承担钥匙串交互，daemon 不直接弹窗），桥接接口即 `MasterKeySource::obtain` 的桌面实现。
